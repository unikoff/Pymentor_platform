# Практика и проверка практики

Этот файл — инструкция для автора занятий. Здесь три разных механизма: не смешивайте их.

| Что нужно студенту | Что добавить | Как занятие засчитывается |
| --- | --- | --- |
| Написать небольшую Python-функцию, результат которой можно сравнить | Блоки `starter` и `tests` прямо в Markdown урока | Автоматически, когда пройдут все тесты |
| Сделать работу вне редактора: Postman, SQLite, Git, Alembic, проект | Путь урока в `SELF_CHECK_LESSONS` | Студент сам нажимает «Выполнено» |
| Получить карточки с шагами и чекбоксами | Запись в `learning/rich_practice.py` | После отметки всех шагов студент нажимает «Выполнено» |

Для новых уроков с Python-кодом начинайте с первого варианта: он не требует правок frontend и не нужно создавать отдельный импорт.
## Быстрый ответ: можно ли создать практику только в `.md`?

Да, если это небольшая задача на чистом Python: студент пишет `solve`, а платформа сравнивает её результат с тестами. Для такого урока достаточно добавить специальную секцию и два служебных блока в Markdown — никаких импортов во frontend не нужно.

Не только Markdown: FastAPI, SQLite, Postman, Alembic, JWT, cookie, файлы и многофайловый проект нельзя честно проверить текущим изолированным редактором. Для них используйте ручную отметку или чеклист.

## Полная карта вкладки `practice.py`

| Состояние урока | Что добавить автору | Что увидит студент | Зачёт в прогресс |
| --- | --- | --- | --- |
| Урок без практики | Только теория Markdown. Нет секции-маркера, нет записи в `SELF_CHECK_LESSONS`, нет карточек. | «Задания скоро появятся». | Не засчитывается и не входит в процент прогресса. |
| Автопроверка `solve` | Специальная секция, `starter` и `tests` в этом же Markdown. | Редактор, «Запустить пример», «Проверить», результат каждого теста. | После успешной проверки, если студент вошёл в аккаунт. |
| Ручная отметка | Путь урока в `SELF_CHECK_LESSONS`; условие и критерий — в Markdown. | Описание и кнопка «Выполнено». | Только после входа и нажатия кнопки. |
| Чеклист | Запись в `rich_practice.py`. | Карточки, шаги с чекбоксами, критерий «Готово, если». | Только после отметки всех шагов, входа и нажатия кнопки. |

### Урок без практики

Если урок пока только про теорию, ничего специально добавлять не нужно: создай обычный `.md` с заголовком и текстом. Главное — не используй в заголовке секции слова `Практика для автопроверки`, `Домашнее задание` или `Контрольная точка`, не добавляй путь в `SELF_CHECK_LESSONS` и не регистрируй чеклист.

Во вкладке `practice.py` будет нейтральное сообщение «Задания скоро появятся». Такой урок не влияет на процент прогресса — это нормально для обзоров, вводных лекций и материалов без обязательной работы.

### Что делают кнопки в автопроверке

- **«Запустить пример»** выполняет только первый тест как демонстрацию, показывает вывод или ошибку и ничего не засчитывает.
- **«Проверить»** запускает все тесты текущей задачи, показывает процент, прошедшие и упавшие тесты.
- Успех означает, что прошли **все тесты одной задачи**. Если студент авторизован, урок помечается выполненным.
- Гость тоже может запускать и проверять код, но результат не сохранится в его прогрессе.

Текущая реализация засчитывает урок после успеха любой одной задачи. Поэтому для одного урока лучше создавать одну обязательную автозадачу; несколько задач подходят только как альтернативные упражнения, пока правило зачёта не будет изменено в коде.

### Важное правило: не смешивай режимы в одном уроке

Не добавляй путь в `SELF_CHECK_LESSONS`, если в том же `.md` есть автопроверяемая практика. `SELF_CHECK_LESSONS` имеет приоритет: backend не разберёт `starter` и `tests`, а вместо редактора покажет ручную кнопку «Выполнено».

Чеклист и `SELF_CHECK_LESSONS` технически можно не дублировать: чеклист сам включает ручную отметку. Markdown с обычным заголовком `## Практика` сам по себе ничего в интерфейсе не создаёт — он лишь показывает текст теории.

## 1. Автопроверяемая Python-практика прямо в `.md`

Добавьте в файл урока секцию с одним из названий:

- `## Практика для автопроверки`
- `## Домашнее задание`
- `## Контрольная точка`

Внутри неё каждое отдельное задание обязательно начинается с заголовка `### Задача ...`.
Секция и названия задач видны студенту; блоки `starter` и `tests` — служебные, из теории они автоматически скрываются.

Скопируйте шаблон и замените условие, стартовый код и тесты:

~~~~md
## Практика для автопроверки

### Задача 1. Нормализовать заголовок

Напишите функцию `solve(title)`. Она должна убрать пробелы по краям и вернуть строку в обычном регистре: первая буква заглавная, остальные строчные.

```starter
def solve(title):
    # Верните нормализованный заголовок.
    pass
```

```tests
[
  {
    "name": "лишние пробелы и смешанный регистр",
    "args": ["  дОмАшНяя РАбоТа  "],
    "expected": "Домашняя работа"
  },
  {
    "name": "уже аккуратный текст",
    "args": ["Проверка API"],
    "expected": "Проверка api"
  },
  {
    "name": "пустая строка",
    "args": [""],
    "expected": ""
  }
]
```
~~~~

После перезапуска backend в `practice.py` появится редактор и кнопка «Проверить». Занятие засчитывается автоматически только после прохождения всех тестов.

### Правила разметки

1. Проверяемая функция всегда называется `solve`.
2. `starter` должен содержать рабочую заготовку этой функции. Если блока нет, платформа попробует взять блок `python` с `def solve(...)`; если и его нет — подставит пустой шаблон.
3. `tests` должен быть корректным JSON-массивом. Комментарии и одинарные кавычки в JSON нельзя использовать.
4. В одном уроке можно создать несколько `### Задача ...`. Первая будет отмечена как лёгкая, остальные — как средние.
5. Не размещайте следующий `##`-заголовок внутри задания: он закончит секцию практики.

### Поля одного теста

| Поле | Обязательно | Значение |
| --- | --- | --- |
| `name` | нет | Читаемое имя проверки. Без него будет `test 1`, `test 2` и так далее. |
| `args` | нет | JSON-массив позиционных аргументов для `solve(*args)`. По умолчанию `[]`. |
| `kwargs` | нет | JSON-объект именованных аргументов для `solve(**kwargs)`. По умолчанию `{}`. |
| `expected` | да | Ожидаемый `return` функции либо ожидаемый вывод при `assert: "stdout"`. |
| `assert` | нет | Значение `"stdout"` переключает проверку с `return` на текст, напечатанный через `print`. |

Пример с именованными аргументами:

~~~~md
```tests
[
  {
    "name": "значения по умолчанию",
    "kwargs": {"title": "Код"},
    "expected": "Код:2:False"
  },
  {
    "name": "все именованные аргументы",
    "kwargs": {"title": "Тесты", "priority": 3, "done": true},
    "expected": "Тесты:3:True"
  }
]
```
~~~~

Пример, когда нужно проверять именно `print`:

~~~~md
```starter
def solve(days_left):
    print("Срочно" if days_left <= 3 else "Можно планировать")
```

```tests
[
  {
    "name": "срочная задача",
    "args": [2],
    "expected": "Срочно",
    "assert": "stdout"
  }
]
```
~~~~

### Ограничения автопроверки

Код запускается изолированно и максимум три секунды. В нём нельзя использовать `import`, чтение или запись файлов, `input`, `eval`, `exec` и похожие системные возможности. Поэтому для FastAPI, SQLite, Postman, Alembic и многoфайловых проектов выбирайте ручную практику ниже.

## 2. Ручная практика с кнопкой «Выполнено»

Это вариант для действий, которые платформа не может честно проверить автоматически: сделать запрос в Postman, поднять FastAPI, открыть SQLite-файл, создать миграцию, написать несколько файлов проекта.

Сначала добавьте в Markdown понятное условие и измеримый результат:

~~~~md
## Практика

1. Создайте файл `database.py` и подключите SQLite через `create_engine`.
2. Запустите приложение и откройте `/docs`.
3. Выполните `POST /tasks`, затем `GET /tasks`.

**Готово, если:** запрос `GET /tasks` возвращает созданную задачу, а после перезапуска приложения данные остаются в SQLite.
~~~~

Затем в `backend/learning/content.py` добавьте относительный путь файла в `SELF_CHECK_LESSONS`:

```python
SELF_CHECK_LESSONS = {
    # ... уже существующие пути ...
    "FastAPI, SQLite и SQLAlchemy - StudyHub Database API/Блок 14 - SQLite и основы SQLAlchemy (6 занятий)/80 - get_db и первая запись из FastAPI.md",
}
```

Путь всегда начинается с имени папки курса и совпадает с `source_file` урока. После перезапуска backend у студента появится кнопка «Выполнено». Автоматических тестов в этом режиме нет: студент сверяется с критерием «Готово, если» и подтверждает работу сам.

## 3. Карточки практики с чекбоксами

Карточки нужны, когда одна ручная работа состоит из нескольких обязательных шагов. Они настраиваются не из Markdown, а в `backend/learning/rich_practice.py`.

1. Создайте название трека и словарь практик:

```python
DATABASE_TRACK = "FastAPI, SQLite и SQLAlchemy - StudyHub Database API"

DATABASE_PRACTICE = {
    80: [
        _exercise(
            "Первое подключение",
            "Подключите FastAPI к SQLite и создайте одну запись.",
            [
                "Создайте engine и SessionLocal в database.py.",
                "Добавьте get_db и endpoint, который сохраняет задачу.",
                "Откройте /docs, создайте задачу и получите её списком.",
            ],
            "В Swagger видна созданная задача, а сессия закрывается после запроса.",
        ),
    ],
}
```

2. Зарегистрируйте словарь в `get_manual_practice`:

```python
practice_by_track = {
    FOUNDATIONS_TRACK: FOUNDATIONS_PRACTICE,
    DEEPER_TRACK: DEEPER_PRACTICE,
    DATABASE_TRACK: DATABASE_PRACTICE,
}.get(track_id, {})
```

3. Не добавляйте этот же урок в `SELF_CHECK_LESSONS`: наличие карточек уже включает ручную отметку.

Платформа покажет каждую запись `_exercise` отдельной карточкой. Студент должен отметить все её шаги, затем сможет нажать «Выполнено».
## 4. Полноценная программа в редакторе (продвинутый режим)

Есть четвёртый, кодовый вариант: `mode: "script"`. В нём студент пишет программу с `print`, а backend сравнивает весь её вывод. Этот режим **нельзя создать только из Markdown**: разметка `.md` всегда создаёт задачу типа `solve`.

Используйте `script` только для маленьких программ без `import`, файлов, `input` и сторонних библиотек. Настройка живёт в `backend/learning/rich_code_tasks.py`; для нового трека нужно:

1. добавить константу с точным именем папки курса;
2. создать словарь задач по номерам уроков;
3. зарегистрировать его в `get_code_tasks`.

Минимальный формат одной задачи:

```python
{
    "title": "Три строки",
    "level": "easy",
    "mode": "script",
    "prompt": "Выведите строки Старт, Работаем, Готово — каждую с новой строки.",
    "starter_code": 'print("Старт")\n# Добавьте ещё две строки\n',
    "tests": [
        {
            "name": "точный порядок строк",
            "namespace": {},
            "expected": "Старт\nРаботаем\nГотово",
        }
    ],
}
```

Для `mode: "script"` поле `namespace` передаёт переменные, которые будут доступны программе до запуска. Проверяется весь `stdout`, а не `return`. Дополнительно в кодовых настройках можно задать `requirements`, чтобы потребовать конструкцию языка: например `{"nodes": ["If"]}` для `if` или `{"calls": ["print"]}` для `print`.

Этот режим не подходит для уроков FastAPI и SQLAlchemy: им нужны импорты, файлы, сервер или база, а изолированный редактор намеренно это запрещает.

## Как практика связана с красивыми frontend-страницами

Файлы `frontend/src/lessons/.../block*.tsx` подменяют только вкладку теории. Они не отменяют и не дублируют практику. Если в Markdown урока есть автозадача, ручная отметка или чеклист, вкладка `practice.py` получит их от backend независимо от того, есть ли у урока дизайнерская страница.

## Рекомендуемый порядок работы автора

1. Сначала создай или заполни Markdown урока.
2. Реши, нужен ли уроку зачёт: нет, автоматический или ручной.
3. Для автоматического зачёта добавь одну задачу `solve` и минимум два теста прямо в `.md`.
4. Для FastAPI, базы данных и других внешних действий напиши шаги и «Готово, если», затем выбери: простую ручную отметку или карточки-чеклисты.
5. Перезапусти backend. Markdown читается при его запуске; менять frontend и пересобирать сайт для новой практики из `.md` не требуется.
6. Открой урок под тестовым аккаунтом, проверь неверное и верное решение, затем убедись, что прогресс изменился как ожидается.

## Что выбрать для новых этапов 4–5

- Логика без файлов и импорта, например нормализация данных, фильтрация списка, расчёт прав — используйте автопроверку в Markdown.
- FastAPI, база данных, миграция, Postman, cookie, JWT, несколько модулей — добавляйте ручную практику с критерием результата и `SELF_CHECK_LESSONS`.
- Длинный сценарий из нескольких шагов — добавляйте карточки в `rich_practice.py`.

## Проверка перед публикацией

1. Убедитесь, что Markdown сохранён в UTF-8.
2. Перезапустите backend и обновите страницу без кэша (`Ctrl+F5`).
3. Откройте урок → `practice.py`.
4. Для автопроверки: введите заведомо неверное решение, убедитесь, что хотя бы один тест падает; затем вставьте верное и добейтесь 100%.
5. Для ручной практики: проверьте текст критерия «Готово, если», отметьте шаги (если есть) и убедитесь, что кнопка «Выполнено» активна только после них.

Если практика не появилась, проверьте четыре вещи: точное название секции, заголовок `### Задача ...`, валидный JSON в `tests` и точный относительный путь в `SELF_CHECK_LESSONS`.

## Где находится логика

- Парсинг Markdown и `SELF_CHECK_LESSONS`: `backend/learning/content.py`.
- Изолированный запуск и сравнение тестов: `backend/learning/code_runner.py`.
- Карточки ручной практики: `backend/learning/rich_practice.py`.
