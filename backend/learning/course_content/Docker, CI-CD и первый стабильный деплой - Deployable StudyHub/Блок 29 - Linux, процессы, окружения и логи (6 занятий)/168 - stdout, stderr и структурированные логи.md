## 168. stdout, stderr и структурированные логи

### Цель

Перейти от случайных `print` к логам, которые отвечают на вопросы «что произошло, где и с каким request».

### Почему тема появляется сейчас

Конфигурация окружения готова, значит уровень и формат логов можно задавать извне.

### Основное содержание

- stdout и stderr;
- уровни DEBUG, INFO, WARNING, ERROR;
- Python logging;
- logger вместо print;
- контекстные поля;
- request id, user id и operation;
- traceback в серверном логе;
- почему пароль и token не попадают в лог;

### Практика StudyHub

Добавить логирование старта, завершения request и ожидаемой ошибки; проверить фильтрацию по request id и уровню.

### Визуальная и интерактивная подача

Сравнение бесполезной строки и структурированного события; карточки уровней; интерактив «какие поля нужны для расследования».

### Результат занятия

По одному request можно восстановить основные этапы обработки без чтения всего вывода сервера.

### Граница занятия

Не строить полноценный observability stack. Формат JSON можно показать как вариант, но не делать обязательным.

### Критерии готовности

- ученик объясняет главную модель своими словами;
- запускает минимальный пример или инфраструктурный сценарий;
- изменяет один параметр и предсказывает результат;
- проверяет успешный и ошибочный путь;
- фиксирует воспроизводимую проверку, тест или runbook;
- делает один осмысленный Git-коммит;
