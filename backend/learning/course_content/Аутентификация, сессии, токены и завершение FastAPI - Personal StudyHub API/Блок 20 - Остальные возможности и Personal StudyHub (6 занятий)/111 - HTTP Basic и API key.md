## Зачем это занятие

Сравним два простых способа доступа без смешения ролей: HTTP Basic подтвердит человека парой логин/пароль, а X-API-Key защитит служебный импорт постоянным секретом интеграции.

> **Связь с курсом.** Пользовательские session и JWT уже работают. Теперь важно увидеть, что FastAPI поддерживает и более простые схемы, но каждая отвечает на другой вопрос.

> **Граница темы.** Basic и API key не заменяют полноценную пользовательскую авторизацию автоматически. Basic передаёт credentials при каждом запросе, а API key обычно идентифицирует сервис или интеграцию.

## Маршрут занятия

1. **Разделить роли.** человек входит по credentials, служебный клиент предъявляет заранее выданный ключ.
2. **Построить dependency.** извлечь credential из стандартного места и сравнить безопасным способом.
3. **Защитить сценарий.** подключить X-API-Key только к endpoint импорта и проверить негативные случаи.
4. **Объяснить границу.** не хранить ключ в query string и не считать Base64 шифрованием.

## 01. Два механизма — две разные задачи

В Personal StudyHub уже есть пользовательские cookie-session и JWT. HTTP Basic и API key добавляются не как «ещё два логина», а как отдельные инструменты для ограниченных сценариев.

### HTTP Basic

клиент отправляет имя и пароль в заголовке Authorization при каждом запросе.

### API key

клиент предъявляет заранее выданный секрет интеграции, например для импорта данных.

### Главный вопрос

кого или что мы идентифицируем: пользователя, устройство, скрипт или внешний сервис.

### два независимых контракта доступа

```text
# Пользовательский сценарий
GET /basic/profile
Authorization: Basic <base64(username:password)>

# Служебный сценарий
POST /integrations/import
X-API-Key: <service-secret>
```

### Что проследить в примере

- **Basic · Человек предъявляет credentials:** Сервер получает username и password и проверяет их как обычные учётные данные. Пример: `Authorization: Basic ...`.
- **API key · Интеграция предъявляет секрет:** Ключ обычно связан со служебным клиентом, а не с интерактивной формой входа пользователя. Пример: `X-API-Key: ...`.
- **Session/JWT · Основной вход StudyHub:** Уже изученные механизмы остаются основой защищённого CRUD пользователей. Пример: `Cookie или Bearer`.

> **Контрольная граница:** Сначала назовите субъект доступа и срок жизни credential. Только после этого выбирайте конкретную схему.

## 02. Что реально передаёт HTTP Basic

Basic складывает username и password в строку, кодирует её Base64 и помещает в Authorization. Base64 позволяет передать байты текстом, но не скрывает содержимое от перехвата.

### Формат

Authorization: Basic <credentials>.

### Base64

это кодирование, которое легко обратимо без секретного ключа.

### Защита канала

Basic допустим только поверх HTTPS, иначе пароль можно прочитать из трафика.

### Base64 можно обратить

```python
import base64

raw = b"mentor:correct-horse"
encoded = base64.b64encode(raw).decode("ascii")
decoded = base64.b64decode(encoded).decode("utf-8")

print(encoded)
print(decoded)
```

### Что проследить в примере

- **Предсказание:** сначала объясните результат без запуска.
- **Ожидаемый вывод:** `nikita:secret`.
- **Подсказка:** Кодирование не использует секретный ключ и потому не является шифрованием.

> **Контрольная граница:** В учебном локальном запуске можно увидеть механику, но реальный Basic endpoint нельзя публиковать без HTTPS.

## 03. HTTPBasicCredentials и безопасное сравнение

FastAPI извлекает пару через HTTPBasic. Dependency получает объект HTTPBasicCredentials, затем приложение само решает, где и как проверить пользователя.

### Извлечение

HTTPBasic читает стандартный заголовок и разбирает credentials.

### Проверка

username и password сравниваются с ожидаемыми значениями или через auth service.

### Ответ

при ошибке возвращается 401 и заголовок WWW-Authenticate: Basic.

### dependency для учебного Basic endpoint

```python
import secrets
from typing import Annotated

from fastapi import Depends, FastAPI, HTTPException, status
from fastapi.security import HTTPBasic, HTTPBasicCredentials

app = FastAPI()
basic = HTTPBasic()


def require_basic_user(
    credentials: Annotated[HTTPBasicCredentials, Depends(basic)],
) -> str:
    username_ok = secrets.compare_digest(credentials.username, "mentor")
    password_ok = secrets.compare_digest(credentials.password, "studyhub")
    if not (username_ok and password_ok):
        raise HTTPException(
            status_code=status.HTTP_401_UNAUTHORIZED,
            detail="Invalid basic credentials",
            headers={"WWW-Authenticate": "Basic"},
        )
    return credentials.username
```

### Что проследить в примере

- **Задача:** Расположите действия от входящего запроса до выполнения endpoint.
  1. `прочитать Authorization`
  2. `разобрать Basic credentials`
  3. `сравнить username и password`
  4. `при ошибке вернуть 401 + WWW-Authenticate`
  5. `передать username в endpoint`
- **Почему:** Security dependency отвечает за извлечение и проверку, а endpoint получает уже подтверждённое имя.

> **Контрольная граница:** В реальном проекте открытый пароль не сравнивают с константой: используется существующий password service с хешем.

## 04. API key в заголовке X-API-Key

Служебный импорт не требует пользовательского профиля. Ему нужен отдельный секрет интеграции, который удобно читать через APIKeyHeader и проверять в dependency.

### Имя заголовка

X-API-Key становится частью публичного контракта интеграции.

### auto_error=False

позволяет сформировать собственный единый ответ при отсутствии ключа.

### Настройки

эталонный ключ приходит из окружения, а не записывается в исходный код.

### dependency служебного ключа

```python
import secrets
from typing import Annotated

from fastapi import Depends, HTTPException, status
from fastapi.security import APIKeyHeader

api_key_header = APIKeyHeader(name="X-API-Key", auto_error=False)


def require_import_key(
    api_key: Annotated[str | None, Depends(api_key_header)],
) -> None:
    if api_key is None or not secrets.compare_digest(
        api_key,
        settings.import_api_key,
    ):
        raise HTTPException(
            status_code=status.HTTP_401_UNAUTHORIZED,
            detail="Invalid API key",
        )
```

### Что проследить в примере

- **Пропуск:** Укажите имя заголовка, которое увидит клиент интеграции.
- **Ответ:** `X-API-Key`. Клиент передаёт ключ в выделенном HTTP-заголовке.

> **Контрольная граница:** Dependency возвращает None: после успешной проверки endpoint не обязан получать сам секрет и случайно логировать его.

## 05. Почему query-параметр хуже заголовка

Ключ в URL технически возможен, но URL чаще попадает в историю браузера, reverse-proxy логи, аналитику и referrer. Заголовок не решает все проблемы, но лучше выражает credential.

### Query

виден прямо в адресе и часто сохраняется инфраструктурой целиком.

### Header

отделяет служебный credential от адреса ресурса.

### Всегда

секрет нельзя печатать, коммитить и возвращать в тексте ошибки.

### URL против credential-заголовка

```text
# Плохой публичный контракт
POST /integrations/import?api_key=top-secret

# Предпочтительный контракт
POST /integrations/import
X-API-Key: top-secret
```

### Что проследить в примере

- **Вариант A — Ключ в URL:** Может оказаться в истории, логах и аналитике URL.
- **Вариант B — Ключ в заголовке:** Credential отделён от пути и query-параметров ресурса.
- **Вывод:** Для API key заголовок лучше выражает назначение и снижает число случайных утечек через URL.

> **Контрольная граница:** Даже правильный заголовок требует HTTPS, безопасного хранения, ротации и запрета на логирование значения.

## 06. Защищаем endpoint импорта StudyHub

Endpoint принимает подготовленный набор задач от доверенного служебного клиента. API key проверяется до чтения и изменения базы, а бизнес-операция остаётся в сервисе импорта.

### До endpoint

Depends(require_import_key) останавливает запрос без доверенного ключа.

### В endpoint

валидированная схема передаётся import service.

### В сервисе

транзакция создаёт записи или откатывается как единая операция.

### endpoint связывает security и import service

```python
from typing import Annotated

from fastapi import APIRouter, Depends, status
from sqlalchemy.orm import Session

router = APIRouter(prefix="/integrations", tags=["integrations"])


@router.post(
    "/tasks/import",
    status_code=status.HTTP_201_CREATED,
    dependencies=[Depends(require_import_key)],
)
def import_tasks(
    payload: TaskImportRequest,
    db: Annotated[Session, Depends(get_db)],
) -> ImportResult:
    return import_service.import_tasks(db, payload.items)
```

### Что проследить в примере

- **заголовка нет:** `401 missing`.
- **ключ неверный:** `401 invalid`.
- **ключ верный:** `run import`.

> **Контрольная граница:** Использование dependencies=[...] подходит, когда результат dependency не нужен среди параметров endpoint.

## 07. Негативные сценарии и ротация ключа

Схема считается законченной, когда проверены отсутствие credential, неправильное значение, старый ключ после ротации и корректный доступ. Секрет должен меняться без правки исходного кода.

### Missing

клиент получает одинаково оформленный отказ и операция не начинается.

### Invalid

ответ не сообщает, какая часть ключа совпала или какой ключ ожидался.

### Rotated

новое значение берётся из окружения после контролируемого перезапуска.

### два обязательных интеграционных теста

```python
from fastapi.testclient import TestClient


def test_import_rejects_missing_api_key(client: TestClient) -> None:
    response = client.post("/integrations/tasks/import", json={"items": []})
    assert response.status_code == 401


def test_import_accepts_valid_api_key(client: TestClient) -> None:
    response = client.post(
        "/integrations/tasks/import",
        headers={"X-API-Key": "test-key"},
        json={"items": []},
    )
    assert response.status_code == 201
```

### Что проследить в примере

- **Проблемный код:** `logger.info("import key=%s", api_key)`.
- **Вопрос:** Какая ошибка опаснее всего?
- **Ответ:** Секрет попадает и в лог, и в HTTP-ответ. Credential нельзя выводить ни в логи, ни в detail ошибки.

> **Контрольная граница:** Логи могут содержать имя интеграции, request id и результат импорта, но не сам ключ.

## 08. Контрольная точка: выбрать схему осознанно

## Итоговая проверка понимания

### Вопрос 1. Что делает Base64 в HTTP Basic?

1. шифрует пароль
2. кодирует строку обратимым способом
3. создаёт JWT

### Вопрос 2. Где предпочтительно передавать API key?

1. в X-API-Key
2. в имени файла
3. в query URL

### Вопрос 3. Что должно защищать Basic в сети?

1. HTTPS
2. длинное имя пользователя
3. формат JSON

### Вопрос 4. Когда результат dependency можно не добавлять в сигнатуру endpoint?

1. когда важен только факт успешной проверки
2. никогда
3. только для GET

## Ответы и объяснения

1. **кодирует строку обратимым способом** — Base64 не использует секрет и легко декодируется.
2. **в X-API-Key** — Выделенный заголовок лучше выражает credential и реже сохраняется как часть URL.
3. **HTTPS** — Без TLS credentials можно перехватить.
4. **когда важен только факт успешной проверки** — dependencies=[Depends(...)] подходит для проверки без передаваемого результата.

## Главное из занятия

- HTTP Basic передаёт credentials при каждом запросе.
- Base64 — кодирование, а не шифрование.
- Basic требует HTTPS и стандартного WWW-Authenticate при отказе.
- API key обычно идентифицирует интеграцию, а не пользователя.
- Служебный ключ лучше передавать в заголовке X-API-Key.
- Секреты приходят из окружения и никогда не попадают в логи.
- Security dependency останавливает запрос до бизнес-операции.
