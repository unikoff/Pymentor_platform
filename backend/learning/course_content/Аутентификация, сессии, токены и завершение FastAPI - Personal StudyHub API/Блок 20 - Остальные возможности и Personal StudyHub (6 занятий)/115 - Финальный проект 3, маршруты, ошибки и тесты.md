## 01. Routers как карта публичного API

Каждый router группирует связанные HTTP-контракты, но не становится новым сервисным слоем. Он отвечает за path, method, status code, dependencies, schemas и перевод результата в ответ.

### auth

регистрация, session login/logout, token, refresh и revoke.

### users/tasks

профиль и protected CRUD текущего пользователя.

### admin

узкие операции, требующие отдельного разрешения.

### main.py только собирает приложение

```python
from fastapi import FastAPI

from app.routers import admin, auth, tasks, users


def create_app() -> FastAPI:
    app = FastAPI(title="Personal StudyHub API")
    app.include_router(auth.router)
    app.include_router(users.router)
    app.include_router(tasks.router)
    app.include_router(admin.router)
    return app


app = create_app()
```

### Что проследить в примере

- **router · HTTP-контракт:** Принимает валидированные данные и вызывает готовый сервис. Пример: `path + method + schemas`.
- **service · Сценарий:** Не знает про APIRouter и выбирает транзакционную границу. Пример: `authenticate/create/update/revoke`.
- **dependency · Контекст запроса:** Подготавливает повторяемое значение до выполнения endpoint. Пример: `db/current_user/permission`.

> **Контрольная граница:** Фабрика create_app упрощает тестирование и не требует глобально переопределять настройки до импорта модулей.

## 02. Session login и logout в HTTP-слое

Session service возвращает raw session id, а router устанавливает его в HttpOnly cookie. Logout читает credential, отзывает запись и удаляет cookie независимо от того, существовала ли она у клиента.

### Login

authenticate_user → create_session → set_cookie.

### Cookie flags

HttpOnly, SameSite, Secure по окружению, Path и Max-Age.

### Logout

revocation на сервере и delete_cookie в ответе.

### cookie transport вокруг session service

```python
from typing import Annotated

from fastapi import APIRouter, Cookie, Depends, Form, Response, status
from sqlalchemy.orm import Session

router = APIRouter(prefix="/auth/session", tags=["auth-session"])


@router.post("/login", status_code=status.HTTP_204_NO_CONTENT)
def session_login(
    response: Response,
    email: Annotated[str, Form()],
    password: Annotated[str, Form()],
    db: Annotated[Session, Depends(get_db)],
) -> None:
    user = authenticate_user(db, email, password)
    session_id = session_service.create(db, user.id)
    response.set_cookie(
        key="studyhub_session",
        value=session_id,
        httponly=True,
        secure=settings.cookie_secure,
        samesite="lax",
        max_age=settings.session_ttl_seconds,
    )


@router.post("/logout", status_code=status.HTTP_204_NO_CONTENT)
def session_logout(
    response: Response,
    db: Annotated[Session, Depends(get_db)],
    session_id: Annotated[str | None, Cookie(alias="studyhub_session")] = None,
) -> None:
    if session_id is not None:
        session_service.revoke(db, session_id)
    response.delete_cookie("studyhub_session")
```

### Что проследить в примере

- **Задача:** Расположите действия успешного входа.
  1. `FastAPI читает email и password из Form`
  2. `authenticate_user проверяет credentials`
  3. `session service создаёт server-side record`
  4. `Response.set_cookie отправляет raw id`
  5. `вернуть 204 без пользовательских секретов`
- **Почему:** Проверка и хранение остаются в services, cookie — в HTTP-слое.

> **Контрольная граница:** Для production cookie Secure включается под HTTPS. Локальная настройка может отличаться, но причина различия фиксируется в config.

## 03. Bearer flow и единый current user

Token endpoint выдаёт пару после authenticate_user. Dependency OAuth2PasswordBearer извлекает access token, token service декодирует sub, а user repository загружает активного пользователя.

### Transport

Authorization: Bearer <access token>.

### Decode

проверяются подпись, exp и type=access.

### Resolve

sub преобразуется в user id, затем проверяются existence и is_active.

### current user из bearer token

```python
from typing import Annotated

from fastapi import Depends
from fastapi.security import OAuth2PasswordBearer
from sqlalchemy.orm import Session

oauth2_scheme = OAuth2PasswordBearer(tokenUrl="/auth/token")


def get_current_token_user(
    token: Annotated[str, Depends(oauth2_scheme)],
    db: Annotated[Session, Depends(get_db)],
) -> UserModel:
    user_id = token_service.decode_access(token)
    user = user_repository.get_by_id(db, user_id)
    if user is None or not user.is_active:
        raise InvalidTokenError()
    return user
```

### Что проследить в примере

- **подпись неверна:** `401`.
- **user удалён:** `401`.
- **user неактивен:** `401`.
- **всё корректно:** `current user`.

> **Контрольная граница:** user_id из request body никогда не заменяет current user. Клиент не выбирает владельца защищённого ресурса.

## 04. Ownership и admin permission

Аутентификация отвечает «кто это», авторизация — «разрешено ли действие». Для задач ownership проверяется запросом по id и owner_id, для admin endpoint — dependency role/permission.

### Own resource

repository возвращает task только при совпадении task_id и user_id.

### Foreign resource

проект выбирает 404, чтобы не подтверждать существование чужого id.

### Admin

current user существует, но role не подходит — 403.

### две формы авторизации

```python
def get_owned_task(
    db: Session,
    task_id: int,
    owner_id: int,
) -> TaskModel:
    statement = select(TaskModel).where(
        TaskModel.id == task_id,
        TaskModel.user_id == owner_id,
    )
    task = db.scalar(statement)
    if task is None:
        raise TaskNotFoundError(task_id)
    return task


def require_admin(
    current_user: Annotated[UserModel, Depends(get_current_token_user)],
) -> UserModel:
    if current_user.role != "admin":
        raise PermissionDeniedError("admin_stats")
    return current_user
```

### Что проследить в примере

- `нет действительного credential` → 401 Unauthorized.
- `user есть, но не admin` → 403 Forbidden.
- `чужая task скрыта ownership query` → 404 Not Found.
- `своя task найдена` → выполнить сценарий.

> **Контрольная граница:** Проверка role строкой сосредоточена в одной dependency. Если появятся permissions, routers не придётся переписывать полностью.

## 05. Доменные исключения и exception handlers

Services поднимают понятные исключения предметной области. FastAPI handlers централизованно переводят их в status code и ErrorResponse, сохраняя одинаковую внешнюю форму.

### Domain error

не зависит от FastAPI и несёт минимальный контекст.

### Handler

выбирает status code и безопасный message.

### Unexpected error

логируется с request id, но клиент получает общий 500 без traceback.

### единый перевод доменной ошибки

```python
from fastapi import FastAPI, Request
from fastapi.responses import JSONResponse


class TaskNotFoundError(Exception):
    def __init__(self, task_id: int) -> None:
        self.task_id = task_id


def install_exception_handlers(app: FastAPI) -> None:
    @app.exception_handler(TaskNotFoundError)
    def task_not_found_handler(
        request: Request,
        error: TaskNotFoundError,
    ) -> JSONResponse:
        return JSONResponse(
            status_code=404,
            content={
                "error": {
                    "code": "task_not_found",
                    "message": "Task was not found",
                    "request_id": getattr(request.state, "request_id", None),
                }
            },
        )
```

### Что проследить в примере

- **Вариант A — В task service:** Сервис становится зависимым от FastAPI transport.
- **Вариант B — В exception handler:** Сервис описывает предметный отказ, handler — HTTP-представление.
- **Вывод:** Разделение позволяет тестировать service без FastAPI и сохраняет единый error envelope.

> **Контрольная граница:** Не превращайте каждую ValueError в отдельную иерархию. Доменные исключения нужны там, где отказ имеет устойчивый смысл для сценария.

## 06. Изолированная SQLite-база и dependency overrides

Тесты не должны читать development database. Они создают отдельный engine, поднимают схему, отдают test Session через override и очищают ресурсы после сценария.

### Engine

отдельный файл во временной папке или in-memory конфигурация с подходящим pool.

### Override

app.dependency_overrides[get_db] заменяет production dependency.

### Cleanup

override удаляется, Session закрывается, база не влияет на следующий тест.

### fixture изолированного приложения

```python
import pytest
from fastapi.testclient import TestClient
from sqlalchemy import create_engine
from sqlalchemy.orm import sessionmaker


@pytest.fixture
def client(tmp_path):
    url = f"sqlite:///{tmp_path / 'test.db'}"
    engine = create_engine(url, connect_args={"check_same_thread": False})
    TestingSession = sessionmaker(bind=engine, expire_on_commit=False)
    Base.metadata.create_all(engine)

    def override_get_db():
        db = TestingSession()
        try:
            yield db
        finally:
            db.close()

    app = create_app()
    app.dependency_overrides[get_db] = override_get_db
    with TestClient(app) as test_client:
        yield test_client
    app.dependency_overrides.clear()
```

### Что проследить в примере

- **Проблемный код:** `def test_create_task():`.
- **Вопрос:** Какой риск скрыт в тесте?
- **Ответ:** Он может использовать production get_db и реальные данные. Без override тест зависит от production configuration и может загрязнить рабочую SQLite-базу.

> **Контрольная граница:** В финальной проверке предпочтительно прогонять Alembic и на тестовой базе. create_all допустим для быстрых unit fixtures, но не заменяет migration test.

## 07. Полные TestClient-сценарии

Хороший интеграционный тест проходит не один endpoint, а пользовательский поток: регистрация, вход, создание ресурса, доступ, отказ второго пользователя и logout/revocation.

### Session flow

cookie автоматически сохраняется внутри одного TestClient.

### JWT flow

access передаётся в Authorization, refresh rotation проверяет старый токен.

### Isolation

два клиента или два token set доказывают ownership.

### двухпользовательский ownership test

```python
def test_two_users_cannot_share_tasks(client: TestClient) -> None:
    first = register_and_login_token(client, "first@example.com")
    second = register_and_login_token(client, "second@example.com")

    created = client.post(
        "/tasks",
        headers=bearer(first.access_token),
        json={"title": "Private SQL task", "priority": 4},
    )
    task_id = created.json()["id"]

    foreign_read = client.get(
        f"/tasks/{task_id}",
        headers=bearer(second.access_token),
    )
    assert foreign_read.status_code == 404

    owner_read = client.get(
        f"/tasks/{task_id}",
        headers=bearer(first.access_token),
    )
    assert owner_read.status_code == 200
```

- **Терминальный сценарий:**

## Итоговая проверка понимания

### Вопрос 1. Что должен делать router?

1. связывать HTTP-контракт с готовым сервисом
2. самостоятельно хешировать пароль
3. создавать engine на каждый запрос

### Вопрос 2. Чем 401 отличается от 403?

1. 401 — нет подтверждённой личности, 403 — личности не хватает права
2. разницы нет
3. 403 означает отсутствующую таблицу

### Вопрос 3. Зачем dependency override в тесте?

1. подменить production get_db на изолированную Session
2. отключить все проверки
3. заменить pytest

### Вопрос 4. Как доказать ownership?

1. сценарием двух пользователей
2. только тестом 200
3. скрытием OpenAPI

## Ответы и объяснения

1. **связывать HTTP-контракт с готовым сервисом** — Router управляет транспортом и вызывает нижние слои.
2. **401 — нет подтверждённой личности, 403 — личности не хватает права** — Эти статусы отражают разные границы доступа.
3. **подменить production get_db на изолированную Session** — Override направляет приложение в тестовую базу.
4. **сценарием двух пользователей** — Второй пользователь должен получить отказ на чужой ресурс.

## Главное из занятия

- Routers отвечают за HTTP-контракт, а services — за сценарии.
- Session service не устанавливает cookie самостоятельно.
- Bearer dependency декодирует access token и загружает active user.
- Ownership проверяется запросом task_id + owner_id.
- 401, 403 и 404 имеют разные смыслы доступа.
- Доменные исключения переводятся централизованными handlers.
- Dependency overrides изолируют тестовую SQLite-базу.
- End-to-end тесты проходят реальные auth и resource flows.
- Release candidate обязан стартовать с чистой базы.

- README и фактическое поведение не противоречат друг другу.

## Типичные ошибки

- Переносить business logic в router ради короткого service.
- Доверять `user_id`, filename, Content-Type или API key из небезопасного места.
- Возвращать клиенту traceback, hash или значение секрета.
- Использовать одну глобальную SQLAlchemy Session.
- Считать один успешный запрос достаточной проверкой security-сценария.
- Запускать тесты на той же SQLite-базе, что и локальное приложение.
- Добавлять новые технологии вместо завершения текущего контракта.
