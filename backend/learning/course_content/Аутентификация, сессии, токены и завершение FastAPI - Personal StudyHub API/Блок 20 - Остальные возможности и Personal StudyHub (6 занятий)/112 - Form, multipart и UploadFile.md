## 01. Три формата тела запроса

Тело HTTP-запроса имеет Content-Type. FastAPI выбирает способ разбора не по названию endpoint, а по объявленным параметрам и фактическому формату клиента.

### application/json

удобен для вложенных структур и обычного API CRUD.

### application/x-www-form-urlencoded

передаёт простые пары полей, как форма логина.

### multipart/form-data

разделяет тело на части и может переносить текст вместе с бинарным файлом.

### один HTTP endpoint — один ожидаемый формат

```text
POST /tasks
Content-Type: application/json
{"title": "Read SQL", "priority": 4}

POST /auth/token
Content-Type: application/x-www-form-urlencoded
username=nikita&password=secret

POST /tasks/7/attachments
Content-Type: multipart/form-data; boundary=...
title=diagram&file=<binary bytes>
```

### Что проследить в примере

- **JSON · Структурированные данные:** Основной формат request schemas и response schemas в REST API. Пример: `{"title": "SQL"}`.
- **Form · Простые поля формы:** Подходит для HTML-форм и OAuth2PasswordRequestForm. Пример: `username=...&password=...`.
- **Multipart · Поля плюс файл:** Каждая часть имеет собственные заголовки и содержимое. Пример: `description + file`.

> **Контрольная граница:** JSON и multipart не объединяются автоматически в одну Pydantic body-модель: формат контракта нужно спроектировать явно.

## 02. Form и установка python-multipart

Параметр Form сообщает FastAPI, что значение приходит не из JSON. Для разбора form-data и multipart проекту нужна зависимость python-multipart.

### Явное объявление

Form(...) отличает поле формы от query-параметра.

### Annotated

метаданные источника остаются рядом с типом.

### Зависимость проекта

python-multipart фиксируется в requirements или pyproject.

### два валидируемых поля формы

```python
from typing import Annotated

from fastapi import APIRouter, Form

router = APIRouter()


@router.post("/feedback")
def create_feedback(
    topic: Annotated[str, Form(min_length=3, max_length=80)],
    message: Annotated[str, Form(min_length=1, max_length=1000)],

    return {"topic": topic, "message": message}
```

## 03. UploadFile вместо bytes для больших файлов

Параметр bytes заставляет собрать всё содержимое в памяти. UploadFile предоставляет метаданные и файловый объект, который может быть временно сохранён на диск и читаться порциями.

### filename

исходное имя клиента полезно только как метаданные для отображения.

### content_type

заявленный MIME-type помогает первичной проверке, но может быть подделан.

### file

синхронный file-like объект подходит для обычного def endpoint текущего этапа.

### метаданные UploadFile

```python
from typing import Annotated

from fastapi import File, UploadFile


def inspect_upload(
    upload: Annotated[UploadFile, File(description="Task attachment")],
) -> dict[str, str | None]:
    return {
        "filename": upload.filename,
        "content_type": upload.content_type,
    }
```

### Что проследить в примере

- **Вариант A — bytes:** Всё содержимое собирается в оперативной памяти до вызова endpoint.
- **Вариант B — UploadFile:** Доступны поток, filename, content_type и временное хранение.
- **Вывод:** UploadFile даёт более подходящий контракт для потоковой проверки и сохранения.

> **Контрольная граница:** На этом этапе endpoint остаётся синхронным: чтение выполняется через upload.file, без введения async/await.

## 04. Размер нужно ограничить до сохранения

Заявленный Content-Length может отсутствовать или быть неверным. Надёжный учебный контроль читает максимум limit + 1 байт и отклоняет файл, если получено больше лимита.

### Лимит

задаётся константой или настройкой и отражается в документации.

### Проверка

читается только ограниченный объём, а не бесконечный поток в память.

### Позиция

после проверки file.seek(0) возвращает поток к началу перед копированием.

### жёсткая граница 2 MiB

```python
from fastapi import HTTPException, UploadFile, status

MAX_UPLOAD_BYTES = 2 * 1024 * 1024


def validate_size(upload: UploadFile) -> None:
    chunk = upload.file.read(MAX_UPLOAD_BYTES + 1)
    if len(chunk) > MAX_UPLOAD_BYTES:
        raise HTTPException(
            status_code=status.HTTP_413_REQUEST_ENTITY_TOO_LARGE,
            detail="File is too large",
        )
    upload.file.seek(0)
```

### Что проследить в примере

1. Приложение знает максимально допустимый объём. Состояние: `limit=4 bytes`.
2. Читаем на один байт больше, чтобы заметить превышение. Состояние: `read=up to 5 bytes`.
3. Сравниваем фактически прочитанный размер. Состояние: `condition=len(data) > 4`.
4. Для допустимого файла возвращаем указатель к началу. Состояние: `position=0`.
5. Только после проверки начинается сохранение. Состояние: `result=save`.

> **Контрольная граница:** 413 сообщает именно о слишком большом теле. Это понятнее, чем общий 400 без причины.

## 05. Не доверяем filename и content_type

Клиент может прислать имя ../../config.py или указать image/png для произвольных байтов. Сервер генерирует собственное имя и ограничивает разрешённые типы.

### Серверное имя

UUID или другой идентификатор исключает путь, переданный клиентом.

### Расширение

выбирается из allowlist по проверенному типу, а не копируется целиком.

### Папка

итоговый Path должен оставаться внутри заранее определённого upload directory.

### сервер выбирает безопасный путь

```python
from pathlib import Path
from uuid import uuid4

ALLOWED_TYPES = {
    "image/png": ".png",
    "image/jpeg": ".jpg",
    "application/pdf": ".pdf",
}


def build_storage_path(upload_dir: Path, content_type: str | None) -> Path:
    suffix = ALLOWED_TYPES.get(content_type or "")
    if suffix is None:
        raise ValueError("Unsupported file type")
    return upload_dir / f"{uuid4().hex}{suffix}"
```

### Что проследить в примере

- **Проблемный код:** `target = UPLOAD_DIR / upload.filename`.
- **Вопрос:** Почему такой путь нельзя считать безопасным?
- **Ответ:** filename контролирует клиент и может содержать путь. Исходное имя нельзя использовать как путь хранения без строгой нормализации и собственной схемы имён.

> **Контрольная граница:** Allowlist MIME-type — только первый барьер. Для чувствительных файлов в реальном продукте нужна проверка содержимого и антивирусная обработка.

## 06. Файл на диске, метаданные в базе

База не обязана хранить бинарные байты. Для StudyHub достаточно сохранить файл в контролируемой папке, а в AttachmentModel записать идентификатор владельца, безопасное имя, исходное имя и размер.

### Файловая система

хранит содержимое по внутреннему storage_name.

### SQLite

хранит связь task_id, owner_id, original_name, media_type и size.

### Компенсация

если commit не прошёл, уже записанный файл удаляется, чтобы не остался сиротой.

### согласование двух хранилищ

```python
from pathlib import Path
import shutil

from sqlalchemy.orm import Session


def save_attachment(
    db: Session,
    task: TaskModel,
    upload: UploadFile,
    target: Path,
    size: int,
) -> AttachmentModel:
    try:
        with target.open("wb") as output:
            shutil.copyfileobj(upload.file, output)

        attachment = AttachmentModel(
            task_id=task.id,
            owner_id=task.user_id,
            storage_name=target.name,
            original_name=upload.filename or "upload",
            media_type=upload.content_type,
            size=size,
        )
        db.add(attachment)
        db.commit()
        db.refresh(attachment)
        return attachment
    except Exception:
        db.rollback()
        target.unlink(missing_ok=True)
        raise
```

### Что проследить в примере

- `target.open("wb")` — создаёт файл только после валидации.
- `AttachmentModel(...)` — фиксирует метаданные и принадлежность.
- `db.commit()` — делает запись видимой в базе.
- `db.rollback()` — восстанавливает Session после ошибки.
- `target.unlink(...)` — компенсирует уже созданный файл.

> **Контрольная граница:** Файловая система и SQLite не образуют одну настоящую транзакцию. Поэтому код явно описывает компенсацию при частичном сбое.

## 08. Контрольная точка: тестируем не только 201

Положительный upload — только один сценарий. Готовый маршрут также отклоняет чужую задачу, неподдерживаемый тип, превышение размера и запрос без файла.

### Успех

метаданные записаны, файл существует, owner и task связаны.

### Валидация

неподдерживаемый MIME-type и большой размер возвращают согласованные ошибки.

### Безопасность

второй пользователь не может добавить или прочитать чужое вложение.

### multipart в TestClient

```python
def test_upload_attachment(client, auth_headers, png_bytes) -> None:
    response = client.post(
        "/tasks/1/attachments",
        headers=auth_headers,
        data={"description": "diagram"},
        files={"upload": ("diagram.png", png_bytes, "image/png")},
    )
    assert response.status_code == 201
    assert response.json()["original_name"] == "diagram.png"


def test_rejects_large_attachment(client, auth_headers) -> None:
    response = client.post(
        "/tasks/1/attachments",
        headers=auth_headers,
        data={"description": "too large"},
        files={"upload": ("big.pdf", b"x" * 50, "application/pdf")},
    )
    assert response.status_code == 413
```

### Что проследить в примере

- **Вопрос:** Почему недостаточно проверить только content_type?
- **Ориентир ответа:** Клиент сам объявляет Content-Type, поэтому строка может не соответствовать фактическим байтам. Она подходит для первичного allowlist, но не является доказательством безопасного содержимого.

> **Контрольная граница:** В тесте лимит можно уменьшить через settings override, чтобы не создавать многомегабайтный объект.

## Итоговая проверка понимания

### Вопрос 1. Когда нужен multipart/form-data?

1. когда есть текстовые части и файл
2. для любого GET
3. только для JSON

### Вопрос 2. Почему UploadFile предпочтительнее bytes для файла?

1. даёт поток и метаданные
2. автоматически удаляет вирусы
3. всегда шифрует содержимое

### Вопрос 3. Можно ли использовать upload.filename как путь хранения?

1. нет, имя контролирует клиент
2. да, всегда
3. только без расширения

### Вопрос 4. Что сделать после чтения файла для проверки размера?

1. seek(0)
2. commit()
3. base64 encode

## Ответы и объяснения

1. **когда есть текстовые части и файл** — Multipart делит тело на несколько частей с собственными метаданными.
2. **даёт поток и метаданные** — UploadFile не требует сразу держать весь файл как bytes в памяти.
3. **нет, имя контролирует клиент** — Сервер должен генерировать собственное безопасное имя.
4. **seek(0)** — Перед последующим копированием указатель возвращают к началу.

## Главное из занятия

- Content-Type определяет формат тела запроса.
- Form читает поля формы, а multipart переносит поля и бинарные части.
- Для Form и UploadFile нужен python-multipart.
- UploadFile предоставляет метаданные и file-like поток.
- Размер проверяется по фактически прочитанным байтам.
- Filename и content_type считаются недоверенными.
- Сервер генерирует storage_name и хранит метаданные отдельно.
- Файл и запись базы требуют компенсации при частичном сбое.

## Типичные ошибки

- Переносить business logic в router ради короткого service.
- Доверять `user_id`, filename, Content-Type или API key из небезопасного места.
- Возвращать клиенту traceback, hash или значение секрета.
- Использовать одну глобальную SQLAlchemy Session.
- Считать один успешный запрос достаточной проверкой security-сценария.
- Запускать тесты на той же SQLite-базе, что и локальное приложение.
- Добавлять новые технологии вместо завершения текущего контракта.
