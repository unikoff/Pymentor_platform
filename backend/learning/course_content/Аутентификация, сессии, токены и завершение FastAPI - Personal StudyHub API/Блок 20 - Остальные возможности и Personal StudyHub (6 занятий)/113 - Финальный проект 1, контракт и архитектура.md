Маршрут проектируется как публичное обещание: method, path, credential, request schema, response schema, status code и ожидаемые ошибки. Такая таблица предотвращает расхождения между routers.

### Auth routes

отделяют регистрацию, cookie-session и bearer token flow.

### Resource routes

### Admin route

требует отдельную dependency разрешения.

### публичная карта API

```text
POST   /auth/register             -> 201 UserRead
POST   /auth/session/login        -> 204 + Set-Cookie
POST   /auth/session/logout       -> 204 + delete cookie
POST   /auth/token                -> 200 TokenPair
POST   /auth/refresh              -> 200 TokenPair
POST   /auth/token/logout         -> 204
GET    /users/me                  -> 200 UserRead
GET    /tasks                     -> 200 Page[TaskRead]
POST   /tasks                     -> 201 TaskRead
PATCH  /tasks/{task_id}           -> 200 TaskRead
DELETE /tasks/{task_id}           -> 204
POST   /tasks/{task_id}/attachments -> 201 AttachmentRead
GET    /admin/stats               -> 200 AdminStats
```

### Что проследить в примере

- `POST /auth/register` — создаёт пользователя, но не возвращает password_hash.
- `POST /auth/session/login` — создаёт server-side session и cookie.
- `POST /auth/token` — возвращает access и refresh pair.
- `GET /tasks` — читает только задачи current user.
- `GET /admin/stats` — требует разрешение admin.

> **Контрольная граница:** Одинаковое действие не обязано использовать одинаковый транспорт: session logout удаляет cookie, token logout отзывает refresh-session.

## 03. Единый формат ошибок

Клиенту трудно обрабатывать API, если один router возвращает detail строкой, второй — произвольный dict, а третий — внутренний traceback. Финальный проект выбирает одну внешнюю форму.

### code

стабильный машинный идентификатор ошибки.

### message

безопасное пояснение для человека.

### request_id

необязательная связь с логом без выдачи внутренних деталей.

### контракт ошибки

```text
from pydantic import BaseModel


class ErrorBody(BaseModel):
    code: str
    message: str
    request_id: str | None = None


class ErrorResponse(BaseModel):
    error: ErrorBody


example = ErrorResponse(
    error=ErrorBody(
        code="task_not_found",
        message="Task was not found",
        request_id="req-42",
    )
)
```

### Что проследить в примере

- **Вариант A — Случайный detail:** Выдаёт внутреннюю деталь и не имеет стабильного кода.
- **Вариант B — Единый envelope:** Форма одинакова для всех доменных отказов.
- **Вывод:** Клиент опирается на code, а внутренний traceback остаётся только в серверном логе.

> **Контрольная граница:** 422 от Pydantic тоже нужно осознанно привести к внешнему контракту или честно документировать как отдельный стандартный формат.

## 04. Схема данных и связи

Таблицы проектируются от сценариев. User владеет задачами и credentials, Category группирует задачи, Session и RefreshSession хранят отзывное состояние, Attachment связан с задачей и владельцем.

### User

id, email, username, password_hash, is_active, role.

### Auth state

server sessions и refresh sessions имеют token hash, expiry и revoked_at.

### Domain

Task с owner_id и category_id, Attachment с task_id и storage_name.

### ER-карта Personal StudyHub

```text
users 1 ─────── * tasks
  │               │
  │               ├──── * attachments
  │               │
  │               └──── 0..1 category
  │
  ├──── * sessions
  └──── * refresh_sessions

categories 1 ─── * tasks
```

### Что проследить в примере

- `users` → личность, парольный хеш и роль.
- `sessions` → отзывная cookie-session на сервере.
- `refresh_sessions` → rotation и revocation refresh token.
- `tasks` → учебные задачи конкретного владельца.
- `attachments` → метаданные файла и связь с task.

> **Контрольная граница:** Access token не обязательно хранить в базе. Отзывное состояние сосредоточено в refresh-session, а access живёт коротко.

## 05. Дерево проекта и направление зависимостей

Папки нужны только там, где упрощают поиск ответственности. Routers знают HTTP, services знают сценарии, repositories или crud знают SQLAlchemy, models описывают таблицы, schemas — внешние данные.

### Верхний слой

main и routers связывают FastAPI, dependencies и response models.

### Средний слой

auth/task services реализуют сценарии и поднимают доменные исключения.

### Нижний слой

database, models и repositories выполняют хранение.

### минимальное дерево финального проекта

```text
app/

├── config.py
├── database.py
├── models/

└── exceptions.py

```

### Что проследить в примере

- **Задача:** Расположите слои для PATCH /tasks/{task_id}.
  1. `router валидирует HTTP-контракт`
  2. `dependency получает current user`
  3. `service проверяет сценарий update`
  4. `repository ищет task_id + owner_id`
  5. `Session выполняет SQL и commit`
- **Почему:** HTTP-слой направляет запрос вниз, а модели хранения не импортируют routers.

> **Контрольная граница:** Для учебного проекта допустим простой crud.py вместо абстрактного repository-интерфейса. Название не важнее ясной ответственности.

## 06. Базовая модель угроз

Security проектируется через конкретные злоупотребления. Для каждого актива называются нарушитель, путь атаки, защита и тест, который доказывает ожидаемый отказ.

### Credential theft

пароль, cookie, API key и refresh token не попадают в логи и ответы.

### IDOR

task ищется по task_id + current_user.id, а не только по id.

### Upload abuse

лимит, allowlist, серверное имя и ownership защищают файловый маршрут.

### минимальный threat register

```python
threat_register = [
    {
        "asset": "task",
        "attack": "second user requests foreign task id",
        "control": "query by id and owner_id",
        "test": "two-user isolation test",
    },
    {
        "asset": "refresh session",
        "attack": "reuse old refresh token",
        "control": "rotation + revoked_at",
        "test": "old token returns 401",
    },
    {
        "asset": "upload directory",
        "attack": "path traversal filename",
        "control": "server-generated storage name",
        "test": "malicious filename stays inside directory",
    },
]
```

### Что проследить в примере

- **IDOR:** Запрашивать чужой task id и проверять owner_id на каждом read/update/delete.
- **Token replay:** После rotation старый refresh token должен быть отозван и давать 401.
- **Path traversal:** Не использовать клиентский filename как путь на диске.
- **Credential leak:** Маскировать заголовки и не включать секреты в detail или логи.

> **Контрольная граница:** Модель угроз не доказывает абсолютную безопасность. Она делает ключевые риски видимыми и превращает их в проверяемые требования.

## Итоговая проверка понимания

### Вопрос 1. С чего начинается финальный проект?

1. с обязательных сценариев и границ
2. с максимального числа папок
3. с Dockerfile

### Вопрос 2. Что должно быть стабильным в ошибке?

1. машинный code и форма envelope
2. полный traceback
3. имя Python-файла

### Вопрос 3. Как защитить задачу от IDOR?

1. искать по task_id и owner_id
2. скрыть id в документации
3. добавить длинное название

### Вопрос 4. Что даёт вертикальный срез?

1. полный проверяемый сценарий маленьким изменением
2. только новые модели
3. отсутствие тестов

## Ответы и объяснения

1. **с обязательных сценариев и границ** — Сначала фиксируется обещанное поведение и scope.
2. **машинный code и форма envelope** — Клиенту нужен предсказуемый внешний контракт.
3. **искать по task_id и owner_id** — Ownership проверяется в запросе к данным, а не доверием к URL.
4. **полный проверяемый сценарий маленьким изменением** — Срез проходит от хранения до HTTP и автоматической проверки.

## Главное из занятия

- Финальный проект начинается с обязательных сценариев и out-of-scope.
- Route map фиксирует method, path, credential, schemas и statuses.
- Единый error envelope отделяет внешний контракт от внутренних ошибок.
- Таблицы появляются из пользовательских и security-сценариев.
- Направление зависимостей идёт от HTTP-слоя к хранению.
- Threat register превращает риски в controls и тесты.
- Вертикальные срезы уменьшают область поиска ошибки.
- Definition of ready предшествует реализации Lesson114.

## Критерии готовности

- Код запускается без ручных исправлений внутри базы данных.
- Используются только механизмы и зависимости, уже введённые в курсе.

- Считать один успешный запрос достаточной проверкой security-сценария.
