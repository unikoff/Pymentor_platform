## 1. Аутентификация отделяется от выдачи доступа

Проверка credentials отвечает только на один вопрос: соответствует ли пара email/password активному пользователю. Создание session, cookie или token является следующим самостоятельным действием.

### Маршрут
1. **Нормализовать email.** использовать ту же форму, что и при регистрации.
2. **Найти User.** выполнить SELECT по уникальному индексированному email.
3. **Проверить password.** вызвать verify_password с кандидатом и сохранённым хешем.
4. **Проверить состояние.** не аутентифицировать деактивированный аккаунт.
5. **Вернуть результат.** User при успехе или None при любом недействительном наборе credentials.

**Результат маршрута:** После занятия сервис готов стать основанием cookie-session и token login.

### Сравнение решений

**Вопрос:** Какая ответственность должна остаться в authenticate_user?

#### Вариант A. Проверить и выдать JWT

```python
return create_access_token(user)
```

Смешиваются проверка credentials и конкретный способ доступа.

#### Вариант B. Вернуть User или None

```python
return user if valid else None
```

Следующий слой сам решит, создать session, token или отказ.

**Предпочтительный выбор:** Вариант B. Чистая граница позволяет переиспользовать аутентификацию в разных механизмах.

> **Связь и граница.** Успешный authenticate_user ещё не означает авторизацию любой операции: после него остаются role, owner_id и другие правила доступа.

### Пример: единая функция нормализации

```python
def normalize_email(email: str) -> str:
    return email.strip().lower()


def get_user_by_email(db: Session, email: str) -> UserModel | None:
    normalized = normalize_email(email)
    statement = select(UserModel).where(UserModel.email == normalized)
    return db.scalar(statement)
```

### Предскажите результат

```python

```

**Подсказка:** Сначала удаляются пробелы по краям, затем меняется регистр.

**Результат:**

```text
student@example.com
```

**Утверждение:** Вход может использовать email как есть, потому что SQLAlchemy сам нормализует строки.

**Ответ:** Ложь. ORM выполняет сравнение, но не придумывает прикладное правило регистра и пробелов.

## 4. Функция authenticate_user

Сервис получает Session и две строки. Он не знает HTTP, cookie и JWT; ожидаемые неуспехи выражаются значением None.

### Фокус сцены

Раздел показывает, как тема влияет на **email/password → authenticate_user → UserModel | None**.

**Перед продолжением объясните:** какие данные входят в механизм, какая проверка выполняется, что считается успехом и какой отказ ожидается.

> **Граница:** Не добавляйте session и token внутрь проверки credentials: это следующий слой проекта.

### Пример: app/services/auth.py

```python
from sqlalchemy.orm import Session

from app.models.user import UserModel
from app.security.passwords import verify_password
from app.services.users import get_user_by_email


def authenticate_user(
    db: Session,
    email: str,
    password: str,
) -> UserModel | None:
    user = get_user_by_email(db, email)

    if user is None:
        return None

    if not verify_password(password, user.password_hash):
        return None

    if not user.is_active:
        return None

    return user
```

### Пошаговое выполнение

```python
user = get_user_by_email(db, email)
if user is None:
    return None
if not verify_password(password, user.password_hash):
    return None
if not user.is_active:
    return None
return user
```
1. Строка 1. Поиск возвращает ORM-объект или None. Состояние: `user = UserModel(id=4)`.
2. Строка 2. Неизвестный email завершается без чтения password_hash. Состояние: `condition = False`.
3. Строка 4. Кандидат проверяется библиотекой против сохранённого хеша. Состояние: `verify = True`.
4. Строка 6. Серверное состояние аккаунта проверяется отдельно. Состояние: `is_active = True`.
5. Строка 8. Только полностью прошедший User возвращается вызывающему слою. Состояние: `result = UserModel(id=4)`.

## 5. HTTP-граница пока не выдаёт session или token

Для учебной проверки можно сделать временный endpoint, который подтверждает корректность credentials без выпуска доступа. В продукте его заменит login с server-side session.

### Фокус сцены

Раздел показывает, как тема влияет на **email/password → authenticate_user → UserModel | None**.

**Перед продолжением объясните:** какие данные входят в механизм, какая проверка выполняется, что считается успехом и какой отказ ожидается.

> **Граница:** Не добавляйте session и token внутрь проверки credentials: это следующий слой проекта.

### Пример: учебный endpoint проверки

```python
from fastapi import HTTPException, Response, status


@router.post("/credentials/check", status_code=status.HTTP_204_NO_CONTENT)
def check_credentials(payload: LoginRequest, db: DbSession):
    user = authenticate_user(db, str(payload.email), payload.password)

    if user is None:
        raise HTTPException(
            status_code=status.HTTP_401_UNAUTHORIZED,
            detail="Неверный email или пароль",
        )

    return Response(status_code=status.HTTP_204_NO_CONTENT)
```

### Карточки понятий

#### 204: Успех без тела

Credentials приняты, но новый ресурс или токен не создаётся.
```text
No Content
```

#### 401: Единый отказ

Неизвестный email, неверный пароль или запрещённое состояние не раскрываются отдельно.
```text
invalid credentials
```

#### next: Login позже

Следующий блок создаст server-side session и установит cookie.
```text
authenticate → create_session
```

### Правда или ложь

**Утверждение:** Endpoint credentials/check является готовым login, потому что возвращает 204.

**Ответ:** Ложь. Он ничего не создаёт для последующих запросов. Login должен выдать или установить средство доступа.

def test_authenticate_user_success(db_session, user_factory):
    user = user_factory(
        email="student@example.com",

    result = authenticate_user(
        db_session,

## 7. Ошибки, которые делают вход опасным

Самая заметная ошибка — сравнение password со строкой хеша. Менее заметные — логирование секрета, разные ответы и отсутствие проверки is_active.

### Фокус сцены

Раздел показывает, как тема влияет на **email/password → authenticate_user → UserModel | None**.

**Перед продолжением объясните:** какие данные входят в механизм, какая проверка выполняется, что считается успехом и какой отказ ожидается.

> **Граница:** Не добавляйте session и token внутрь проверки credentials: это следующий слой проекта.

### Поиск ошибки

```python
def authenticate_user(user, password):
    print(f"login password={password}")
    if password == user.password_hash:
        return user
    return None
```

**Вопрос:** Какие две проблемы находятся в функции?

**Варианты:**

- Пароль логируется и сравнивается с хешем напрямую
- Нельзя использовать return
- Имя user слишком короткое

**Ответ:** Пароль логируется и сравнивается с хешем напрямую. Секрет не попадает в логи, а проверка выполняется через verify_password.

#### Исправленный вариант

```python
def authenticate_user(user, password):
    if not verify_password(password, user.password_hash):
        return None
    if not user.is_active:
        return None
    return user
```

### Сборка последовательности: Соберите безопасную диагностику входа

Выберите допустимые шаги без записи секрета.

**Фрагменты:**
- `записать событие login_failed`
- `использовать внутренний user_id, если он известен`
- `добавить request/correlation id`
- `записать password для отладки` — секрет запрещён
- `вернуть общее сообщение клиенту`

**Правильный порядок:** `event` → `user` → `request` → `response`.

**Объяснение:** Для диагностики достаточно категории события и технического контекста без credentials.

> **Связь и граница.** Неуспешный вход является ожидаемым сценарием, но массовые повторения позже потребуют rate limiting и наблюдаемости.

## Главное из занятия

- Регистрация и вход используют одно правило нормализации email.
- authenticate_user ищет пользователя, проверяет password hash и is_active.
- Неизвестный email и неверный пароль получают одинаковый внешний отказ.
- Сервис возвращает UserModel или None и не зависит от HTTP.
- Проверка credentials не равна выдаче session или token.
- Блок готовит прямой переход к server-side sessions.
