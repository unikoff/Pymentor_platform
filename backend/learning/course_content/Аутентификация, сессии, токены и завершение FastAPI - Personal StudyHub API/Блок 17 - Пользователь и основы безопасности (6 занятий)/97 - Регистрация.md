## 1. Регистрация — операция создания пользователя

Регистрация похожа на создание задачи, но цена ошибки выше: в запросе присутствует секрет, email должен быть уникальным, а ответ обязан быть уже очищенным от внутренних полей.

### Маршрут
1. **Валидация.** проверить формат email, длину username и password до вызова сервиса.
2. **Нормализация.** привести email к единой форме, чтобы регистр не создавал псевдодубликаты.
3. **Уникальность.** выполнить понятную предварительную проверку и сохранить ограничение базы.
4. **Хеширование.** создать password_hash до построения ORM-модели.
5. **Безопасный ответ.** вернуть 201 и UserRead без password/password_hash.

**Результат маршрута:** Endpoint остаётся тонким, а транзакционная операция живёт в сервисе.

### Карточки понятий

#### 201: Создано

Новая запись успешно сохранена и представлена через response schema.
```text
HTTP_201_CREATED
```

#### 422: Неверная форма

Pydantic отклоняет данные до предметной операции.
```text
invalid email / short password
```

#### 409: Конфликт

Email уже занят и новая запись нарушает уникальность.
```text
HTTP_409_CONFLICT
```

> **Связь и граница.** Регистрация создаёт аккаунт, но не обязана сразу создавать session или token. Это отдельное решение следующего слоя.
## 2. Входная схема останавливает плохую форму

Pydantic проверяет структуру, типы и локальные ограничения. Он не ходит в базу за уникальностью — эта проверка относится к сервису.

### Фокус сцены

Раздел показывает, как тема влияет на **UserCreate → service → Session → UserRead**.

**Перед продолжением объясните:** какие данные входят в механизм, какая проверка выполняется, что считается успехом и какой отказ ожидается.

> **Граница:** Не смешивайте создание аккаунта с выдачей cookie или JWT: регистрация заканчивается безопасной записью пользователя.

### Пример: UserCreate с Pydantic v2

```python
from pydantic import BaseModel, EmailStr, Field, field_validator


class UserCreate(BaseModel):
    email: EmailStr
    username: str = Field(min_length=2, max_length=50)
    password: str = Field(min_length=8, max_length=128)

    @field_validator("username")
    @classmethod
    def username_must_not_be_blank(cls, value: str) -> str:
        cleaned = value.strip()
        if not cleaned:
            raise ValueError("username не должен быть пустым")
        return cleaned
```

### Исследование ветвей

```python
payload = UserCreate.model_validate(data)
if email_has_bad_format:
    return 422
if username_is_blank:
    return 422
if password_is_short:
    return 422
return payload
```
- **email = bad** → строка 3: 422 до сервиса.
- **username = пробелы** → строка 5: 422 до сервиса.
- **корректная форма** → строка 8: UserCreate.

### Правда или ложь

**Утверждение:** Field(min_length=8) доказывает, что пароль устойчив к подбору.

**Ответ:** Ложь. Это только минимальное формальное ограничение. Реальная стойкость зависит от значения, политики и password hashing.
## 3. Нормализация email и предварительный поиск

До запроса к базе email приводится к согласованной форме. Иначе User@Example.com и user@example.com могут стать разными прикладными идентификаторами.

### Фокус сцены

Раздел показывает, как тема влияет на **UserCreate → service → Session → UserRead**.

**Перед продолжением объясните:** какие данные входят в механизм, какая проверка выполняется, что считается успехом и какой отказ ожидается.

> **Граница:** Не смешивайте создание аккаунта с выдачей cookie или JWT: регистрация заканчивается безопасной записью пользователя.

### Пример: нормализация и поиск

```python
from sqlalchemy import select
from sqlalchemy.orm import Session


def normalize_email(email: str) -> str:
    return email.strip().lower()


def get_user_by_email(db: Session, email: str) -> UserModel | None:
    statement = select(UserModel).where(UserModel.email == email)
    return db.scalar(statement)
```

### Пошаговое выполнение

```python
email = normalize_email(payload.email)
existing = get_user_by_email(db, email)
if existing is not None:
    raise EmailAlreadyExists
password_hash = hash_password(payload.password)
```
1. Строка 1. Email принимает единую форму до поиска и сохранения. Состояние: `email = "user@example.com"`.
2. Строка 2. Session выполняет SELECT по индексированной колонке. Состояние: `existing = None`.
3. Строка 3. Ветка конфликта проверяется до дорогого хеширования. Состояние: `condition = False`.
4. Строка 5. Только свободный email доходит до password service. Состояние: `password_hash = $argon2id$...`.

## 4. Явное преобразование UserCreate в UserModel

Не нужно передавать model_dump() целиком: во входной схеме есть password, а в ORM-модели требуется password_hash. Явное сопоставление делает границу заметной.

### Фокус сцены

Раздел показывает, как тема влияет на **UserCreate → service → Session → UserRead**.

**Перед продолжением объясните:** какие данные входят в механизм, какая проверка выполняется, что считается успехом и какой отказ ожидается.

> **Граница:** Не смешивайте создание аккаунта с выдачей cookie или JWT: регистрация заканчивается безопасной записью пользователя.

### Сравнение решений

**Вопрос:** Как безопаснее построить ORM-объект?

#### Вариант A. Распаковать весь payload

```python
user = UserModel(**payload.model_dump())
```

ORM не ждёт password, а секрет может попасть в неподходящее место.

#### Вариант B. Назвать поля явно

```python
user = UserModel(
    email=email,
    username=payload.username,
    password_hash=hash_password(payload.password),
)
```

Видно преобразование секрета и серверные defaults.

**Предпочтительный выбор:** Вариант B. На чувствительной границе явность важнее одной короткой строки.

### Пример: создание модели

```python
user = UserModel(
    email=email,
    username=payload.username,
    password_hash=hash_password(payload.password),
    role="user",
    is_active=True,
)

db.add(user)
```

### Заполните пропуск

```python

```

**Варианты:** `hash_password(payload.password)`, `payload.password`, `payload.model_dump()`

**Ответ:** `hash_password(payload.password)`. В сохраняемую модель передаётся только результат password hashing.
## 5. Commit, IntegrityError и обязательный rollback

Два параллельных запроса могут пройти предварительный SELECT почти одновременно. Unique constraint остановит второй commit, после чего Session нужно вернуть в рабочее состояние.

### Фокус сцены

Раздел показывает, как тема влияет на **UserCreate → service → Session → UserRead**.

**Перед продолжением объясните:** какие данные входят в механизм, какая проверка выполняется, что считается успехом и какой отказ ожидается.

> **Граница:** Не смешивайте создание аккаунта с выдачей cookie или JWT: регистрация заканчивается безопасной записью пользователя.

### Пример: транзакционная часть сервиса

```python
from sqlalchemy.exc import IntegrityError


def create_user(db: Session, payload: UserCreate) -> UserModel:
    email = normalize_email(str(payload.email))

    if get_user_by_email(db, email) is not None:
        raise EmailAlreadyExists

    user = UserModel(
        email=email,
        username=payload.username,
        password_hash=hash_password(payload.password),
    )
    db.add(user)

    try:
        db.commit()
    except IntegrityError as error:
        db.rollback()
        raise EmailAlreadyExists from error

    db.refresh(user)
    return user
```

### Поиск ошибки

```python
try:
    db.commit()
except IntegrityError:
    raise HTTPException(status_code=409)
```

**Вопрос:** Чего не хватает после неудачного commit?

**Варианты:**

- db.rollback()
- ещё одного db.add()
- повторного hash_password()

**Ответ:** db.rollback(). Session остаётся в failed transaction и не должна использоваться до rollback.

#### Исправленный вариант

```python
try:
    db.commit()
except IntegrityError as error:
    db.rollback()
    raise EmailAlreadyExists from error
```

> **Важно.** Сервис переводит SQLAlchemy-ошибку в прикладную EmailAlreadyExists. Router позже переводит прикладную ошибку в HTTP 409.
## 6. Тонкий endpoint и безопасный response_model

Router связывает HTTP с сервисом: получает payload и Session, вызывает use case, переводит ожидаемый конфликт и возвращает результат через UserRead.

### Фокус сцены

Раздел показывает, как тема влияет на **UserCreate → service → Session → UserRead**.

**Перед продолжением объясните:** какие данные входят в механизм, какая проверка выполняется, что считается успехом и какой отказ ожидается.

> **Граница:** Не смешивайте создание аккаунта с выдачей cookie или JWT: регистрация заканчивается безопасной записью пользователя.

### Пример: app/routers/auth.py

```python
from typing import Annotated

from fastapi import APIRouter, Depends, HTTPException, status
from sqlalchemy.orm import Session

from app.database import get_db
from app.schemas.user import UserCreate, UserRead
from app.services.users import EmailAlreadyExists, create_user


router = APIRouter(prefix="/auth", tags=["auth"])
DbSession = Annotated[Session, Depends(get_db)]


@router.post(
    "/register",
    response_model=UserRead,
    status_code=status.HTTP_201_CREATED,
)
def register_user(payload: UserCreate, db: DbSession):
    try:
        return create_user(db, payload)
    except EmailAlreadyExists as error:
        raise HTTPException(
            status_code=status.HTTP_409_CONFLICT,
            detail="Email уже зарегистрирован",
        ) from error
```

### Карточки понятий

#### router: HTTP-контракт

Path, status code, response model и перевод ожидаемой ошибки.
```text
POST /auth/register
```

#### service: Use case

Нормализация, уникальность, хеширование и транзакция.
```text
create_user(db, payload)
```

#### schema: Фильтр ответа

Разрешает только публичные поля UserRead.
```python
response_model=UserRead
```

### Правда или ложь

**Утверждение:** Endpoint регистрации должен сам создавать JWT, иначе пользователь не считается созданным.

**Ответ:** Ложь. Создание аккаунта и выдача способа доступа являются разными операциями и могут иметь разные правила.
## 7. Интеграционные проверки регистрации

Тесты проходят через HTTP-границу и отдельную тестовую базу. Они проверяют не внутренние строки хеша, а статус, сохранённые данные и отсутствие секретов в JSON.

### Фокус сцены

Раздел показывает, как тема влияет на **UserCreate → service → Session → UserRead**.

**Перед продолжением объясните:** какие данные входят в механизм, какая проверка выполняется, что считается успехом и какой отказ ожидается.

> **Граница:** Не смешивайте создание аккаунта с выдачей cookie или JWT: регистрация заканчивается безопасной записью пользователя.

### Пример: tests/test_register.py

```python
def test_register_user(client, db_session):
    response = client.post(
        "/auth/register",
        json={
            "email": "student@example.com",
            "username": "student",
            "password": "safe-password-42",
        },
    )

    assert response.status_code == 201
    body = response.json()
    assert body["email"] == "student@example.com"
    assert "password" not in body
    assert "password_hash" not in body

    user = get_user_by_email(db_session, "student@example.com")
    assert user is not None
    assert user.password_hash != "safe-password-42"
    assert verify_password("safe-password-42", user.password_hash)
```

### Компактная таблица

| Понятие / действие | Смысл |
| --- | --- |
| `201` | валидный пользователь создан |
| `422` | неверный email, короткий password или пустой username |
| `409` | повторный email отклонён |
| `response` | нет password и password_hash |
| `database` | email нормализован, пароль проверяется через verify |

## 8. Контрольная точка: полный конвейер регистрации

Ученик должен проследить одно значение password и доказать, что оно исчезает до ORM-модели, базы и ответа.

### Сборка последовательности: Соберите регистрацию

Расположите этапы от HTTP до ответа.

**Фрагменты:**
- `UserCreate валидирует форму`
- `нормализовать email`
- `проверить существующего пользователя`
- `вычислить password_hash`
- `add → commit → refresh`
- `вернуть UserRead и 201`
- `вернуть model_dump с password` — секрет не должен выходить наружу

**Правильный порядок:** `schema` → `normalize` → `lookup` → `hash` → `commit` → `response`.

**Объяснение:** Каждая граница выполняет одну роль и не передаёт открытый пароль дальше необходимого.

**Что делать после IntegrityError на commit?**

- B. refresh

**Ответ:** A. rollback. Session должна выйти из failed transaction.
