Database API уже хранит задачи и категории. Чтобы перейти к персональному приложению, каждой задаче нужен владелец, а серверу — устойчивая запись пользователя.

### Маршрут
1. **Описать User.** зафиксировать идентификатор, публичные поля, хеш пароля, активность и роль.
2. **Разделить схемы.** принимать пароль в UserCreate, но никогда не включать его или хеш в UserRead.
3. **Связать задачи.** добавить owner_id как внешний ключ на users.id.
4. **Изменить схему базы.** создать миграцию и осознанно обработать существующие записи.

**Результат маршрута:** После занятия модель готова к безопасной регистрации, но пароль ещё не хешируется — это следующий урок.

### Карточки понятий

#### ORM: UserModel

Описывает таблицу users и серверное состояние.
```text
email, password_hash, role
```

#### input: UserCreate

Контракт регистрации: email, username и сырой password только на входе.
```text

```

#### output: UserRead

Безопасное публичное представление пользователя.
```text

```

> **Связь и граница.** ORM-модель, request schema и response schema решают разные задачи. Одинаковая сущность не означает одинаковый набор полей на каждой границе.

## 3. SQLAlchemy-модель users

ORM-модель описывает таблицу и ограничения базы. Уникальность email должна существовать в базе, а не только в предварительной Python-проверке.

### Фокус сцены

Раздел показывает, как тема влияет на **UserModel, UserCreate, UserRead и связь Task.owner_id**.

**Перед продолжением объясните:** какие данные входят в механизм, какая проверка выполняется, что считается успехом и какой отказ ожидается.

> **Граница:** Не делайте ORM-модель публичным контрактом и не добавляйте password_hash в response schema.

### Пример: app/models/user.py

```python
from sqlalchemy import Boolean, String
from sqlalchemy.orm import Mapped, mapped_column, relationship

from app.database import Base


class UserModel(Base):
    __tablename__ = "users"

    id: Mapped[int] = mapped_column(primary_key=True)
    email: Mapped[str] = mapped_column(
        String(320), unique=True, index=True, nullable=False
    )
    username: Mapped[str] = mapped_column(String(50), nullable=False)
    password_hash: Mapped[str] = mapped_column(String(255), nullable=False)
    is_active: Mapped[bool] = mapped_column(Boolean, default=True, nullable=False)
    role: Mapped[str] = mapped_column(String(20), default="user", nullable=False)

    tasks: Mapped[list["TaskModel"]] = relationship(back_populates="owner")
```

### Карточки понятий

#### unique: Гарантия базы

Две строки не могут сохранить один email даже при конкурентных запросах.
```python
unique=True
```

#### index: Поиск входа

Индекс помогает находить пользователя по email.
```python
WHERE users.email = ?
```

#### nullable: Обязательность

Сервер не может сохранить пользователя без ключевых данных.
```python
nullable=False
```

## 4. Request и response schemas

Pydantic-схемы проектируются по направлению данных. Пароль нужен сервису регистрации, а password_hash нужен только внутреннему слою и базе.

### Фокус сцены

Раздел показывает, как тема влияет на **UserModel, UserCreate, UserRead и связь Task.owner_id**.

**Перед продолжением объясните:** какие данные входят в механизм, какая проверка выполняется, что считается успехом и какой отказ ожидается.

> **Граница:** Не делайте ORM-модель публичным контрактом и не добавляйте password_hash в response schema.

### Пример: app/schemas/user.py

```python
from pydantic import BaseModel, ConfigDict, EmailStr, Field


class UserCreate(BaseModel):
    email: EmailStr
    username: str = Field(min_length=2, max_length=50)
    password: str = Field(min_length=8, max_length=128)


class UserRead(BaseModel):
    model_config = ConfigDict(from_attributes=True)

    id: int
    email: EmailStr
    username: str
    is_active: bool
    role: str
```

### Сравнение решений

**Вопрос:** Какой response_model безопасен для регистрации?

#### Вариант A. UserModel как есть

```python
return db_user.__dict__
```

Внутренние поля и password_hash могут попасть в ответ.

#### Вариант B. Явный UserRead

```python
@router.post(..., response_model=UserRead)
```

Ответ ограничен заранее названными публичными полями.

**Предпочтительный выбор:** Вариант B. Безопасность ответа строится через allowlist полей, а не через надежду удалить секрет позже.

## 5. User владеет задачами

Связь one-to-many означает: один пользователь может иметь много задач, каждая задача принадлежит одному пользователю.

### Фокус сцены

Раздел показывает, как тема влияет на **UserModel, UserCreate, UserRead и связь Task.owner_id**.

**Перед продолжением объясните:** какие данные входят в механизм, какая проверка выполняется, что считается успехом и какой отказ ожидается.

> **Граница:** Не делайте ORM-модель публичным контрактом и не добавляйте password_hash в response schema.

### Пример: дополнение TaskModel

```python
from sqlalchemy import ForeignKey
from sqlalchemy.orm import Mapped, mapped_column, relationship


class TaskModel(Base):
    __tablename__ = "tasks"

    id: Mapped[int] = mapped_column(primary_key=True)
    title: Mapped[str]
    owner_id: Mapped[int] = mapped_column(
        ForeignKey("users.id"), nullable=False, index=True
    )

    owner: Mapped["UserModel"] = relationship(back_populates="tasks")
```

### Карточки понятий

#### FK: owner_id

Реальная колонка и ограничение ссылки на users.id.
```text
ForeignKey("users.id")
```

#### ORM: task.owner

Навигация от задачи к объекту пользователя.
```text
relationship(...)
```

#### collection: user.tasks

Навигация от пользователя к его задачам.
```text
Mapped[list[TaskModel]]
```

### Правда или ложь

**Утверждение:** relationship создаёт внешний ключ owner_id автоматически, даже если mapped_column отсутствует.

**Ответ:** Ложь. Внешний ключ задаётся колонкой. relationship добавляет ORM-навигацию поверх схемы.

Если таблица tasks уже содержит строки, сразу добавить обязательный owner_id обычно нельзя: старым записям нечего записать в новую колонку.

> **Граница:** Не делайте ORM-модель публичным контрактом и не добавляйте password_hash в response schema.

### Сборка последовательности: Соберите безопасный план миграции

Расположите этапы для базы с существующими задачами.

**Фрагменты:**
- `создать таблицу users`
- `добавить owner_id nullable=True`
- `создать технического/первого пользователя`
- `заполнить owner_id старых задач`
- `сделать owner_id nullable=False`
- `сразу nullable=False без default` — старые строки нарушат ограничение

**Правильный порядок:** `users` → `nullable` → `system` → `backfill` → `constraint`.

**Объяснение:** Схема и данные изменяются последовательными проверяемыми шагами.

### Терминальная демонстрация: Alembic workflow

```text

$ alembic upgrade head
Running upgrade ...
```

> **Важно.** Autogenerate предлагает изменения схемы, но не знает бизнес-правило заполнения старых owner_id. Data migration нужно спроектировать вручную.
## 7. Безопасная сериализация и границы модели

Хеш не является открытым паролем, но остаётся чувствительным материалом. Его утечка даёт атакующему возможность подбирать пароль вне сервера.

### Фокус сцены

Раздел показывает, как тема влияет на **UserModel, UserCreate, UserRead и связь Task.owner_id**.

**Перед продолжением объясните:** какие данные входят в механизм, какая проверка выполняется, что считается успехом и какой отказ ожидается.

> **Граница:** Не делайте ORM-модель публичным контрактом и не добавляйте password_hash в response schema.

### Поиск ошибки

```python
@router.get("/users/{user_id}")
def get_user(user_id: int, db: Session):
    user = db.get(UserModel, user_id)
    return user.__dict__
```

**Вопрос:** Почему возврат __dict__ опасен?

**Варианты:**

- В ответ могут попасть внутренние поля и password_hash
- Pydantic запрещает словари
- id станет строкой

**Ответ:** В ответ могут попасть внутренние поля и password_hash. Внутренняя модель шире публичного контракта.

#### Исправленный вариант

```python
@router.get("/users/{user_id}", response_model=UserRead)
def get_user(user_id: int, db: Session):
    return get_user_or_404(db, user_id)
```

### Карточки для самопроверки

- **password**
  - Кратковременно существует на входной границе и передаётся в hash_password.

- **password_hash**
  - Хранится в users, используется verify_password, не сериализуется наружу.

- **role**
  - Хранится сервером; клиент может читать только при необходимости, но не назначает.

- **email**
  - Публичность зависит от продукта; в текущем API входит в собственный профиль.

> **Связь и граница.** Response schema является техническим контрактом и частью защиты от случайной утечки новых внутренних колонок.
