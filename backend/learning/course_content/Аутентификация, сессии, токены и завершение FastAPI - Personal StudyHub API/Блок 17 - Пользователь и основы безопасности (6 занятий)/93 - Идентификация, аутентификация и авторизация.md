## 1. Один запрос — три разных вопроса

До этого Database API работал с задачами без понятия пользователя. Теперь одна и та же команда должна давать разный результат в зависимости от того, кто отправил запрос и принадлежит ли ему ресурс.

### Маршрут
1. **Идентифицировать.** получить заявленную личность, например email, username или идентификатор сессии.
2. **Аутентифицировать.** проверить доказательство: пароль, session id, API key или токен.
3. **Авторизовать.** сопоставить подтверждённого пользователя с правилом доступа к конкретному действию.

**Результат маршрута:** Результат урока — функция проверки владельца и ясная карта ответов 401/403.

### Карточки понятий

#### до блока: Общий API

Любой клиент видит одну коллекцию задач. У запроса нет подтверждённого владельца.
```text
GET /tasks
```

#### после: Персональный контекст

Система знает current user и проверяет связь задачи с ним.
```python
task.owner_id == current_user.id
```

#### граница: Без токенов пока

В этом блоке строится модель безопасности. Cookie-сессии и JWT появятся позднее.
```text
authenticate → authorize
```

> **Связь и граница.** Аутентификация не выдаёт права автоматически: подтверждённый пользователь всё ещё может не иметь доступа к чужой задаче.

## 3. Пользователь, роль и владелец ресурса

Права можно получить по разным причинам. Обычный пользователь управляет своими задачами, администратор может выполнять отдельные служебные операции, а неактивный пользователь не должен продолжать работу.

### Фокус сцены

Раздел показывает, как тема влияет на **current_user, role, owner_id и HTTP-статус**.

**Перед продолжением объясните:** какие данные входят в механизм, какая проверка выполняется, что считается успехом и какой отказ ожидается.

> **Граница:** Не подменяйте авторизацию фактом успешного входа: право проверяется для конкретного действия и ресурса.

### Компактная таблица

| Понятие / действие | Смысл |
| --- | --- |
| `current_user` | подтверждённый пользователь текущего запроса |
| `role` | общая категория полномочий: user или admin |
| `owner_id` | ссылка ресурса на конкретного владельца |
| `is_active` | может ли учётная запись использовать приложение |

### Исследование ветвей

```python
if current_user is None:
    return 401
if not current_user.is_active:
    return 403
if task.owner_id == current_user.id:
    return 200
if current_user.role == "admin":
    return 200
return 403
```
- **нет пользователя** → строка 2: 401: личность не подтверждена.
- **своя задача** → строка 6: 200: владелец совпал.
- **чужая задача** → строка 9: 403: пользователь известен, права нет.

## 4. 401 и 403 сообщают о разных проблемах

HTTP-статус является частью контракта API. Клиент по нему решает: запросить вход заново или показать сообщение о недостатке прав.

### Фокус сцены

Раздел показывает, как тема влияет на **current_user, role, owner_id и HTTP-статус**.

**Перед продолжением объясните:** какие данные входят в механизм, какая проверка выполняется, что считается успехом и какой отказ ожидается.

> **Граница:** Не подменяйте авторизацию фактом успешного входа: право проверяется для конкретного действия и ресурса.

### Сравнение решений

**Вопрос:** Какой статус выбрать для подтверждённого пользователя, который открывает чужую задачу?

#### Вариант A. 401 Unauthorized

```python
raise HTTPException(status_code=401)
```

Подходит, когда сервер не получил действительное подтверждение личности.

#### Вариант B. 403 Forbidden

```python
raise HTTPException(status_code=403)
```

Подходит, когда пользователь известен, но действие ему запрещено.

**Предпочтительный выбор:** Вариант B. Пользователь уже аутентифицирован, поэтому проблема находится на уровне авторизации.

### Карточки понятий

#### 401: Нужно подтвердить личность

Нет credentials, они неверны или срок доказательства истёк.
```text
WWW-Authenticate: Bearer
```

#### 403: Личность известна

Правило роли, владельца или состояния аккаунта запрещает действие.
```python
detail="Forbidden"
```

#### 404: Ресурс не найден

Иногда API скрывает существование чужого ресурса через 404, но это должно быть единым осознанным правилом.
```python
detail="Task not found"
```

> **Важно.** Название 401 исторически сбивает с толку: на практике его читают как «не аутентифицирован», а 403 — «аутентифицирован, но запрещено».
## 5. Путь защищённого запроса

Разберём последовательность до предметной логики. Чем раньше отклоняется недействительный запрос, тем меньше кода получает сомнительные данные.

### Фокус сцены

Раздел показывает, как тема влияет на **current_user, role, owner_id и HTTP-статус**.

**Перед продолжением объясните:** какие данные входят в механизм, какая проверка выполняется, что считается успехом и какой отказ ожидается.

> **Граница:** Не подменяйте авторизацию фактом успешного входа: право проверяется для конкретного действия и ресурса.

### Пошаговое выполнение

```python
def update_task(task_id, payload, current_user, db):
    task = get_task_or_404(db, task_id)
    ensure_active(current_user)
    ensure_owner(task, current_user)
    task.title = payload.title
    db.commit()
    return task
```
1. Строка 1. FastAPI уже передал проверенный payload и зависимость current_user. Состояние: `task_id = 17`, `current_user.id = 4`.
2. Строка 2. Задача загружается из базы или запрос получает 404. Состояние: `task.owner_id = 4`.
3. Строка 3. Неактивная учётная запись останавливается до изменения данных. Состояние: `is_active = True`.
4. Строка 4. Сравнивается владелец конкретной записи. Состояние: `4 == 4 = True`.
5. Строка 5. Только после проверок меняется предметное поле. Состояние: `title = "Новая тема"`.
6. Строка 6. Успешное изменение фиксируется транзакцией. Состояние: `commit = done`.

### Сборка последовательности: Соберите защиту endpoint

Расположите этапы до изменения задачи.

**Фрагменты:**
- `получить current_user`
- `загрузить task`
- `проверить is_active`
- `сравнить owner_id`
- `изменить task`
- `выполнить commit`
- `сначала изменить, потом проверить` — проверка слишком поздняя

**Правильный порядок:** `identity` → `load` → `active` → `owner` → `change` → `commit`.

**Объяснение:** Доверие и право проверяются до предметного изменения и commit.
## 6. Явная функция проверки владельца

Повторяемое правило лучше назвать. Функция получает ресурс и пользователя, ничего не ищет глобально и либо завершает проверку, либо останавливает запрос.

### Фокус сцены

Раздел показывает, как тема влияет на **current_user, role, owner_id и HTTP-статус**.

**Перед продолжением объясните:** какие данные входят в механизм, какая проверка выполняется, что считается успехом и какой отказ ожидается.

> **Граница:** Не подменяйте авторизацию фактом успешного входа: право проверяется для конкретного действия и ресурса.

### Пример: access.py

```python
from fastapi import HTTPException, status


def ensure_task_owner(task, current_user) -> None:
    if task.owner_id != current_user.id:
        raise HTTPException(
            status_code=status.HTTP_403_FORBIDDEN,
            detail="Недостаточно прав",
        )
```

### Заполните пропуск

**Задание:** Дополните сравнение владельца.

```python
if task.owner_id ____ current_user.id:
    raise HTTPException(status_code=403)
```

**Варианты:** `!=`, `==`, `in`

**Ответ:** `!=`. Запрет возникает, когда идентификаторы владельца и пользователя не совпали.

### Сравнение решений

**Вопрос:** Где понятнее хранить правило владельца?

#### Вариант A. Повтор в каждом endpoint

```python
if task.owner_id != user.id:
    raise HTTPException(403)
```

Правило легко изменить не во всех местах.

#### Вариант B. Одна именованная проверка

```python
ensure_task_owner(task, current_user)
```

Смысл читается в endpoint, реализация находится в одном месте.

**Предпочтительный выбор:** Вариант B. Общее правило имеет одно имя и одну точку изменения.
## 7. Типичные ошибки проектирования доступа

Большинство ошибок здесь не синтаксические. Код запускается, но доверяет данным клиента или проверяет право не на тот объект.

### Фокус сцены

Раздел показывает, как тема влияет на **current_user, role, owner_id и HTTP-статус**.

**Перед продолжением объясните:** какие данные входят в механизм, какая проверка выполняется, что считается успехом и какой отказ ожидается.

> **Граница:** Не подменяйте авторизацию фактом успешного входа: право проверяется для конкретного действия и ресурса.

### Поиск ошибки

```python
def update_task(task_id, payload, current_user, db):
    task = db.get(TaskModel, task_id)
    task.owner_id = payload.owner_id
    task.title = payload.title
    db.commit()
```

**Вопрос:** Какая строка позволяет захватить чужую задачу?

**Варианты:**

- owner_id берётся из тела запроса
- title изменяется строкой
- используется db.get

**Ответ:** owner_id берётся из тела запроса. Владелец является серверным фактом. Клиент не должен назначать его произвольным значением.

#### Исправленный вариант

```python
task = db.get(TaskModel, task_id)
ensure_task_owner(task, current_user)
task.title = payload.title
db.commit()
```

### Карточки для самопроверки

- **«В теле owner_id = 7»**
  - Это только вход клиента, а не доказательство личности.

- **«Пользователь admin»**
  - Роль проверяется сервером по загруженной модели User.

- **«Задача существует»**
  - Существование не означает право чтения или изменения.

- **«Пароль совпал»**
  - Аутентификация завершена, но авторизация ещё нужна.

> **Связь и граница.** Не доверяйте owner_id, role и is_active из пользовательского JSON. Эти значения загружаются из доверенного серверного состояния.

## Итоговый квиз

**Что делает идентификация?**
- A. называет предполагаемую личность

### Вопрос 2

**Когда подходит статус 401?**
- A. личность не подтверждена
- B. пользователь открыл чужую задачу
- C. поле title слишком длинное

**Ответ:** A. личность не подтверждена. 401 относится к отсутствующим или недействительным credentials.

### Вопрос 3

**Что проверяет авторизация владельца?**
- A. task.owner_id == current_user.id
- B. payload.owner_id существует
- C. пароль длиннее восьми символов

**Ответ:** A. task.owner_id == current_user.id. Сравниваются серверные идентификаторы ресурса и подтверждённого пользователя.

### Вопрос 4

**Почему нельзя доверять role из JSON клиента?**

- B. JSON не умеет хранить строки
- C. роль всегда равна user

## Главное из занятия

- Идентификация называет личность, аутентификация проверяет доказательство.
- Авторизация выполняется после аутентификации и относится к конкретному действию.
- 401 означает отсутствие действительной аутентификации, 403 — недостаток прав.
- owner_id является серверным фактом конкретного ресурса.
- Роль и владение решают разные задачи.
- Проверки доступа выполняются до изменения модели и commit.
