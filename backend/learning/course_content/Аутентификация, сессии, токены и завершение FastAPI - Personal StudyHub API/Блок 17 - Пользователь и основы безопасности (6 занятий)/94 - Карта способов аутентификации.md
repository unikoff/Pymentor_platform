## 1. Почему одного слова «логин» недостаточно

Разные проекты подтверждают пользователя разными механизмами. Выбор нельзя делать по популярности: нужно понимать транспорт credentials, место хранения состояния и способ отзыва доступа.

### Маршрут
1. **Что передаёт клиент.** пароль, ключ, идентификатор сессии или токен.
2. **Что хранит сервер.** учётные данные, активные сессии, отозванные токены или только секрет проверки подписи.
3. **Как завершить доступ.** удалить server-side session, отозвать ключ, дождаться TTL или вести дополнительное состояние.
4. **Для кого механизм.** человек в браузере, мобильное приложение, внутренний сервис или внешний интегратор.

**Результат маршрута:** Итог — сравнительная карта без преждевременной реализации cookie и JWT.

### Карточки понятий

#### credential: Доказательство

То, что клиент предъявляет серверу: пароль, API key, session id или token.
```text
Authorization / Cookie
```

#### transport: Способ передачи

HTTP-заголовок или cookie. Транспорт не определяет внутреннее устройство проверки.
```text
Bearer <token>
```

#### state: Состояние

Данные, которые сервер должен помнить между запросами.
```text
sessions[session_id]
```

> **Связь и граница.** Cookie и Bearer — способы доставки значения. Внутри cookie может быть session id, а Bearer token может быть opaque-строкой или JWT.
## 2. HTTP Basic и API key

Basic передаёт логин и пароль при каждом запросе, а API key обычно идентифицирует приложение или интеграцию. Оба механизма требуют HTTPS: кодирование не делает секрет зашифрованным.

### Фокус сцены

Раздел показывает, как тема влияет на **credential, транспорт, server-side state и способ revoke**.

**Перед продолжением объясните:** какие данные входят в механизм, какая проверка выполняется, что считается успехом и какой отказ ожидается.

> **Граница:** Не реализуйте все схемы одновременно: сначала сравните их, затем выберите одну под пользовательский сценарий.

### Сравнение решений

**Вопрос:** Как точнее описать заголовок Basic?

#### Вариант A. «Пароль зашифрован Base64»

```python
Authorization: Basic dXNlcjpwYXNz
```

Base64 легко декодируется и не является шифрованием.

#### Вариант B. «Пара закодирована для передачи»

```python
base64("user:pass")
```

Конфиденциальность обеспечивает HTTPS, а не Base64.

**Предпочтительный выбор:** Вариант B. Кодирование меняет представление данных, но не скрывает секрет от того, кто их перехватил.

### Компактная таблица

| Понятие / действие | Смысл |
| --- | --- |
| `HTTP Basic` | простая встроенная схема; пароль участвует в каждом запросе |
| `API key` | длинный секрет для клиента или интеграции; может иметь отдельные права |
| `HTTPS` | защищает credentials при передаче по сети |
| `rotation` | замена ключа без смены всей учётной записи |

### Правда или ложь

**Утверждение:** API key всегда обозначает конкретного человека и подходит для браузерного логина.

**Ответ:** Ложь. Чаще ключ выдают приложению или интеграции. Модель владельца и правила использования проектируются отдельно.
## 3. Cookie и server-side session

При серверной сессии после успешного входа сервер создаёт случайный session id, хранит связанную запись у себя и отправляет идентификатор браузеру в cookie.

### Фокус сцены

Раздел показывает, как тема влияет на **credential, транспорт, server-side state и способ revoke**.

**Перед продолжением объясните:** какие данные входят в механизм, какая проверка выполняется, что считается успехом и какой отказ ожидается.

> **Граница:** Не реализуйте все схемы одновременно: сначала сравните их, затем выберите одну под пользовательский сценарий.

### Пошаговое выполнение

```python
user = authenticate_user(email, password)
session_id = create_session(user.id)
response.set_cookie("session_id", session_id)

# следующий запрос
session_id = request.cookies.get("session_id")
user_id = find_active_session(session_id)
```
1. Строка 1. Пароль проверяется один раз во время входа. Состояние: `user.id = 12`.
2. Строка 2. Сервер создаёт непредсказуемый идентификатор и хранит соответствие. Состояние: `session_id = "a8f..."`.
3. Строка 3. Браузер получает cookie, а не пароль пользователя. Состояние: `Set-Cookie = session_id=a8f...`.
4. Строка 6. В следующем запросе браузер возвращает cookie. Состояние: `Cookie = session_id=a8f...`.
5. Строка 7. Сервер ищет активную запись и восстанавливает пользователя. Состояние: `user_id = 12`.

### Карточки понятий

#### browser: Cookie

Небольшое значение, которое браузер отправляет подходящему домену по правилам cookie.
```python
session_id=a8f...
```

#### server: Session store

Таблица или другое хранилище активных сессий и их срока жизни.
```text
session_id → user_id
```

#### logout: Удаление сессии

Сервер деактивирует запись, а ответ удаляет cookie у клиента.
```python
DELETE session; Max-Age=0
```

### Активное воспроизведение

**Вопрос:** Почему cookie не является готовой server-side session?

<details>
<summary>Показать ответ</summary>

Cookie только переносит значение между браузером и сервером. Сессия появляется, когда сервер создаёт и проверяет собственную запись состояния.

## 4. Opaque Bearer token

Bearer означает: обладатель значения получает связанный доступ. Opaque token выглядит как случайная строка без читаемой внутренней структуры; сервер ищет её в своём хранилище.

### Фокус сцены

Раздел показывает, как тема влияет на **credential, транспорт, server-side state и способ revoke**.

**Перед продолжением объясните:** какие данные входят в механизм, какая проверка выполняется, что считается успехом и какой отказ ожидается.

> **Граница:** Не реализуйте все схемы одновременно: сначала сравните их, затем выберите одну под пользовательский сценарий.

### Пример: заголовок запроса

```python
Authorization: Bearer Qx7pY2mK9...
```

### Исследование ветвей

```python
token = read_bearer_token(request)
record = find_token_hash(token)
if record is None:
    return 401
if record.revoked_at is not None:
    return 401
if record.expires_at <= now:
    return 401
return record.user_id
```
- **неизвестный token** → строка 4: 401.
- **отозванный token** → строка 6: 401.
- **активный token** → строка 9: user_id.

### Сравнение решений

**Вопрос:** Чем opaque token похож на session id?

#### Вариант A. Сервер хранит запись

```python
token_hash → user_id, expires_at
```

Для проверки нужен поиск серверного состояния.

#### Вариант B. Токен содержит весь профиль

```python
decode(token).user
```

Это уже другая модель, похожая на подписанный self-contained token.

**Предпочтительный выбор:** Вариант A. Opaque token сам по себе ничего не сообщает приложению; смысл находится в серверной записи.
## 5. JWT: подписанный, но не секретный контейнер

JWT обычно состоит из header, payload и signature. Подпись позволяет обнаружить изменение данных, но payload часто лишь кодирован и читается без секретного ключа.

### Фокус сцены

Раздел показывает, как тема влияет на **credential, транспорт, server-side state и способ revoke**.

**Перед продолжением объясните:** какие данные входят в механизм, какая проверка выполняется, что считается успехом и какой отказ ожидается.

> **Граница:** Не реализуйте все схемы одновременно: сначала сравните их, затем выберите одну под пользовательский сценарий.

### Карточки понятий

#### header: Метаданные

Описывает тип токена и алгоритм подписи.
```text
{"alg": "HS256", "typ": "JWT"}
```

#### payload: Claims

Содержит утверждения: subject, срок действия и другие минимальные данные.
```text
{"sub": "12", "exp": 1710000000}
```

#### signature: Подпись

Связывает предыдущие части с ключом и защищает от незаметного изменения.
```text
sign(header.payload, key)
```

### Поиск ошибки

```python
payload = {
    "sub": str(user.id),
    "password": user.password,
    "card_number": user.card_number,
}
```

**Вопрос:** Почему payload небезопасен даже при корректной подписи?

**Варианты:**

- Содержимое JWT обычно можно прочитать
- JWT не поддерживает числа
- Поле sub запрещено

**Ответ:** Содержимое JWT обычно можно прочитать. Подпись подтверждает целостность, но не шифрует пользовательские данные.

#### Исправленный вариант

```python
payload = {
    "sub": str(user.id),
    "exp": expires_at,
}
```

### Правда или ложь

**Утверждение:** Если JWT имеет корректную подпись, его payload автоматически скрыт от клиента.

**Ответ:** Ложь. Обычный подписанный JWT не является зашифрованным контейнером.
## 6. Access, refresh и OAuth2 — разные уровни

Access token обслуживает короткий доступ к API, refresh token помогает получить новую пару, а OAuth2 описывает протокол делегирования и несколько потоков. Эти понятия нельзя заменять друг другом.

### Фокус сцены

Раздел показывает, как тема влияет на **credential, транспорт, server-side state и способ revoke**.

**Перед продолжением объясните:** какие данные входят в механизм, какая проверка выполняется, что считается успехом и какой отказ ожидается.

> **Граница:** Не реализуйте все схемы одновременно: сначала сравните их, затем выберите одну под пользовательский сценарий.

### Сопоставление

**Задание:** Соедините понятие и его основную роль.
- **access token** → предъявляется защищённому API
- **refresh token** → используется для обновления доступа
- **OAuth2** → протокол выдачи делегированного доступа
- **Bearer** → схема предъявления токена обладателем

**Разбор:** Термины связаны, но описывают разные части системы.

### Сборка последовательности: Соберите типичный цикл пары токенов

Расположите действия без реализации криптографии.

**Фрагменты:**
- `проверить credentials`
- `выдать access и refresh`
- `использовать access для API`
- `access истёк`
- `предъявить refresh`
- `получить новую пару`

**Правильный порядок:** `login` → `issue` → `api` → `expire` → `refresh` → `rotate`.

**Объяснение:** Refresh не отправляют каждому endpoint и не используют как обычный access token.

> **Связь и граница.** В этом занятии строится карта. Реализация OAuth2 password flow, JWT, refresh rotation и revoke будет позже, после server-side sessions.
## 7. Сравнительная матрица и критерии выбора

Ни один механизм не выигрывает во всех строках. Удобство отзыва, объём серверного состояния и тип клиента образуют компромисс.

### Фокус сцены

Раздел показывает, как тема влияет на **credential, транспорт, server-side state и способ revoke**.

**Перед продолжением объясните:** какие данные входят в механизм, какая проверка выполняется, что считается успехом и какой отказ ожидается.

> **Граница:** Не реализуйте все схемы одновременно: сначала сравните их, затем выберите одну под пользовательский сценарий.

### Компактная таблица

| Понятие / действие | Смысл |
| --- | --- |
| `HTTP Basic` | минимум инфраструктуры; пароль участвует в каждом запросе |
| `API key` | удобен для интеграций; нужен выпуск, хранение, scope и revoke |
| `Cookie + session` | понятный browser login и простой централизованный logout |
| `Opaque Bearer` | токен в заголовке и серверный контроль состояния |
| `JWT` | проверяемая подпись и меньше lookup; revoke требует отдельного решения |

### Карточки для самопроверки

- **Нужен немедленный logout**
  - Server-side session или stateful token отзывается напрямую.

- **Внешняя интеграция**
  - API key или Bearer token обычно удобнее браузерной cookie.

- **Минимум данных в клиенте**
  - Храните только непрозрачный идентификатор, а смысл — на сервере.

- **JWT без state**
  - Учитывайте короткий TTL и сложность мгновенного revoke.

> **Важно.** Выбор механизма является частью модели угроз и пользовательского сценария, а не соревнованием по количеству аббревиатур.

**Что защищает HTTP Basic в сети?**
- A. HTTPS
- B. Base64

**Ответ:** A. HTTPS. Base64 только кодирует пару, а конфиденциальность канала обеспечивает HTTPS.

**Что обычно хранит cookie при server-side session?**

**Ответ:** A. случайный session id. Смысл и состояние сессии остаются на сервере.

**Что верно для обычного подписанного JWT?**
- A. payload можно прочитать
- B. payload всегда зашифрован
- C. подпись хранит пароль

**Ответ:** A. payload можно прочитать. Подпись обеспечивает целостность, а не секретность содержимого.

### Вопрос 4

**Для чего нужен refresh token?**
- A. получить новый доступ
- B. заменить owner_id

**Ответ:** A. получить новый доступ. Он обслуживает обновление доступа и требует отдельной защиты.

## Главное из занятия

- Механизмы сравнивают по credentials, серверному состоянию, клиенту и отзыву.

- Cookie переносит значение, а server-side session хранится на сервере.

- Подписанный JWT не скрывает payload.
- Access, refresh, Bearer и OAuth2 описывают разные уровни системы.
