## 1. Почему открытый пароль нельзя хранить

Пароль является секретом пользователя, а не данными профиля. Сервер должен проверять его, не имея возможности прочитать исходное значение из базы.

### Маршрут
1. **Получить пароль.** принять его только по защищённому соединению во входной схеме.
2. **Преобразовать.** вычислить медленный односторонний password hash через проверенную библиотеку.
3. **Сохранить.** записать только результат хеширования в password_hash.
4. **Проверить позже.** передать новый кандидат и сохранённый хеш функции verify.

**Результат маршрута:** После урока проект получает маленький password service и тесты его поведения.

### Сравнение решений

**Вопрос:** Какое хранение переживает утечку базы лучше?

#### Вариант A. Открытый password

```python
password = "qwerty123"
```

Секрет сразу пригоден для входа и повторного использования на других сайтах.

#### Вариант B. Password hash

```python
password_hash = "$argon2id$..."
```

Для проверки кандидата требуется дорогой подбор.

**Предпочтительный выбор:** Вариант B. Хеш не отменяет последствия слабого пароля, но не раскрывает исходный секрет напрямую.

> **Связь и граница.** Пароль не логируют, не включают в traceback вручную, не помещают в URL и не возвращают в ответе — даже на локальной разработке.
## 2. Хеширование, шифрование и кодирование

Три операции часто смешивают. Для хранения пароля нужен односторонний verifier, а не способ восстановить исходный текст.

**Перед продолжением объясните:** какие данные входят в механизм, какая проверка выполняется, что считается успехом и какой отказ ожидается.

> **Граница:** Не изобретайте алгоритм и не сравнивайте новый salted hash с сохранённой строкой.

### Карточки понятий

#### encode: Кодирование

Меняет представление для совместимости. Обратное преобразование не требует секрета.
```text
Base64
```

#### encrypt: Шифрование

Скрывает данные и предполагает расшифрование ключом.
```text
ciphertext ↔ plaintext
```

#### hash: Хеширование пароля

Создаёт verifier для проверки кандидата без восстановления исходного пароля.
```text
verify(candidate, stored_hash)
```

### Сопоставление

**Задание:** Соедините задачу и подходящий механизм.
- **передать бинарные данные текстом** → кодирование
- **позже восстановить секретный документ** → шифрование
- **проверить пароль пользователя** → password hashing

**Разбор:** Операции имеют разные цели и свойства.

### Правда или ложь

**Утверждение:** Если password_hash можно проверить, значит сервер сначала расшифровывает его в исходный пароль.

**Ответ:** Ложь. Verify запускает алгоритм над кандидатом и сравнивает результат по правилам формата хеша.
## 3. Salt разрушает одинаковые отпечатки

Случайный salt добавляется к каждому паролю перед вычислением. Поэтому одинаковые пароли разных пользователей получают разные хеши.

### Фокус сцены

Раздел показывает, как тема влияет на **password, password_hash, salt и verify_password**.

**Перед продолжением объясните:** какие данные входят в механизм, какая проверка выполняется, что считается успехом и какой отказ ожидается.

> **Граница:** Не изобретайте алгоритм и не сравнивайте новый salted hash с сохранённой строкой.

### Пошаговое выполнение

```python
first = password_hash.hash("same-password")
second = password_hash.hash("same-password")

print(first == second)
print(password_hash.verify("same-password", first))
print(password_hash.verify("wrong", first))
```
1. Строка 1. Библиотека создаёт новый случайный salt и хеш. Состояние: `first = $argon2id$...A`.
2. Строка 2. Для того же пароля создаётся другой salt. Состояние: `second = $argon2id$...B`.
3. Строка 4. Строки хешей различаются. Состояние: `first == second = False`.
4. Строка 5. Формат первого хеша содержит параметры и salt для корректной проверки. Состояние: `verify = True`.
5. Строка 6. Неверный кандидат не проходит проверку. Состояние: `verify = False`.

### Предскажите результат

```python
first = hash_password("secret123")
second = hash_password("secret123")
print(first == second)
print(verify_password("secret123", first))
```

**Подсказка:** Новый salt меняет строку хеша, но verify понимает его формат.

**Результат:**

```text
False
True
```

## 4. Стоимость вычисления замедляет подбор

Обычный быстрый SHA-256 хорош для контроля целостности файлов, но плох как самостоятельный password hash: атакующий может перебирать кандидаты слишком быстро.

### Фокус сцены

Раздел показывает, как тема влияет на **password, password_hash, salt и verify_password**.

**Перед продолжением объясните:** какие данные входят в механизм, какая проверка выполняется, что считается успехом и какой отказ ожидается.

> **Граница:** Не изобретайте алгоритм и не сравнивайте новый salted hash с сохранённой строкой.

### Сравнение решений

**Вопрос:** Какой подход подходит для паролей?

#### Вариант A. Один быстрый SHA-256

```python
sha256(password.encode()).hexdigest()
```

Очень быстрый массовый перебор и нет готовой политики параметров.

#### Вариант B. Специализированный password hash

```python
PasswordHash.recommended()
```

Алгоритм хранит salt и параметры стоимости в стандартном формате.

**Предпочтительный выбор:** Вариант B. Для паролей нужна управляемая стоимость вычисления и готовая реализация безопасной проверки.

### Компактная таблица

| Понятие / действие | Смысл |
| --- | --- |
| `time cost` | сколько раундов вычисления выполняется |
| `memory cost` | сколько памяти требует алгоритм |
| `parallelism` | настройка параллельной работы алгоритма |
| `rehash` | обновление старого хеша при изменении рекомендуемых параметров |

> **Важно.** Чем выше стоимость, тем дороже и вход пользователя, и атака. Параметры измеряют на реальном окружении, а не увеличивают бесконечно.

```text

Successfully installed pwdlib argon2-cffi ...
```

### Пример: app/security/passwords.py

```python
from pwdlib import PasswordHash


password_hash = PasswordHash.recommended()


def hash_password(password: str) -> str:
    return password_hash.hash(password)


def verify_password(password: str, stored_hash: str) -> bool:
    return password_hash.verify(password, stored_hash)
```

**Варианты:** `password, stored_hash`, `stored_hash, password`, `password_hash, password`

**Ответ:** `password, stored_hash`. Первый аргумент — открытый кандидат текущего входа, второй — verifier из базы.
## 6. Регистрация и вход используют функции по-разному

При регистрации пароль преобразуется один раз перед сохранением. При входе новый кандидат сравнивается с уже сохранённым хешем — новый хеш для прямого сравнения строк создавать нельзя.

### Фокус сцены

Раздел показывает, как тема влияет на **password, password_hash, salt и verify_password**.

**Перед продолжением объясните:** какие данные входят в механизм, какая проверка выполняется, что считается успехом и какой отказ ожидается.

> **Граница:** Не изобретайте алгоритм и не сравнивайте новый salted hash с сохранённой строкой.

### Исследование ветвей

```python
# регистрация
stored_hash = hash_password(password)

# вход
if verify_password(password, stored_hash):
    return authenticated_user
return None
```
- **регистрация** → строка 2: создан новый salted hash.
- **верный пароль** → строка 5: authenticated_user.
- **неверный пароль** → строка 7: None.

### Поиск ошибки

```python
candidate_hash = hash_password(login.password)
if candidate_hash == user.password_hash:
    return user
```

**Вопрос:** Почему сравнение почти всегда ложно?

**Варианты:**

- Новый hash получает новый salt
- Строки нельзя сравнивать
- password_hash всегда None

**Ответ:** Новый hash получает новый salt. Для salted hashes используют verify, а не повторное хеширование и сравнение строк.

#### Исправленный вариант

```python
if verify_password(login.password, user.password_hash):
    return user
```

> **Связь и граница.** Password service не знает о FastAPI, Session и HTTPException. Он получает строки и возвращает строку или bool.
## 7. Тестируем контракт, а не случайную строку

Из-за salt точная строка хеша меняется. Тест должен проверять полезное поведение: исходный пароль проходит, неверный — нет, открытый пароль не сохраняется как результат.

### Пример: tests/test_passwords.py

```python
from app.security.passwords import hash_password, verify_password


def test_hash_and_verify_password():
    raw = "correct-horse-42"

    stored_hash = hash_password(raw)

    assert stored_hash != raw
    assert verify_password(raw, stored_hash) is True
    assert verify_password("wrong-password", stored_hash) is False


def test_same_password_gets_different_hashes():
    first = hash_password("same-password")
    second = hash_password("same-password")

    assert first != second
    assert verify_password("same-password", first)
    assert verify_password("same-password", second)
```

### Сравнение решений

**Вопрос:** Какой тест устойчив к случайному salt?

#### Вариант A. Сравнить с фиксированной строкой


Зависит от salt и конкретных параметров.

#### Вариант B. Проверить поведение

```python

```

Фиксирует контракт, а не внутреннюю строку.

**Предпочтительный выбор:** Вариант B. Тесты должны позволять библиотеке безопасно менять случайные данные и параметры.

### Правда или ложь

## 8. Контрольная точка: безопасный жизненный цикл

Пароль должен существовать открытым минимально необходимое время и не выходить за входную границу, password service и вызов verify.

### Сборка последовательности: Соберите путь регистрации

Расположите действия до сохранения User.

**Фрагменты:**
- `получить UserCreate`
- `проверить ограничения password`
- `вызвать hash_password`
- `создать UserModel(password_hash=...)`
- `сохранить и вернуть UserRead`

**Правильный порядок:** `receive` → `validate` → `hash` → `model` → `commit`.

**Объяснение:** Открытый пароль заменяется хешем до создания сохраняемой модели.

## Главное из занятия

- Открытый пароль нельзя сохранять, логировать или возвращать.
- Кодирование, шифрование и password hashing решают разные задачи.
- Salt делает хеши одинаковых паролей различными.
- Специализированный алгоритм намеренно дорог для массового перебора.
- pwdlib скрывается за маленькими функциями hash_password и verify_password.
- Salted hashes проверяются через verify, а не сравнением новых строк.
