# 99 — Cookie в браузере

## Паспорт занятия

- **Блок:** 18 · Cookie и серверные сессии
- **Проект:** Personal StudyHub API
- **Продолжительность:** 60–90 минут
- **До занятия:** пользователь регистрируется и authenticate_user умеет проверить credentials
- **После занятия:** ученик умеет установить, прочитать, удалить и диагностировать cookie
- **Главная модель:** сервер отправляет Set-Cookie, браузер хранит значение и возвращает его в Cookie следующего подходящего запроса

## Результат занятия

Вы создадите три учебных endpoint для cookie, проверите атрибуты HttpOnly, Secure, SameSite, Path и Max-Age, а затем воспроизведёте сценарий через один TestClient.

## Введение

Успешный login — это только один HTTP-запрос. Следующий запрос приходит отдельно, поэтому серверу нужен маркер, который клиент сможет вернуть. Cookie решает транспортную часть задачи, но пока не подтверждает пользователя.

## 1. Почему login не запоминается автоматически

HTTP не хранит историю предыдущих запросов за endpoint. После проверки email и пароля сервер не может определить пользователя в следующем GET только по факту прошлого успеха.

Cookie позволяет связать последовательность запросов на уровне клиента: браузер сохраняет небольшое значение и автоматически прикрепляет его к подходящим URL.

### Маршрут значения

1. POST /auth/session/login проверяет credentials.
2. Response содержит Set-Cookie.
3. Браузер сохраняет cookie.
4. Следующий GET содержит заголовок Cookie.
5. Сервер проверяет полученное значение.

### Задание: объясните границу

Почему наличие cookie ещё не доказывает личность?

**Ответ:** Cookie хранится у клиента и может быть удалена, заменена или скопирована. Сервер должен проверить её по собственной модели session.

> **Профессиональная граница:** Не называйте cookie сессией. Cookie — способ доставки значения; server-side session появится в уроке 100.

## 2. Set-Cookie и Cookie — разные направления

`Set-Cookie` находится в HTTP-ответе и даёт браузеру инструкцию сохранить запись. `Cookie` находится в следующем HTTP-запросе и содержит подходящие пары name=value.

В DevTools эти заголовки находятся в разных местах. Для диагностики всегда проверяйте сначала ответ login, затем следующий request.

### Ответ сервера

```http
HTTP/1.1 200 OK
Set-Cookie: studyhub_demo=lesson-99; Path=/; HttpOnly; SameSite=Lax
```

### Следующий запрос

```http
GET /demo/cookie HTTP/1.1
Cookie: studyhub_demo=lesson-99
```

## 3. Атрибуты cookie

`HttpOnly` ограничивает чтение через клиентский JavaScript. `Secure` разрешает отправку только по HTTPS. `SameSite` влияет на межсайтовую отправку. `Path` задаёт область URL, а `Max-Age` — клиентский срок хранения.

Ни один отдельный атрибут не превращает значение в доверенное. Они уменьшают поверхность риска и задают правила браузеру.

### Карта атрибутов

| Атрибут | Назначение |
| --- | --- |
| `HttpOnly` | не отдавать значение `document.cookie` |
| `Secure` | отправлять только по HTTPS |
| `SameSite=Lax` | ограничить часть cross-site запросов |
| `Path=/` | разрешить маршрутам всего приложения |
| `Max-Age=1800` | хранить у клиента 30 минут |

### Задание: проверьте утверждение

HttpOnly-cookie не отправляется браузером автоматически.

**Ответ:** Неверно. HttpOnly ограничивает JavaScript, но браузер продолжает отправлять cookie серверу.

## 4. Установка cookie в FastAPI

Endpoint получает объект `Response` и вызывает `set_cookie`. При этом он может вернуть обычный словарь или Pydantic-модель: FastAPI добавит заголовок к финальному ответу.

### установить cookie

```python
from fastapi import FastAPI, Response

app = FastAPI()

@app.post("/demo/cookie")
def set_demo_cookie(response: Response):
    response.set_cookie(
        key="studyhub_demo",
        value="lesson-99",
        httponly=True,
        samesite="lax",
        path="/",
    )
    return {"status": "saved"}
```

> **Профессиональная граница:** `set_cookie` формирует response header. Она не изменяет текущий request и не записывает данные в SQLAlchemy.

## 5. Чтение и удаление

Для чтения используется параметр `Cookie` с `alias`, равным HTTP-имени. Значение делается optional, чтобы функция могла осмысленно обработать отсутствие.

При удалении должны совпадать имя и область исходной cookie, прежде всего `path` и при необходимости `domain`.

### прочитать и удалить

```python
from typing import Annotated
from fastapi import Cookie, Response

@app.get("/demo/cookie")
def read_demo_cookie(
    value: Annotated[str | None, Cookie(alias="studyhub_demo")] = None,
):
    return {"cookie_value": value}

@app.delete("/demo/cookie")
def delete_demo_cookie(response: Response):
    response.delete_cookie(key="studyhub_demo", path="/")
    return {"status": "deleted"}
```

## 6. Почему нельзя хранить профиль и права

Клиент контролирует собственное хранилище. Поэтому поля вроде `role=admin`, `user_id=7` или `password_hash` нельзя принимать как источник истины.

В дальнейшем cookie будет содержать только непрозрачный случайный token. Пользователь и права будут загружаться из базы после server-side проверки.

### Задание: выберите контракт

Что лучше хранить: JSON профиля или случайный token?

- JSON с id и role
- Непрозрачный случайный token

**Ответ:** Непрозрачный token. Он только указывает на server-side session и не назначает права.

> **Профессиональная граница:** Cookie не должна содержать пароль или password_hash ни при каких учебных упрощениях.

## 7. Диагностика через DevTools и TestClient

Последовательный тест выполняется одним объектом клиента, потому что cookie jar принадлежит конкретному экземпляру. Два новых `TestClient(app)` не разделяют сохранённые cookies.

### один клиент

```python
from fastapi.testclient import TestClient

client = TestClient(app)
client.post("/demo/cookie")

response = client.get("/demo/cookie")
assert response.json() == {"cookie_value": "lesson-99"}

client.delete("/demo/cookie")
assert client.get("/demo/cookie").json() == {"cookie_value": None}
```

### Четыре точки проверки

1. Set-Cookie в response.
2. Cookie jar браузера или TestClient.
3. Совпадение Path и Domain.
4. Cookie в следующем request.

## 8. Собранная модель

Cookie — это управляемый браузером транспорт небольшого значения. Сервер задаёт атрибуты, но всё равно обязан считать значение недоверенным и проверить его.

На следующем занятии учебное значение `lesson-99` будет заменено на криптографически случайный token, связанный с таблицей `user_sessions`.

### Задание: объясните своими словами

Опишите путь cookie от response до следующего request.

**Ответ:** Set-Cookie инструктирует браузер сохранить запись; браузер проверяет область и атрибуты; затем автоматически формирует Cookie подходящему URL.

## Итоговый квиз

### Вопрос 1

Какой заголовок устанавливает cookie?

- A. Set-Cookie в response
- B. Cookie в response
- C. Authorization

**Правильный ответ:** A. Set-Cookie в response

**Почему:** Сервер передаёт инструкцию через Set-Cookie.

### Вопрос 2

Что делает HttpOnly?

- A. Ограничивает чтение JavaScript
- B. Шифрует cookie
- C. Создаёт session в базе

**Правильный ответ:** A. Ограничивает чтение JavaScript

**Почему:** Браузер всё ещё отправляет cookie, но JS не читает её напрямую.

### Вопрос 3

Можно ли доверять role из cookie?

- A. Нет
- B. Да
- C. Только в GET

**Правильный ответ:** A. Нет

**Почему:** Клиент может изменить значение.

### Вопрос 4

Почему тест использует один TestClient?

- A. Он хранит cookie jar
- B. Он создаёт SQL
- C. Он компилирует Pydantic

**Правильный ответ:** A. Он хранит cookie jar

**Почему:** Состояние cookies принадлежит экземпляру клиента.

## Основные выводы

- `Set-Cookie` и `Cookie` работают в противоположных направлениях.
- Cookie остаётся клиентскими недоверенными данными.
- Атрибуты задают область и политику доставки.
- HttpOnly не отменяет автоматическую отправку браузером.
- Для session будет использоваться непрозрачный token.
- Login и logout должны использовать одинаковые настройки cookie.
- Диагностика требует последовательности минимум из двух запросов.
