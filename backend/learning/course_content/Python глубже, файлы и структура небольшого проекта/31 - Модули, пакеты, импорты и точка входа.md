## 1. Когда одного файла мало

Один файл похож на рабочий стол. Пока на нём несколько инструментов, всё видно. Когда появляются меню, сервисы, валидаторы, исключения и хранение, связанные функции нужно разложить по подписанным ящикам-модулям.

### Модуль

Обычный файл .py с понятной ответственностью.

### Пакет

Папка связанных модулей с явной границей.

### Критерий разделения

Новый файл создаётся ради отдельной области ответственности, а не ради количества папок.

### До разделения

## 2. Модуль и формы импорта

Файл validators.py становится модулем, когда Python импортирует его. Другой файл получает доступ к функциям без копирования реализации.

### import module

Сохраняет явный источник в записи validators.normalize_title.

### from import

Удобен для небольшого числа часто используемых имён.

### as

Создаёт понятный псевдоним при конфликте или длинном названии.

### Validators.py

```python
def normalize_title(title):
    return title.strip()

def validate_priority(priority):
    if not 1 <= priority <= 5:
        raise ValueError("priority должен быть от 1 до 5")
    return priority
```

### Main.py

```python
from validators import normalize_title, validate_priority

title = normalize_title("  SQL  ")
priority = validate_priority(4)
```

> Избегайте from module import *: источник имён и возможные конфликты становятся неявными.

## 3. Что выполняет import

При первом импорте Python находит файл, создаёт объект модуля и выполняет верхнеуровневые инструкции. Поэтому print и input вне функций могут неожиданно сработать при обычном импорте.

### Определение функции

Python создаёт объект функции, но тело не запускается до вызова.

### Верхнеуровневый код

Присваивания и print выполняются во время импорта.

### Кэш

Повторный обычный import в том же процессе использует уже созданный модуль.

### Config.py

```python
print("config imported")
DATA_PATH = "data/tasks.json"
```

### Main.py

```python
import config
import config
print(config.DATA_PATH)
```

### Опасный services.py

```python
print("Меню")
command = input("Команда: ")

def create_task(...):
    ...
```

> Библиотечный модуль не должен запускать пользовательский сценарий как побочный эффект импорта.

## 4. Точка входа и __name__

Один файл можно запустить напрямую или импортировать. Переменная __name__ позволяет отличить режимы и не запускать меню во время тестового импорта.

### Прямой запуск

__name__ равен строке __main__.

### Импорт

__name__ содержит имя модуля, например app.main.

### Защита

if __name__ == "__main__" вызывает run только в режиме точки входа.

### Main.py

```python
def run():
    print("StudyHub started")

if __name__ == "__main__":
    run()
```

### Проверка импорта

```python
python -c "import app.main"
```

> Конструкция нужна файлу, который может запускаться напрямую. Её не требуется добавлять в каждый модуль.

## 5. Пакет и __init__.py

Связанные модули складываются в папку-пакет. Файл __init__.py делает границу явной и при необходимости задаёт небольшой публичный интерфейс.

### Пустой файл

Пустой __init__.py уже обозначает пакет и часто является лучшим началом.

### Публичные имена

Можно экспортировать несколько основных функций, но не нужно собирать всё подряд.

### Абсолютные импорты

Путь from app.services import create_task читается от корня пакета.

### Структура

```python
studyhub/
├── app/
│   ├── __init__.py
│   ├── main.py
│   ├── services.py
│   ├── validators.py
│   └── exceptions.py
└── README.md
```

### __init__.py

```python
from app.services import create_task
from app.exceptions import TaskNotFoundError

__all__ = ["create_task", "TaskNotFoundError"]
```

## 6. Направление зависимостей

Импорты удобно рисовать стрелками. Main собирает приложение и знает о сервисах. Сервисы знают о валидаторах и исключениях. Нижние уровни не должны импортировать интерфейс.

### Верхний уровень

main получает ввод и вызывает сервисы.

### Средний уровень

services реализует операции и получает состояние аргументом.

### Нижний уровень

validators и exceptions не зависят от main.

### Желательное направление

```python
main
  ↓
services
  ↓
validators + exceptions
```

### Плохая зависимость

```python
# services.py
from app.main import tasks

def add_task(task):
    tasks.append(task)
```

### Правильный контракт

```python
def add_task(tasks, task):
    tasks.append(task)
```

### Самопроверка раздела

Сформулируйте основную идею без подсказки, предскажите поведение первого примера и только затем запустите код.

> Передача состояния аргументом делает зависимость явной и предотвращает связь services с точкой запуска.

## 7. Циклический импорт

Цикл похож на двух сотрудников, каждый из которых ждёт завершения другого. Python начинает main, переходит в services, а services пытается получить имя из ещё не завершённого main.

### Частично созданный модуль

Импортируемый файл ещё не выполнил все присваивания.

### Архитектурная причина

Два слоя зависят друг от друга или общее правило лежит не там.

### Исправление

Разорвать обратную стрелку, передать состояние аргументом или перенести правило ниже.

### Цикл

```python
# services.py
from app.main import tasks

# main.py
from app.services import add_task
tasks = []
```

### Второй цикл

```python
# validators.py
from app.services import create_task

# services.py
from app.validators import validate_priority
```

### Самопроверка раздела

Сформулируйте основную идею без подсказки, предскажите поведение первого примера и только затем запустите код.

> Локальный import внутри функции иногда скрывает симптом, но не исправляет направление ответственности.

## Итоговый квиз

1. **Что является модулем?**
   - A. `файл .py`
   - B. `только Git-репозиторий`
   - C. `каждая функция`
   - **Ответ: A.** Модуль обычно соответствует Python-файлу.

2. **Зачем защита __name__?**
   - A. `не запускать интерфейс при импорте`
   - B. `создать venv`
   - C. `поймать ValueError`
   - **Ответ: A.** Прямой запуск отделяется от импорта.

3. **Что делает __init__.py?**
   - A. `обозначает пакет`
   - B. `обязательно запускает main`
   - C. `хранит JSON`
   - **Ответ: A.** Он задаёт границу пакета и может быть пустым.

4. **Какой импорт создаёт плохое направление?**
   - A. `services импортирует main`
   - B. `main импортирует services`
   - C. `services импортирует validators`
   - **Ответ: A.** Нижний уровень не должен зависеть от точки входа.

## Главное из занятия

- Модуль — файл с понятной ответственностью.
- Import выполняет верхнеуровневый код.
- Интерфейс не запускается при импорте.
- __name__ отделяет прямой запуск.
- Пакет объединяет модули.
- __init__.py может быть пустым.
- Абсолютные импорты показывают путь.
- Однонаправленные зависимости предотвращают циклы.
