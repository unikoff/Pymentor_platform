# 72. Настройки и переменные окружения

## Паспорт занятия

- Этап: 4 — FastAPI, SQLite и SQLAlchemy.
- Блок: 13 — FastAPI как цельное приложение.
- Длительность: 85–110 минут.
- Предварительные знания: Depends, Annotated, Pydantic v2, файлы, .gitignore и основные понятия окружения.
- Результат: ученик создаёт SettingsConfigDict, читает .env, различает конфигурацию и секреты, использует lru_cache и подготавливает DATABASE_URL к следующему блоку.

## Введение

Вынесем изменяемые значения из кода: начнём с environment variables, создадим Settings через pydantic-settings, подключим .env и .env.example и получим настройки через dependency.

## 1. Настройка — значение, которое меняется между окружениями

Название приложения, режим API и будущий DATABASE_URL могут отличаться на компьютере разработчика, в тестах и при запуске сервиса. Если они зашиты в код, каждое изменение требует редактирования файла.

### Code

Описывает стабильное поведение приложения.

### Configuration

Выбирает значение для конкретного окружения.

### Secret

Конфигурация, которую нельзя публиковать: ключ, пароль, token.

### Пример

SQLite URL в development и отдельный URL тестовой базы.

### Жёстко в коде

```python
APP_NAME = "StudyHub Database API"
API_MODE = "development"
DATABASE_URL = "sqlite:///./studyhub.db"
```

### Ожидаемая идея

```python
код приложения
+ значения окружения
→ готовая конфигурация запуска
```


### Активная проверка

Разделите примеры по смыслу.

- `DATABASE_URL` → environment setting
- `SECRET_KEY` → secret setting
- `MAX_PRIORITY = 5` → business constant
- `APP_NAME` → display/config setting

Настройки выбирают окружение, предметные константы определяют правило.

> Не каждое число является setting. Константа предметного правила, например MAX_PRIORITY=5, не обязана меняться между окружениями.

## 2. Environment variable приходит в Python как текст

Операционная система хранит пары имя–значение. Python может получить их через os.getenv, но преобразование типов и обязательность придётся контролировать вручную.

### Имя

Обычно используется uppercase: APP_NAME, API_MODE, DATABASE_URL.

### Строка

Внешнее значение поступает как str.

### Default

os.getenv может вернуть запасное значение.

### Ограничение

Ручные conversions быстро повторяются.

### Первый шаг

```python
import os

app_name = os.getenv(
    "APP_NAME",
    "StudyHub Database API",
)

api_mode = os.getenv(
    "API_MODE",
    "development",
)
```

### Powershell

```python
$env:APP_NAME = "StudyHub Local API"
$env:API_MODE = "test"
python -m uvicorn app.main:app --reload
```


### Активная проверка

**проверка environment**

```text
$ python -c "import os; print(os.getenv('API_MODE'))"
development
```

> os.getenv полезен для понимания механизма, но объект Settings даст типы, defaults и единое место конфигурации.

## 3. Pydantic Settings собирает и проверяет конфигурацию

Пакет pydantic-settings использует знакомую модель полей. При создании Settings он читает environment variables, преобразует типы и применяет defaults.

### Отдельный пакет

BaseSettings импортируется из pydantic_settings.

### Type hints

Поля получают ожидаемые Python-типы.

### Defaults

Локальный запуск возможен без полного набора variables.

### Required secret

Поле без default потребует внешнее значение.

### Установка

```python
python -m pip install pydantic-settings
```

### Config.py

```python
from pydantic_settings import BaseSettings


class Settings(BaseSettings):
    app_name: str = "StudyHub Database API"
    api_mode: str = "development"
    database_url: str = "sqlite:///./studyhub.db"
    debug: bool = False
```

### Создание

```python
settings = Settings()

print(settings.app_name)
print(type(settings.debug))
```


### Активная проверка

Выберите актуальный импорт.

```text
from ____ import BaseSettings
```

Варианты: `pydantic_settings`, `pydantic`, `fastapi`

**Ответ:** `pydantic_settings`. Settings вынесены в отдельный пакет.

> В новом коде BaseSettings не импортируется из pydantic. Для Pydantic v2 используется пакет pydantic-settings.

## 4. .env удобен для локального запуска

Файл .env хранит пары KEY=VALUE рядом с проектом. SettingsConfigDict сообщает модели, какой dotenv-файл прочитать и в какой кодировке.

### Локальная конфигурация

Не нужно каждый раз вводить несколько variables в терминале.

### Приоритет

Реальные environment variables могут переопределять значения dotenv.

### Кодировка

UTF-8 делает поведение предсказуемым.

### Не публикация

Рабочий .env обычно исключается из Git.

### Settingsconfigdict

```python
from pydantic_settings import (
    BaseSettings,
    SettingsConfigDict,
)


class Settings(BaseSettings):
    app_name: str = "StudyHub Database API"
    api_mode: str = "development"
    database_url: str = "sqlite:///./studyhub.db"

    model_config = SettingsConfigDict(
        env_file=".env",
        env_file_encoding="utf-8",
    )
```

### .env

```python
APP_NAME=StudyHub Local API
API_MODE=development
DATABASE_URL=sqlite:///./studyhub.db
```


### Активная проверка

Где хранить локальный DATABASE_URL?

**Вариант A — В router**

```text
DATABASE_URL = "..."
```

HTTP-модуль становится источником конфигурации.

**Вариант B — В .env + Settings**

```text
DATABASE_URL=sqlite:///./studyhub.db
```

Значение выбирается окружением и проверяется моделью.

**Разбор:** Router получает готовые settings через dependency.

> .env не является Python-файлом: кавычки, пробелы и синтаксис должны соответствовать dotenv-формату.

## 5. .env.example документирует обязательные имена

Рабочий .env не коммитится, но новый разработчик должен знать, какие variables требуются. Для этого создаётся безопасный .env.example без настоящих secrets.

### Шаблон

Содержит имена и безопасные демонстрационные значения.

### README

Объясняет копирование .env.example → .env.

### Без secrets

Настоящий SECRET_KEY или пароль не попадает в repository.

### Синхронизация

При добавлении новой setting обновляются model, example и README.

### .env.example

```python
APP_NAME=StudyHub Database API
API_MODE=development
DATABASE_URL=sqlite:///./studyhub.db
SECRET_KEY=replace-me-locally
```

### .gitignore

```python
.env
.venv/
__pycache__/
.pytest_cache/
*.pyc
```


### Активная проверка

```python
.env
SECRET_KEY=real-production-secret

# затем
git add .env
git commit -m "add config"
```

В чём риск?

1. Секрет попадает в историю Git
2. dotenv не поддерживает строки
3. FastAPI удалит файл

**Ответ:** 1. Даже удалённый следующим commit secret останется в истории и должен считаться раскрытым.

Исправление:

```python
.gitignore
.env

.env.example
SECRET_KEY=replace-me
```

> Само имя SECRET_KEY можно публиковать. Нельзя публиковать реальное значение, которое используется сервисом.

## 6. get_settings и lru_cache

Создание Settings может читать dotenv. Для одного процесса достаточно создать объект один раз и возвращать его через dependency. lru_cache хранит результат обычной Python-функции между requests.

### get_settings

Единая функция-provider для FastAPI.

### lru_cache

После первого вызова возвращает тот же объект для одинаковых аргументов.

### Dependency

Endpoints объявляют SettingsDep.

### Тест

Provider можно override, не изменяя рабочий endpoint.

### Provider

```python
from functools import lru_cache
from typing import Annotated

from fastapi import Depends


@lru_cache
def get_settings() -> Settings:
    return Settings()


SettingsDep = Annotated[
    Settings,
    Depends(get_settings),
]
```

### Endpoint

```python
@router.get("/info")
def get_info(settings: SettingsDep):
    return {
        "app_name": settings.app_name,
        "api_mode": settings.api_mode,
    }
```


### Активная проверка

Почему не создать Settings() прямо в каждом endpoint?

**Вариант A — В каждом handler**

```text
settings = Settings()
```

Повторное чтение конфигурации и жёсткое создание внутри route.

**Вариант B — Provider + cache**

```text
settings: SettingsDep
```

Получение централизовано и заменяется в тестах.

**Разбор:** FastAPI dependency и lru_cache дают ясную точку получения settings.

> lru_cache и request dependency cache — разные уровни. Первый живёт в Python-процессе, второй действует при решении одного request graph.

## 7. Development и test используют разные значения

Тесты не должны случайно использовать рабочий database_url или режим. Dependency override позволяет передать TestSettings либо отдельный Settings с тестовыми значениями.

### Development

Локальный SQLite-файл и debug-настройки.

### Test

Отдельный временный URL и api_mode=test.

### Override

app.dependency_overrides связывает рабочий provider с тестовым.

### Очистка

Override удаляется после теста, чтобы сценарии не влияли друг на друга.

### Test provider

```python
def get_test_settings() -> Settings:
    return Settings(
        api_mode="test",
        database_url="sqlite:///./test.db",
    )

app.dependency_overrides[get_settings] = get_test_settings
```

### Cleanup

```python
app.dependency_overrides.clear()
```


### Активная проверка

```text
test starts
→ override get_settings
→ request
→ endpoint receives test settings
→ assertions
→ clear overrides
```

1. Строка 1: Тест подготавливает окружение. — mode=test
2. Строка 2: Рабочий provider заменяется. — override=active
3. Строка 3: TestClient отправляет request. — request=GET /info
4. Строка 4: Endpoint получает test Settings. — database_url=test.db
5. Строка 5: Проверяется response. — status=200
6. Строка 6: Override очищается. — isolation=restored

> В блоке 14 DATABASE_URL начнёт реально управлять SQLAlchemy engine. Сейчас важно подготовить безопасную конфигурационную границу.

## Итоговый квиз

1. **Откуда импортируется BaseSettings в Pydantic v2?**
   - A. `pydantic_settings`
   - B. `fastapi`
   - C. `sqlite3`
   - **Ответ: A.** Settings находятся в отдельном пакете.

2. **Зачем .env.example?**
   - A. `документировать имена без настоящих secrets`
   - B. `хранить production secret`
   - C. `заменить .gitignore`
   - **Ответ: A.** Новый разработчик видит шаблон конфигурации.

3. **Что делает lru_cache для get_settings?**
   - A. `переиспользует созданный объект в процессе`
   - B. `кэширует HTTP body`
   - C. `создаёт cookie`
   - **Ответ: A.** Settings не читается заново на каждый request.

4. **Что нельзя возвращать из /info?**
   - A. `SECRET_KEY`
   - B. `app_name`
   - C. `api_mode`
   - **Ответ: A.** Секреты не становятся публичным response.

## Главное из занятия

- Configuration отделяется от стабильного кода.
- Environment variables поступают как текст.
- Pydantic Settings преобразует и проверяет значения.
- SettingsConfigDict подключает .env.
- .env обычно не коммитится.
- .env.example документирует безопасный шаблон.
- get_settings используется как dependency provider.
- lru_cache и dependency override упрощают запуск и тесты.

Создайте config.py, .env и .env.example, подключите SettingsDep к /info и проверьте отдельные development/test значения.

- Обычный успешный request проходит.
- Один ожидаемый error response проверен.
- Путь значений объясняется своими словами.
- Код не использует будущую тему без подготовки.
- Postman или Swagger показывает контракт.
- Сделан отдельный Git-коммит.
