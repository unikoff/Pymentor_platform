## 1. Зачем отделять чтение списка от чтения одной записи

После первой записи через FastAPI проект умеет сохранять задачу, но клиенту нужно получить данные обратно. Список и один объект похожи только внешне: у них разные запросы, результаты и сценарии отсутствия.

### Маршрут

1. Сформулировать вопрос к данным.
2. Построить SQLAlchemy-выражение.
3. Выполнить его через `Session`.
4. Проверить пустой результат или ошибку.
5. Вернуть стабильный HTTP-ответ.

## 2. select строит объект запроса

Функция select не обращается к базе немедленно. Она создаёт описание будущего SQL-запроса, которое Session выполнит позже.

```python
statement = select(TaskModel).order_by(TaskModel.id)
result = db.execute(statement)
tasks = result.scalars().all()
return tasks
```

## 3. Session.execute выполняет запрос

Сессия отправляет выражение в базу и получает Result. Result содержит строки результата, а не готовый список ORM-объектов.

```python
statement = select(TaskModel).where(TaskModel.id == task_id)
task = db.execute(statement).scalar_one_or_none()

if task is None:
    raise HTTPException(status_code=404, detail="Task not found")

return task
```

## 4. scalars и all возвращают модели

Метод scalars извлекает из каждой строки первый ORM-объект, а all собирает все найденные объекты в список.

```python
statement = select(TaskModel).order_by(TaskModel.id)
result = db.execute(statement)
tasks = result.scalars().all()
return tasks
```

## 5. Один объект через where

Для поиска по id к select добавляется where. Выражение TaskModel.id == task_id превращается в SQL-условие, а значение передаётся безопасным параметром.

```python
statement = select(TaskModel).where(TaskModel.id == task_id)
task = db.execute(statement).scalar_one_or_none()

if task is None:
    raise HTTPException(status_code=404, detail="Task not found")

return task
```

## 6. first и one_or_none

first возвращает первый объект или None. one_or_none требует не больше одной строки и лучше выражает ожидание уникального primary key.

```python
task = db.execute(
    select(TaskModel).where(TaskModel.id == task_id)
).scalar_one_or_none()

if task is None:
    raise HTTPException(404, "Task not found")

return task
```

## 7. 404 как часть HTTP-контракта

Отсутствие строки в базе является ожидаемым сценарием. Endpoint преобразует None в HTTPException со статусом 404, а не обращается к полям несуществующего объекта.


## Итоговый квиз

### Вопрос 1
Что делает select(TaskModel)?

- A. создаёт описание запроса
- B. сразу возвращает список
- C. создаёт таблицу

**Ответ:** A. создаёт описание запроса
**Почему:** select строит SQL-выражение, но не выполняет его.

### Вопрос 2
Зачем нужен scalars()?

- A. извлечь ORM-объекты
- B. сохранить изменения
- C. закрыть session

**Ответ:** A. извлечь ORM-объекты
**Почему:** Result может содержать строки из нескольких колонок; scalars берёт первый элемент каждой строки.

### Вопрос 3
Что возвращает scalar_one_or_none()?

- A. один объект или None
- B. только список
- C. всегда исключение

**Ответ:** A. один объект или None
**Почему:** Метод выражает ожидание нуля или одной строки.

### Вопрос 4
Когда endpoint возвращает 404?

- A. запись не найдена
- B. список пуст
- C. commit успешен

**Ответ:** A. запись не найдена
**Почему:** Для конкретного id отсутствие ресурса превращается в 404.

## Основные выводы

- SQLAlchemy разделяет описание запроса и его выполнение.
- `Session` является рабочим контекстом запроса и не должна быть глобальной.
- Пустой результат, конфликт и ошибка ограничения являются частью контракта API.
- Клиент получает предметный HTTP-ответ, а внутренние детали остаются в логах.
- Каждая операция проверяется на успешном и граничном сценарии.

## Критерии готовности

- Код запускается синхронно через обычные `def` endpoint-функции.
- Используется одна Session на запрос через `get_db`.
- Нет ручной склейки SQL из пользовательских строк.
- Отсутствие ресурса и конфликты имеют понятные статусы.
- После ошибки транзакции Session не остаётся в сломанном состоянии.
- Ученик может объяснить маршрут данных своими словами.
