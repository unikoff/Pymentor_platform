## 2. where получает SQL-выражение

Сравнение TaskModel.is_done == is_done не вычисляется как обычный bool. SQLAlchemy строит выражение для будущего WHERE.

```python
statement = select(TaskModel)

if is_done is not None:
    statement = statement.where(TaskModel.is_done == is_done)

if priority_min is not None:
    statement = statement.where(TaskModel.priority >= priority_min)

if search:
    statement = statement.where(TaskModel.title.ilike(f"%{search}%"))
```

## 3. Несколько условий соединяются

Каждый вызов where добавляет ещё одно условие AND. Запрос удобно собирать по шагам, сохраняя его в той же переменной.

```python
@router.get("/tasks")
def list_tasks(
    is_done: bool | None = None,
    priority_min: int | None = Query(None, ge=1, le=5),
    search: str | None = None,
    db: Session = Depends(get_db),
):
    ...
```

## 4. Необязательные query-параметры

Значение None означает, что клиент фильтр не задавал. Важно отличать None от False, потому что False является полноценным условием.

```python
statement = select(TaskModel)

if is_done is not None:
    statement = statement.where(TaskModel.is_done == is_done)

if priority_min is not None:
    statement = statement.where(TaskModel.priority >= priority_min)

if search:
    statement = statement.where(TaskModel.title.ilike(f"%{search}%"))
```

## Итоговый квиз

### Вопрос 1
Почему нельзя писать if is_done?

- A. False будет пропущен
- B. True станет строкой
- C. Session закроется

**Ответ:** A. False будет пропущен
**Почему:** False является нужным фильтром, а не отсутствием значения.

### Вопрос 2
Что делает второй where?

- A. добавляет условие AND
- B. заменяет таблицу
- C. выполняет commit

**Ответ:** A. добавляет условие AND
**Почему:** Условия накапливаются в выражении.

### Вопрос 3
Где лучше фильтровать большой набор?

- A. в SQL-запросе
- B. после all в Python
- C. в README

**Ответ:** A. в SQL-запросе
**Почему:** База возвращает меньше строк и выполняет отбор ближе к данным.

### Вопрос 4
Почему не склеиваем SQL вручную?

- A. параметры безопаснее и понятнее
- B. SQLAlchemy запрещает строки
- C. иначе нельзя использовать GET

**Ответ:** A. параметры безопаснее и понятнее
**Почему:** Параметризованный запрос отделяет код от значений.

## Основные выводы

- SQLAlchemy разделяет описание запроса и его выполнение.
- `Session` является рабочим контекстом запроса и не должна быть глобальной.
- Пустой результат, конфликт и ошибка ограничения являются частью контракта API.
- Клиент получает предметный HTTP-ответ, а внутренние детали остаются в логах.
- Каждая операция проверяется на успешном и граничном сценарии.

## Критерии готовности

- Код запускается синхронно через обычные `def` endpoint-функции.
- Используется одна Session на запрос через `get_db`.
- Нет ручной склейки SQL из пользовательских строк.
- Отсутствие ресурса и конфликты имеют понятные статусы.
- После ошибки транзакции Session не остаётся в сломанном состоянии.
- Ученик может объяснить маршрут данных своими словами.
