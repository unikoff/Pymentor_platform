## 2. order_by задаёт порядок

Без order_by база не обещает устойчивый порядок. Для страниц это критично: одинаковый запрос должен возвращать ожидаемую последовательность.

```python
offset = (page - 1) * page_size
statement = (
    select(TaskModel)
    .order_by(TaskModel.priority.desc(), TaskModel.id.asc())
    .limit(page_size)
    .offset(offset)
)
```

## 4. offset пропускает начало

Для страницы page вычисляем offset = (page - 1) * page_size. Первая страница пропускает ноль строк.

## Итоговый квиз

### Вопрос 1
Зачем нужен order_by при пагинации?

- A. обеспечить предсказуемый порядок
- B. создать таблицу
- C. сделать commit

**Ответ:** A. обеспечить предсказуемый порядок
**Почему:** Страницы должны строиться поверх устойчивой последовательности.

### Вопрос 2
Как вычислить offset для page=3, size=10?

- A. 20
- B. 30
- C. 3

**Ответ:** A. 20
**Почему:** Пропускаются две полные предыдущие страницы.

### Вопрос 3
Почему page_size ограничивают сверху?

- A. защитить API от огромного ответа
- B. SQL запрещает большие числа
- C. иначе id исчезнет

**Ответ:** A. защитить API от огромного ответа
**Почему:** Сервер сохраняет контроль над объёмом работы.

### Вопрос 4
Зачем второй ключ id?

- A. стабилизировать равные значения
- B. ускорить commit
- C. заменить primary key

**Ответ:** A. стабилизировать равные значения
**Почему:** При одинаковом первом ключе id задаёт однозначный порядок.

## Основные выводы

- SQLAlchemy разделяет описание запроса и его выполнение.
- `Session` является рабочим контекстом запроса и не должна быть глобальной.
- Пустой результат, конфликт и ошибка ограничения являются частью контракта API.
- Клиент получает предметный HTTP-ответ, а внутренние детали остаются в логах.
- Каждая операция проверяется на успешном и граничном сценарии.

## Критерии готовности

- Код запускается синхронно через обычные `def` endpoint-функции.
- Используется одна Session на запрос через `get_db`.
- Нет ручной склейки SQL из пользовательских строк.
- Отсутствие ресурса и конфликты имеют понятные статусы.
- После ошибки транзакции Session не остаётся в сломанном состоянии.
- Ученик может объяснить маршрут данных своими словами.
