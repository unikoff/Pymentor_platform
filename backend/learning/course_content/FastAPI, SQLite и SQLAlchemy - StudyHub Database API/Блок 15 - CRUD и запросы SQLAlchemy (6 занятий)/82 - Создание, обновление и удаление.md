# 82. Создание, обновление и удаление

## Паспорт занятия

- **Блок:** 15 · CRUD и запросы SQLAlchemy
- **Проект:** StudyHub Database API
- **Формат:** теория → чтение кода → прогноз → запуск → изменение → диагностика → объяснение
- **Продолжительность:** 60–90 минут

## Результат

Соберём полный жизненный цикл задачи: создадим ORM-объект, изменим только переданные поля, удалим запись и будем фиксировать каждую операцию через commit.

## Состояние проекта до занятия

FastAPI уже подключён к SQLite через синхронный SQLAlchemy. Есть `engine`, фабрика `Session`, dependency `get_db`, ORM-модель `TaskModel` и endpoint создания задачи. В этом занятии развивается тот же проект без смены архитектурного контекста.

## 1. CRUD как четыре разных контракта

Create, Read, Update и Delete работают с одной таблицей, но получают разные данные и обещают разные результаты. Не стоит прятать их в одну универсальную функцию.

### Маршрут

1. Сформулировать вопрос к данным.
2. Построить SQLAlchemy-выражение.
3. Выполнить его через `Session`.
4. Проверить пустой результат или ошибку.
5. Вернуть стабильный HTTP-ответ.

## 3. add, commit и refresh

add помещает объект в текущую единицу работы, commit фиксирует транзакцию, refresh повторно читает строку и получает сгенерированный id.

```python
changes = payload.model_dump(exclude_unset=True)
for field, value in changes.items():
    setattr(task, field, value)

db.commit()
db.refresh(task)
return task
```

## 5. model_dump exclude_unset

Pydantic v2 умеет вернуть только поля, которые клиент действительно прислал. Это защищает старые значения от случайной замены на None.

```python
changes = payload.model_dump(exclude_unset=True)
for field, value in changes.items():
    setattr(task, field, value)

db.commit()
db.refresh(task)
return task
```

## 7. Типичные ошибки CRUD

Нельзя забывать 404, commit или проверку пустого PATCH. После удаления нельзя продолжать использовать объект как существующий ресурс.

```python

    setattr(task, field, value)
# commit отсутствует
```

## Итоговый квиз

### Вопрос 1
Что фиксирует изменение в базе?

- A. commit
- B. add
- C. refresh

**Ответ:** A. commit
**Почему:** Commit завершает транзакцию.

### Вопрос 2
Зачем exclude_unset=True?

- A. оставить только присланные поля
- B. удалить все None
- C. закрыть session

**Ответ:** A. оставить только присланные поля
**Почему:** PATCH не должен затирать поля, которых не было в запросе.

### Вопрос 3
Что делает db.delete(task)?

- A. помечает объект на удаление
- B. сразу удаляет файл базы
- C. возвращает 404

**Ответ:** A. помечает объект на удаление
**Почему:** Удаление фиксируется только после commit.

### Вопрос 4
Зачем refresh после создания?

- A. получить значения из базы
- B. отменить транзакцию
- C. создать схему

**Ответ:** A. получить значения из базы
**Почему:** База может сгенерировать id и defaults.

## Основные выводы

- SQLAlchemy разделяет описание запроса и его выполнение.
- `Session` является рабочим контекстом запроса и не должна быть глобальной.
- Пустой результат, конфликт и ошибка ограничения являются частью контракта API.
- Клиент получает предметный HTTP-ответ, а внутренние детали остаются в логах.
- Каждая операция проверяется на успешном и граничном сценарии.

## Критерии готовности

- Код запускается синхронно через обычные `def` endpoint-функции.
- Используется одна Session на запрос через `get_db`.
- Нет ручной склейки SQL из пользовательских строк.
- Отсутствие ресурса и конфликты имеют понятные статусы.
- После ошибки транзакции Session не остаётся в сломанном состоянии.
- Ученик может объяснить маршрут данных своими словами.
