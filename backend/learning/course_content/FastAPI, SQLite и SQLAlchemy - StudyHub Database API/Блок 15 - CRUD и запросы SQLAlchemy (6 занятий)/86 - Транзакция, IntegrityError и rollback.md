# 86. Транзакция, IntegrityError и rollback

## Паспорт занятия

- **Блок:** 15 · CRUD и запросы SQLAlchemy
- **Проект:** StudyHub Database API
- **Формат:** теория → чтение кода → прогноз → запуск → изменение → диагностика → объяснение
- **Продолжительность:** 60–90 минут

## Результат

Разберём отказоустойчивую запись: commit может завершиться IntegrityError, после чего Session нужно явно откатить, а клиенту вернуть безопасный 409 вместо внутренней ошибки базы.

## Состояние проекта до занятия

FastAPI уже подключён к SQLite через синхронный SQLAlchemy. Есть `engine`, фабрика `Session`, dependency `get_db`, ORM-модель `TaskModel` и endpoint создания задачи. В этом занятии развивается тот же проект без смены архитектурного контекста.

## 1. Commit — граница операции

До commit изменения существуют в текущей Session. Commit просит базу проверить ограничения и зафиксировать все изменения как одну транзакцию.

### Маршрут

1. Сформулировать вопрос к данным.
2. Построить SQLAlchemy-выражение.
3. Выполнить его через `Session`.
4. Проверить пустой результат или ошибку.
5. Вернуть стабильный HTTP-ответ.

## 2. Транзакция: всё или ничего

Несколько изменений внутри одной транзакции либо фиксируются вместе, либо откатываются. Это защищает данные от частично выполненной операции.

```python
try:
    db.add(category)
    db.commit()
    db.refresh(category)
except IntegrityError:
    db.rollback()
    raise HTTPException(
        status_code=409,
        detail="Category already exists",
    )
```

## 3. Где возникает IntegrityError

Ошибка уникальности появляется не обязательно на add. Обычно база проверяет ограничение во время flush или commit.

```python
category = CategoryModel(name=payload.name)

try:
    db.add(category)
    db.commit()
except IntegrityError as error:
    db.rollback()
    logger.warning("category conflict", exc_info=error)
    raise HTTPException(409, "Category already exists")
```

## 4. Session после неудачного commit

После ошибки транзакция помечена как failed. Любой следующий запрос без rollback приводит к PendingRollbackError.

```python
try:
    db.add(category)
    db.commit()
    db.refresh(category)

    db.rollback()
```

## 5. try, except и rollback

Рискованный commit помещается в try. В except IntegrityError сначала вызывается rollback, затем формируется контролируемый HTTP-ответ.

```python
category = CategoryModel(name=payload.name)

try:
    db.add(category)
    db.commit()
except IntegrityError as error:
    db.rollback()
    logger.warning("category conflict", exc_info=error)
    raise HTTPException(409, "Category already exists")
```

## 7. Повторная работа Session

После rollback сессия снова готова к запросам. Это нужно проверить отдельным сценарием: дубликат, затем чтение списка или создание другого объекта.

```python

    db.commit()
except IntegrityError:
    raise HTTPException(409, "Conflict")
```

## Итоговый квиз

### Вопрос 1
Когда база обычно проверяет UNIQUE?

- A. во время flush или commit
- B. при создании класса
- C. при импорте FastAPI

**Ответ:** A. во время flush или commit
**Почему:** add только прикрепляет объект к Session.

### Вопрос 2
Что обязательно после IntegrityError?

- A. rollback
- B. refresh
- C. create_all

**Ответ:** A. rollback
**Почему:** Сессия остаётся в failed-состоянии до отката.

### Вопрос 3
Что гарантирует транзакция?

- A. все изменения вместе или ни одного
- B. автоматический 404
- C. сортировку по id

**Ответ:** A. все изменения вместе или ни одного
**Почему:** Транзакция защищает атомарность операции.

### Вопрос 4
Что отправлять клиенту?

- A. стабильное предметное сообщение
- B. полный traceback
- C. сырой SQL

**Ответ:** A. стабильное предметное сообщение
**Почему:** Внутренние детали остаются в логах.

## Основные выводы

- SQLAlchemy разделяет описание запроса и его выполнение.
- `Session` является рабочим контекстом запроса и не должна быть глобальной.
- Пустой результат, конфликт и ошибка ограничения являются частью контракта API.
- Клиент получает предметный HTTP-ответ, а внутренние детали остаются в логах.
- Каждая операция проверяется на успешном и граничном сценарии.

## Критерии готовности

- Код запускается синхронно через обычные `def` endpoint-функции.
- Используется одна Session на запрос через `get_db`.
- Нет ручной склейки SQL из пользовательских строк.
- Отсутствие ресурса и конфликты имеют понятные статусы.
- После ошибки транзакции Session не остаётся в сломанном состоянии.
- Ученик может объяснить маршрут данных своими словами.
