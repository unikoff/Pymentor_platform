## 2. COUNT считает строки в базе

func.count строит агрегатный SQL-запрос. scalar_one возвращает одно числовое значение.

```python
total = db.scalar(select(func.count(TaskModel.id)))
completed = db.scalar(
    select(func.count(TaskModel.id)).where(TaskModel.is_done.is_(True))
)
```

## 3. EXISTS отвечает булевым значением

exists позволяет базе остановиться после первого совпадения. Это точнее, чем загружать объект, если его поля не нужны.

```python
statement = select(
    exists().where(CategoryModel.name == name)
)
is_taken = db.scalar(statement)
```

## 4. Python-проверка и ограничение базы

Проверка SELECT перед INSERT улучшает сообщение, но не гарантирует уникальность при двух одновременных запросах. Гарантию даёт UNIQUE.

```python
total = db.scalar(select(func.count(TaskModel.id)))
completed = db.scalar(
    select(func.count(TaskModel.id)).where(TaskModel.is_done.is_(True))
)
```

## 5. unique=True в ORM-модели

Ограничение становится частью схемы таблицы. Для существующей базы оно добавляется миграцией, а не простым изменением класса.

```python
statement = select(
    exists().where(CategoryModel.name == name)
```

## 6. 409 Conflict

Когда клиент создаёт ресурс с уже занятым уникальным значением, ответ 409 точнее, чем 400 или 500.

```python
is_taken = db.scalar(
    select(exists().where(CategoryModel.name == name))
)
if is_taken:
    raise HTTPException(409, "Category already exists")
```

## Итоговый квиз

### Вопрос 1
Что возвращает COUNT?

- A. число строк
- B. список моделей
- C. новую таблицу

**Ответ:** A. число строк
**Почему:** COUNT является агрегатом.

### Вопрос 2
Когда полезен EXISTS?

- A. нужен только факт совпадения
- B. нужно обновить все поля
- C. нужно закрыть session

**Ответ:** A. нужен только факт совпадения
**Почему:** EXISTS выражает булевый вопрос.

### Вопрос 3
Что действительно гарантирует уникальность?

- A. UNIQUE в базе
- B. предварительный if
- C. описание в README

**Ответ:** A. UNIQUE в базе
**Почему:** Только ограничение базы защищает при конкурирующих операциях.

### Вопрос 4
Какой статус подходит для дубликата?

- A. 409
- B. 201
- C. 404

**Ответ:** A. 409
**Почему:** Конфликт с текущим состоянием ресурса выражается через 409.

## Основные выводы

- SQLAlchemy разделяет описание запроса и его выполнение.
- `Session` является рабочим контекстом запроса и не должна быть глобальной.
- Пустой результат, конфликт и ошибка ограничения являются частью контракта API.
- Клиент получает предметный HTTP-ответ, а внутренние детали остаются в логах.
- Каждая операция проверяется на успешном и граничном сценарии.

## Критерии готовности

- Код запускается синхронно через обычные `def` endpoint-функции.
- Используется одна Session на запрос через `get_db`.
- Нет ручной склейки SQL из пользовательских строк.
- Отсутствие ресурса и конфликты имеют понятные статусы.
- После ошибки транзакции Session не остаётся в сломанном состоянии.
- Ученик может объяснить маршрут данных своими словами.
