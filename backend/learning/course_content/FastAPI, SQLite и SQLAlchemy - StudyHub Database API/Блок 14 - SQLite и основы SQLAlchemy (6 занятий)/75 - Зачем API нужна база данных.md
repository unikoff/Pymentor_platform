## 1. Почему старый подход больше не решает задачу

В Planner API список `tasks` находился внутри процесса Python. Это был правильный промежуточный этап: ученик мог сосредоточиться на HTTP и Pydantic. Но у решения есть жёсткая граница — после остановки процесса оперативная память освобождается.

Профессиональная формулировка проблемы: текущее хранилище не обеспечивает **durability**, то есть сохранность зафиксированного состояния после завершения процесса.

```text
POST /tasks
    ↓
tasks.append(...)
    ↓
задача существует в RAM
    ↓
остановка процесса
    ↓
состояние потеряно
```

## 2. Главная модель постоянного хранения

Постоянное хранилище живёт дольше одного запроса и одного процесса. Endpoint принимает HTTP-запрос, но не обязан сам знать, как байты записываются на диск. Это разделение позволяет позже заменить SQLite на PostgreSQL без переписывания публичного HTTP-контракта.

```text
client
  ↓ JSON
FastAPI endpoint
  ↓ validated data
storage layer
  ↓ durable write
SQLite file
```

## 3. Таблица, строка, колонка и схема

Реляционная база организует данные в таблицах. Таблица `tasks` хранит однотипные записи задач. Каждая строка представляет одну задачу, а каждая колонка имеет устойчивый смысл: `title`, `priority`, `is_done`.

Слово **schema** в контексте базы означает структуру таблиц и ограничений. Это не то же самое, что Pydantic schema, хотя обе конструкции описывают форму данных на разных границах.

```text
tasks
┌────┬────────────────┬──────────┬─────────┐

├────┼────────────────┼──────────┼─────────┤

```

## 4. Primary key и идентичность строки

`primary key` — колонка или набор колонок, которые однозначно определяют строку. В StudyHub отдельный числовой `id` надёжнее названия: названия могут повторяться и изменяться.

Клиент не должен выбирать `id` новой задачи. Идентификатор назначает система хранения, чтобы избежать конфликтов и сохранить внутреннюю целостность.

```python
tasks = [
    {"id": 1, "title": "Python"},
    {"id": 2, "title": "Python"},
]

task_id = 2
task = next(item for item in tasks if item["id"] == task_id)
print(task)
```

## 5. SQLite как встроенная реляционная СУБД

SQLite работает как библиотека внутри приложения и обычно хранит базу в одном файле. Отдельный серверный процесс, порт и пользователь базы для локального запуска не требуются.

Фраза «один файл» не означает «обычный файл с JSON». SQLite выполняет SQL, поддерживает таблицы, ограничения и транзакции. Корректный термин — **embedded relational database management system**.

```text
FastAPI process
├── application code
├── SQLAlchemy
├── sqlite3 driver
└── SQLite engine → studyhub.db
```

## 6. Преимущества и эксплуатационные границы

SQLite удобна для обучения, локальной разработки, автотестов и небольших приложений. Она снижает инфраструктурный порог: не нужно устанавливать и администрировать отдельный сервер.

При этом SQLite имеет иной профиль конкурентного доступа, чем PostgreSQL. Большое число одновременных пишущих операций и несколько серверов приложения с общей базой — повод рассматривать серверную СУБД.

## 7. Плавный маршрут миграции StudyHub

Мы не переписываем весь CRUD одновременно. В этом блоке вводятся только инфраструктура подключения, ORM-модель, физическая таблица, Session и одна операция создания.

`SELECT`, фильтры, обновление, удаление, связи и Alembic намеренно отложены. Такая последовательность уменьшает когнитивную нагрузку и позволяет диагностировать одну границу за раз.

```text
75 проблема хранения
 ↓
76 Engine и URL
 ↓
77 ORM-модель
 ↓
78 физическая таблица
 ↓
79 Session и транзакция
 ↓
80 POST /tasks → SQLite
```
