## Введение

Главная модель этапа: HTTP request получает управляемую Session через dependency, SQLAlchemy строит и выполняет statement, database transaction меняет rows, ORM-объекты превращаются в response schema, а Alembic хранит историю schema.

## 1. Почему in-memory storage перестаёт подходить

Список Python был правильным учебным хранилищем для первого API: он позволял увидеть HTTP-контракт без базы. Но после появления реального проекта его ограничения становятся частью поведения и требуют отдельного решения.

### Данные исчезают

**Метка:** `restart`.

Новый процесс создаёт новый пустой list.

```text
tasks = []
```

### Состояние расходится

**Метка:** `workers`.

Два процесса получили бы два независимых списка.

```text
worker A ≠ worker B
```

### Ограничения держит код

**Метка:** `rules`.

Уникальность и links проверяются вручную в каждом пути.

```text
if duplicate: ...
```

### Сравнение: Какое требование уже нельзя надёжно закрыть одним list?

**Вариант A — Показать задачи**

```text
return tasks
```

Один процесс легко возвращает текущие элементы.

**Вариант B — Сохранить и согласовать данные**

```text
restart + multiple requests + constraints
```

Нужно устойчивое состояние и правила на уровне database.

**Разбор:** База появляется тогда, когда данные должны переживать процесс и сохранять целостность.

### Новая ответственность

```text
API contract отвечает: как клиент общается
Database отвечает: как данные хранятся и проверяются
SQLAlchemy отвечает: как Python-код формулирует работу с database
Alembic отвечает: как schema изменяется во времени
```

> База не заменяет validation Pydantic. Schema request проверяет вход API, а database constraints защищают сохранённое состояние.

| Понятие | Смысл |
| --- | --- |
| `restart` | После остановки API rows должны остаться. |
| `concurrent clients` | Несколько requests должны видеть согласованное состояние. |
| `unique value` | Дубликат должен отклоняться независимо от endpoint. |
| `relation` | Task не должна ссылаться на отсутствующую category. |
| `query` | Filters и pagination должны выполняться без загрузки всего набора в Python. |

### Требование → механизм

```text
persistence → table rows
identity → primary key
valid relation → foreign key
unique title → unique constraint
atomic change → transaction
fast lookup → index
reproducible schema → migration
```

## 2. SQLite, SQLAlchemy и Alembic — разные инструменты

Три названия часто сливаются в одно слово «база». Полезно разделить файл database, Python toolkit и историю schema.

### Database engine

**Метка:** `SQLite`.

Хранит tables, rows, indexes и constraints в локальном файле.

```text
studyhub.db
```

### Python toolkit и ORM

**Метка:** `SQLAlchemy`.

Создаёт engine, Session, models и SQL expressions.

```text
select(Task)
```

### Migration tool

**Метка:** `Alembic`.

Хранит revisions с upgrade и downgrade.

```text
alembic upgrade head
```

### Сопоставление: Соедините проблему и инструмент.

- `сохранить rows после restart` → SQLite
- `построить SELECT из Python` → SQLAlchemy
- `создать Session для transaction` → SQLAlchemy
- `повторить изменение schema` → Alembic
- `посмотреть фактические tables` → SQLite viewer

Инструменты работают вместе, но не взаимозаменяют друг друга.

### Слои

```text
FastAPI
  ↓ dependency
SQLAlchemy Session
  ↓ SQL
SQLite database file

Alembic ── управляет изменениями schema SQLite
```

### Правда или ложь

Alembic хранит пользовательские задачи вместо SQLite.

**Ответ:** ложь. Alembic хранит migration scripts и version state, а rows находятся в database tables.

> На будущем PostgreSQL SQLAlchemy и Alembic останутся, а database URL и особенности engine изменятся. Поэтому роли изучаются отдельно.

### Одна операция на трёх уровнях

```text
SQLite: INSERT INTO tasks (...) VALUES (...)
SQLAlchemy: session.add(task); session.commit()
FastAPI: POST /tasks → 201 Created
```

### Вопрос для объяснения

**Вопрос:** Почему эти три строки нельзя считать взаимозаменяемыми?

Подсказка: Определите, кто является читателем каждой формы.

**Ориентир ответа:** Они описывают одну операцию на разных границах: SQL database, Python persistence layer и HTTP API contract.

### SQLite отвечает на SQL

Создаёт tables, хранит rows и применяет constraints.

### SQLAlchemy строит работу Python

Engine и Session отправляют SQL, ORM maps rows в objects.

### Alembic меняет schema

Migration scripts переводят database между revisions.

### FastAPI связывает request

Dependency выдаёт Session, endpoint выбирает operation и response.

### одна и та же база с разных сторон

```text

```

## 4. Engine, connection, Session и transaction

SQLAlchemy разделяет несколько уровней. Engine знает, как подключаться; connection является конкретным каналом; Session организует работу ORM; transaction определяет атомарную группу изменений.

1. **Engine.** Создаётся один раз и хранит database URL и pool configuration.
2. **Connection.** Берётся engine, когда нужно фактически выполнить SQL.
3. **Session.** Отслеживает ORM-объекты и выполняет statements через connection.
4. **Transaction.** Объединяет изменения до commit или rollback.
5. **Close.** Освобождает ресурсы после request.

В endpoint обычно передаётся Session, а engine и connection остаются внутренней инфраструктурой database layer.

### Жизненный цикл записи

```text
session = SessionLocal()
try:
    task = Task(title="SQLAlchemy", priority=4)
    session.add(task)
    session.commit()
    session.refresh(task)
finally:
    session.close()
```

### Пошаговое выполнение

```text
session.add(task)
session.commit()
session.refresh(task)
session.close()
```

1. Строка 1: Task добавляется в unit of work Session. — state=pending
2. Строка 2: SQL INSERT отправляется и transaction подтверждается. — transaction=committed
3. Строка 3: Объект получает значения database, например id. — task.id=generated
4. Строка 4: Session освобождает ресурсы. — session=closed

> Commit подтверждает transaction, но не закрывает Session. Rollback отменяет текущую transaction state, но Session затем можно использовать снова.

### Сравнение: Где должна жить transaction boundary одного request?

**Вариант A — Разрозненные commits**

```text
helper_a() → commit
helper_b() → commit
```

Часть operation может сохраниться, даже если следующий шаг завершился ошибкой.

**Вариант B — Одна осознанная transaction**

```text
perform changes
→ commit once
→ rollback on error
```

Operation подтверждается целиком или откатывается.

**Разбор:** Transaction boundary выбирается по смыслу operation, а не по количеству строк кода.

### Разбор сценариев

```text
Session begins transaction
add ORM object
flush SQL
commit
refresh object
close Session

IntegrityError
rollback
close Session
```

- `успешная запись` → строка 5 → `committed object with id`
- `ошибка constraint` → строка 8 → `rollback before reuse`
- `request завершён` → строка 6 → `Session closed`

| Понятие | Смысл |
| --- | --- |
| `add` | Поместить object в unit of work. |
| `flush` | Отправить SQL внутри текущей transaction без финального подтверждения. |
| `commit` | Подтвердить transaction. |
| `refresh` | Повторно получить database-generated values. |
| `rollback` | Отменить failed или незавершённую transaction. |
| `close` | Освободить Session resources. |

## 5. ORM-модель, Pydantic-schema и response — не одно и то же

В Database API одновременно существуют несколько представлений задачи. Они похожи по полям, но принадлежат разным границам и не должны сливаться в один универсальный класс.

### Task model

**Метка:** `ORM`.

Описывает table, columns, keys и relationships.

```text
class Task(Base)
```

### TaskCreate schema

**Метка:** `input`.

Проверяет JSON body, который прислал client.

```text
title + priority
```

### TaskRead schema

**Метка:** `output`.

Определяет безопасную форму response.

```text
id + title + is_done
```

### Три формы

```text
class Task(Base):
    __tablename__ = "tasks"
    id: Mapped[int] = mapped_column(primary_key=True)
    title: Mapped[str]

class TaskCreate(BaseModel):
    title: str
    priority: int

class TaskRead(BaseModel):
    id: int
    title: str
    priority: int
    is_done: bool

    model_config = ConfigDict(from_attributes=True)
```

### Сравнение: Почему не возвращать ORM-object без response schema?

**Вариант A — Случайная сериализация**

```text
return db_task
```

Форма зависит от attributes и может раскрыть внутренние поля.

**Вариант B — Явный response model**

```text
@router.get(..., response_model=TaskRead)
```

Client получает стабильный документированный contract.

**Разбор:** ORM отвечает за persistence, Pydantic — за API boundary.

> Похожесть полей не означает одинаковую ответственность. Разделение schemas особенно важно при password, internal flags и relationships.

1. **Request schema.** Отбирает разрешённые client fields и выполняет validation.
2. **CRUD input.** Обычная функция получает schema data и Session.
3. **ORM model.** Представляет database row внутри transaction.
4. **Response schema.** Сериализует только стабильный public contract.

Разделение предотвращает случайную передачу internal columns и делает изменение database layer менее опасным для client.

### Mapping direction

```text
JSON body
→ TaskCreate
→ model_dump()
→ Task ORM
→ INSERT row
→ Task ORM with id
→ TaskRead
→ JSON response
```

## 6. Путь request через get_db и query

Ключевая сквозная модель этапа связывает предыдущий FastAPI pipeline с database lifecycle. Dependency создаёт Session, endpoint или CRUD-функция строит statement, database возвращает rows, response schema формирует JSON.

### Get_db dependency

```text
def get_db():
    db = SessionLocal()

    try:
        yield db
    finally:
        db.close()
```

### Endpoint чтения

```text
@router.get("/", response_model=list[TaskRead])
def get_tasks(
    db: Annotated[Session, Depends(get_db)],
):
    statement = select(Task).order_by(Task.id)
    return db.scalars(statement).all()
```

### Соберите путь GET /tasks

Расположите этапы request и database query.

Правильный порядок:
1. `client отправляет GET /tasks`
2. `FastAPI выбирает router endpoint`
3. `get_db создаёт Session и yield`
4. `код строит select(Task)`
5. `Session выполняет SQL`
6. `database возвращает rows как ORM objects`
7. `TaskRead сериализует attributes`
8. `finally закрывает Session`
9. `client получает 200 + JSON array`

Session существует внутри request boundary и освобождается независимо от успешного или ошибочного результата.

> Yield dependency особенно важна для ресурсов: код до yield подготавливает resource, код finally гарантирует cleanup.

1. **Создать Session.** Dependency подготавливает database resource.
2. **Выполнить operation.** CRUD-функция строит statement и работает с ORM objects.
3. **Сформировать response.** Pydantic читает attributes до завершения request.
4. **Освободить resource.** Finally закрывает Session даже при HTTPException.

Lifecycle resource должен охватывать всю database operation, но не жить глобально между независимыми requests.

### Пошаговое выполнение

```text
db = next(get_db())
statement = select(Task)
items = db.scalars(statement).all()
return items
```

1. Строка 1: Dependency создаёт Session и передаёт её operation. — db=Session
2. Строка 2: Statement описывает выборку, но ещё не содержит result rows. — statement=Select
3. Строка 3: Session выполняет SQL и scalars извлекает ORM objects. — items=list[Task]
4. Строка 4: Response model читает attributes и создаёт JSON. — response=200 array

### наблюдение SQL

```text
SELECT tasks.id, tasks.title, tasks.priority
FROM tasks
ORDER BY tasks.id
LIMIT ? OFFSET ?
```

## 7. Query, constraints и transaction errors

Database API должен различать пустой результат, неизвестный id, невалидный request и нарушение constraint. Эти ситуации возникают на разных слоях и требуют разных реакций.

| Понятие | Смысл |
| --- | --- |
| `422` | Pydantic или FastAPI отклонили вход до database operation. |
| `404` | SELECT выполнен, но row с заданным id не найден. |
| `200 / []` | Collection query успешен, но подходящих rows нет. |
| `IntegrityError` | INSERT или UPDATE нарушили unique, foreign key или другой constraint. |
| `rollback` | Обязателен после failed transaction перед дальнейшей работой Session. |
| `500` | Непредвиденный defect, который нельзя выдавать client как traceback. |

### Найдите ошибку

```python
try:
    session.add(task)
    session.commit()
except IntegrityError:
    raise HTTPException(409, "duplicate")
```

Почему Session может остаться непригодной для следующего query?

1. После IntegrityError не выполнен rollback
2. HTTPException нельзя использовать с SQLAlchemy
3. Commit всегда закрывает Session

**Ответ:** 1. Failed transaction должна быть явно откатана.

Исправление:

```python
try:
    session.add(task)
    session.commit()
except IntegrityError as error:
    session.rollback()
    raise HTTPException(
        status_code=409,
        detail="Task already exists",
    ) from error
```

### Разбор сценариев

```text
validate request
execute SELECT or INSERT
check empty result
commit transaction
catch IntegrityError
rollback
return response
```

- `invalid body` → строка 1 → `422`
- `unknown id` → строка 3 → `404`
- `successful insert` → строка 4 → `201`
- `duplicate unique` → строка 6 → `rollback + 409`

> Rollback — не способ скрыть ошибку. Он восстанавливает transaction state Session, после чего API возвращает честный error response.

### Error responsibility matrix

```text
request schema invalid     → 422
row not found              → 404
unique constraint conflict → rollback + 409
foreign key conflict       → rollback + 400/409
empty collection           → 200 + []
unexpected defect          → log + 500
```

### Определить слой

Ошибка возникла до Session, во время SELECT, на commit или при serialization?

### Сохранить причину

Использовать raise ... from error и server log без утечки traceback client.

### Восстановить Session

После failed commit выполнить rollback.

### Вернуть честный status

Client должен отличить validation, missing resource и conflict.

## 8. Relations, loading и migrations завершают модель

Когда задачи связаны с категориями, database начинает отвечать не только за отдельные rows, но и за целостность графа данных. Alembic фиксирует изменение этого графа во времени.

### Карточки терминов

- **Foreign key:** Database constraint: tasks.category_id должен ссылаться на существующую category.
- **Relationship:** ORM navigation: task.category и category.tasks связывают Python objects.
- **Lazy loading:** Связанные rows могут загружаться при обращении к attribute, создавая дополнительные queries.
- **N+1:** Один query получает parents, затем отдельный query выполняется для каждого набора children.
- **Eager loading:** Связи загружаются заранее выбранной стратегией, например selectinload.
- **Migration:** Versioned script переводит schema из одного состояния в другое.

### Migration как переход

```text
revision A
  tasks(id, title, priority)
      ↓ upgrade
revision B
  tasks(id, title, priority, category_id)
  categories(id, name)
      ↓ downgrade
revision A
```

### типичный migration workflow

```text
$ alembic revision --autogenerate -m "add categories"
Generating ..._add_categories.py
$ alembic upgrade head
Running upgrade ... -> ...
$ alembic current
<revision> (head)
```

### Правда или ложь

Если ORM-модель изменилась, существующая database schema обновится автоматически при обычном запуске приложения.

**Ответ:** ложь. Изменение Python-класса и изменение существующей table — разные действия; schema обновляется migration.

**Финальная проверка общей теории:**

> Итоговый навык — объяснить не только код models.py, но и фактические tables, SQL queries, transaction boundaries и migration history.

| Понятие | Смысл |
| --- | --- |
| `request boundary` | Какие path, query, body и headers пришли от client? |
| `dependency boundary` | Создалась ли Session и будет ли она закрыта? |
| `query boundary` | Какой statement построен и какие rows ожидаются? |
| `transaction boundary` | Где commit, где возможен IntegrityError и rollback? |
| `serialization boundary` | Какая response schema читает ORM attributes? |
| `migration boundary` | Соответствует ли фактическая schema текущей Alembic revision? |

## Главное

- Database решает устойчивость и целостность сохранённых данных.
- SQLite, SQLAlchemy и Alembic выполняют разные роли.
- ORM опирается на tables, columns, keys и constraints.
- Engine, Session и transaction нельзя смешивать в одно понятие.
- Pydantic schema и ORM model принадлежат разным границам.
- Get_db связывает request lifecycle и Session lifecycle.
- IntegrityError требует rollback и понятного API response.
- Relations и migrations завершают воспроизводимую модель Database API.

## Практика

Возьмите любой endpoint будущего StudyHub и письменно проследите: request → dependency → Session → SQL statement → transaction → ORM object → response schema → JSON.

### Критерий готовности

- Ученик объясняет модель своими словами.
- Может проследить один сценарий от начала до результата.
- Называет ограничение текущего решения.
- Понимает, какой следующий урок решает это ограничение.
