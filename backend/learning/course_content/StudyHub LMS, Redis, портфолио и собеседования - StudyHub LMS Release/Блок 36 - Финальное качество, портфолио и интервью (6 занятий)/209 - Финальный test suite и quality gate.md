## 209. Финальный test suite и quality gate

### Цель

Собрать тестовую стратегию вокруг главных рисков проекта, а не максимального процента coverage.

### Почему тема появляется сейчас

После аудита критичные контракты должны защищаться автоматически.

### Основное содержание

- unit, integration и end-to-end tests;
- auth и permissions cases;
- transaction rollback;
- migrations from clean database;
- cache and Redis failure;
- external API mock;
- deterministic tests;
- CI quality gate;

### Практика StudyHub

Составить risk-based test matrix, закрыть пробелы и запустить полный pipeline с чистой PostgreSQL/Redis.

### Визуальная и интерактивная подача

Пирамида тестов как ориентир, но не догма; матрица риск → тест; финальный CI маршрут.

### Результат занятия

Release блокируется при нарушении ключевого контракта.

### Граница занятия

Не гнаться за 100% coverage и не тестировать implementation details без пользы.

### Критерии готовности

- ученик объясняет главную модель своими словами;
- запускает минимальный пример или инфраструктурный сценарий;
- изменяет один параметр и предсказывает результат;
- проверяет успешный и ошибочный путь;
- фиксирует воспроизводимую проверку, тест или runbook;
- делает один осмысленный Git-коммит;
