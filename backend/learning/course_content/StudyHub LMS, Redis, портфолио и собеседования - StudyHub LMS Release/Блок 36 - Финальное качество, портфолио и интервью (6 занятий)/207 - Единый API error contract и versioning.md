## 207. Единый API error contract и versioning

### Цель

Убрать случайные формы ошибок и зафиксировать стабильную внешнюю границу API.

### Почему тема появляется сейчас

Функциональность завершена; теперь клиент не должен разбирать разные error bodies для каждого router.

### Основное содержание

- единый error schema;
- code, message, details, request_id;
- validation errors;
- domain errors;
- unexpected 500;
- exception handlers;
- API version prefix;
- backward compatibility как модель;

### Практика StudyHub

Привести 400/401/403/404/409/422/429/500 к согласованному контракту и обновить OpenAPI examples.

### Визуальная и интерактивная подача

Матрица error source → handler → response; CompareSolutions случайных и единых ошибок.

### Результат занятия

Клиент получает предсказуемый error body с request id.

### Граница занятия

Поддержка нескольких активных major versions не реализуется.

### Критерии готовности

- ученик объясняет главную модель своими словами;
- запускает минимальный пример или инфраструктурный сценарий;
- изменяет один параметр и предсказывает результат;
- проверяет успешный и ошибочный путь;
- фиксирует воспроизводимую проверку, тест или runbook;
- делает один осмысленный Git-коммит;
