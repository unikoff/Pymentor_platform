## 206. Тестирование кеша, rate limit и фоновых ошибок

### Цель

Закрепить эксплуатационные функции через наблюдаемые тесты и режим деградации.

### Почему тема появляется сейчас

Финал блока должен доказать, что Redis не сделал API хрупким.

### Основное содержание

- fake Redis или изолированный test instance;
- cache hit/miss tests;
- invalidation test;
- TTL test без долгого ожидания;
- rate limit test;
- Redis unavailable fallback;
- background task test;
- метрики и логи блока;

### Практика StudyHub

Создать набор tests для кеша, invalidation, 429, Redis failure и mock-notification; повторить измерение catalog latency.

### Визуальная и интерактивная подача

Контрольная панель scenarios; матрица dependency failure → API behavior.

### Результат занятия

Redis-функции проверяются и имеют документированную деградацию.

### Граница занятия

Не маскировать ошибки PostgreSQL кешем и не утверждать пользу без измерений.

### Критерии готовности

- ученик объясняет главную модель своими словами;
- запускает минимальный пример или инфраструктурный сценарий;
- изменяет один параметр и предсказывает результат;
- проверяет успешный и ошибочный путь;
- фиксирует воспроизводимую проверку, тест или runbook;
- делает один осмысленный Git-коммит;

## Контрольная точка блока

- Каталог использует контролируемый cache-aside, rate limit защищает один сценарий, а простое уведомление выполняется после response.
- есть основной успешный сценарий;
- есть ожидаемый сбой и понятная диагностика;
- результат воспроизводится другим человеком;
- README, схема, test matrix или runbook обновлены;
- ученик прослеживает путь данных и ответственность компонентов без подсказки;
