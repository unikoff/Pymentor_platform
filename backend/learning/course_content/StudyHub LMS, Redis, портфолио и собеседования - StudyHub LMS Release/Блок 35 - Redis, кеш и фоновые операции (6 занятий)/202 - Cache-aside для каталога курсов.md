## 202. Cache-aside для каталога курсов

### Цель

Кешировать дорогой read-path, сохраняя PostgreSQL источником истины.

### Почему тема появляется сейчас

Есть публичный каталог с повторяющимся запросом и стабильным response.

### Основное содержание

- cache-aside pattern;
- read from cache;
- miss → PostgreSQL → set cache;
- TTL;
- response serialization;
- cache key с параметрами filters/page;
- измерение latency;
- fallback при Redis error;

### Практика StudyHub

Добавить кеш `/courses`, сравнить SQL logs и latency первого и второго request.

### Визуальная и интерактивная подача

Ветвление hit/miss; CompareSolutions cache whole ORM object vs serialized response.

### Результат занятия

Повторный запрос каталога не обращается к PostgreSQL до истечения TTL.

### Граница занятия

Не кешировать всё API. Выбирается один измеримый read-path.

### Критерии готовности

- ученик объясняет главную модель своими словами;
- запускает минимальный пример или инфраструктурный сценарий;
- изменяет один параметр и предсказывает результат;
- проверяет успешный и ошибочный путь;
- фиксирует воспроизводимую проверку, тест или runbook;
- делает один осмысленный Git-коммит;
