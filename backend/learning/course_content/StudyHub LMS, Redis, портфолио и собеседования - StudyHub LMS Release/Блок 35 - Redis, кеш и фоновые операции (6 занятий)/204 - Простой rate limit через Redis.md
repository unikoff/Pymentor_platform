## 204. Простой rate limit через Redis

### Цель

Ограничить один чувствительный endpoint по счётчику и временному окну.

### Почему тема появляется сейчас

Redis уже используется как временное shared state; rate limit показывает второе обоснованное применение.

### Основное содержание

- request key;
- counter;
- time window и TTL;
- 429 Too Many Requests;
- Retry-After;
- идентификатор user/IP как проектное решение;
- атомарность increment на прикладном уровне;
- fail-open/fail-closed как осознанная граница;

### Практика StudyHub

Ограничить endpoint отправки учебного уведомления или login-like action и проверить сброс после TTL.

### Визуальная и интерактивная подача

Счётчик запросов внутри окна; BranchExplorer allowed/blocked/reset.

### Результат занятия

Один endpoint возвращает 429 по ясному контракту и не затрагивает остальные.

### Граница занятия

Не строить глобальный distributed rate limiter production-уровня.

### Критерии готовности

- ученик объясняет главную модель своими словами;
- запускает минимальный пример или инфраструктурный сценарий;
- изменяет один параметр и предсказывает результат;
- проверяет успешный и ошибочный путь;
- фиксирует воспроизводимую проверку, тест или runbook;
- делает один осмысленный Git-коммит;
