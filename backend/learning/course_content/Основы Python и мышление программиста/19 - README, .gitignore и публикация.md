# README, .gitignore и публикация

Подготовим проект к передаче другому человеку: README объяснит запуск, .gitignore уберёт лишние файлы, а Git-история покажет понятные этапы разработки.

**Темы:** README и запуск · Git и публикация

## 01. Что нужно подготовить перед публикацией

Перед передачей устройства в коробку кладут инструкцию, проверяют комплект и убирают личные данные. С проектом так же: сначала проверяется запуск, затем README, состав будущего коммита и только после этого выполняется `git push`.

1. **Объяснить проект:** README описывает назначение, возможности и ограничения.
2. **Сделать запуск воспроизводимым:** другой человек выполняет конкретные команды без устных подсказок.
3. **Проверить историю:** `git status`, `git diff --staged` и осмысленные коммиты предшествуют push.

Дополнительно вы разберёте назначение `.gitignore`, опасность уже отслеживаемых секретов и проверку репозитория после публикации.

> Проверка README простая: сможет ли другой человек понять назначение проекта и запустить его без устных подсказок автора.

## 02. README как инструкция внутри коробки

README похож на инструкцию внутри коробки с устройством. Человек впервые видит проект и должен быстро понять, что он делает, что требуется для запуска и какие ограничения есть у текущей версии.

**минимальная структура**

```text
# StudyHub Planner

Консольный планировщик учебных задач на Python.

## Возможности

## Требования

## Установка и запуск

## Использование

## Ограничения
```

### Описание — что

Что делает приложение и для какого сценария оно создано.

### Запуск — как

Какие команды нужно выполнить в новой папке.

### Ограничения — границы

Что сознательно ещё не реализовано.

### Сравнение решений

**Вопрос:** Какое описание полезнее?

### Общее

```python
Это мой учебный проект.
```

Не объясняет назначение и возможности.

### Конкретное

```python
Консольный планировщик учебных задач на Python.
Поддерживает добавление, завершение и статистику.
```

Читатель понимает результат до запуска.

**Вывод:** предпочтителен правый вариант. Описание должно отвечать, что получит пользователь.

## 03. Документация не должна обещать лишнего

Инструкция должна описывать устройство, которое действительно лежит в коробке. Если README обещает сохранение данных, а задачи исчезают после перезапуска, документация вводит пользователя в заблуждение.

### Сравнение решений

**Вопрос:** Как описать хранение данных первого месяца?

### Скрыть ограничение

```python
Задачи сохраняются в приложении.
```

Фраза может создать ожидание сохранения после перезапуска.

### Сказать точно

```python
Данные хранятся только в памяти программы
и исчезают после завершения процесса.
```

Ограничение текущей версии понятно заранее.

**Вывод:** предпочтителен правый вариант. Честное ограничение показывает понимание архитектуры, а не слабость проекта.

### Верно или неверно

README можно обновить заранее, добавив возможности следующей версии, чтобы репозиторий выглядел серьёзнее.

**Ответ:** неверно. Документация должна соответствовать текущему поведению.

### Вспомните модель

**Вопрос:** Когда README нужно проверять на актуальность?

**Ответ:** После изменения пользовательского поведения, команды запуска, структуры файлов, требований или известных ограничений.

## 04. Другой человек должен повторить запуск

Хорошая инструкция похожа на маршрут с конкретными поворотами. Читатель не должен угадывать папку, имя файла или порядок команд. Каждый шаг нужно предварительно повторить в новой папке.

**Windows PowerShell**

```powershell
git clone <адрес-репозитория>
cd studyhub-planner

python -m venv .venv
.\.venv\Scripts\Activate.ps1

python main.py
```

### Порядок чистого запуска

Расположите действия в правильном порядке.

**Фрагменты**

- `git clone <url>`
- `cd studyhub-planner`
- `python -m venv .venv`
- `.\.venv\Scripts\Activate.ps1`
- `python main.py`

**Правильный порядок**

```powershell
git clone <url>
cd studyhub-planner
python -m venv .venv
.\.venv\Scripts\Activate.ps1
python main.py
```

Сначала создаётся локальная копия и выбирается папка, затем окружение и запуск.

> Лучший тест инструкции — выполнить её в новой папке строго по README, не используя знания, которых там нет.

## 05. .gitignore как список «не упаковывать»

`.gitignore` похож на список вещей, которые нельзя класть в посылку: виртуальное окружение, служебные файлы и секреты. Но если предмет уже отправили раньше, новый список не вернёт его из истории.

**базовые правила**

```powershell
.venv/
__pycache__/
*.pyc
.env
```

### Сопоставление

Соедините правило с причиной.

| Левая часть | Правая часть |
| --- | --- |
| .venv/ | окружение воспроизводится и зависит от системы |
| __pycache__/ | служебные файлы Python |
| *.pyc | скомпилированные технические файлы |
| .env | может содержать токены и пароли |

Исходный код и документация остаются в Git, воспроизводимые и секретные файлы — нет.

### Верно или неверно

Если `.env` уже был закоммичен, достаточно добавить его в `.gitignore`, и секрет станет безопасным.

**Ответ:** неверно. Файл уже находится в истории; секрет нужно заменить и отдельно убрать отслеживание.

> `.gitignore` — профилактика, а не машина времени. Скомпрометированный токен нужно отозвать или заменить.

## 06. Проверяем, что попадёт в коммит

`git status` показывает список подготовленных коробок, а `git diff --staged` позволяет открыть их и увидеть точное содержимое будущего коммита. Это последняя проверка перед созданием точки истории.

### Сопоставление

Соедините команду с вопросом, на который она отвечает.

| Левая часть | Правая часть |
| --- | --- |
| git status | какие файлы новые, изменённые и staged? |
| git diff | что изменено, но ещё не добавлено? |
| git diff --staged | что войдёт в следующий коммит? |
| git log --oneline | как выглядит история проекта? |

Команды проверяют разные уровни состояния репозитория.

### Безопасный коммит

Расположите действия перед публикацией.

**Фрагменты**

- `python main.py`
- `git status`
- `git add main.py README.md .gitignore`
- `git diff --staged`
- `git commit -m "write setup instructions"`

**Правильный порядок**

```powershell
python main.py
git status
git add main.py README.md .gitignore
git diff --staged
git commit -m "write setup instructions"
```

Сначала проверяется поведение, затем состав и содержимое коммита.

### Разбор ошибки

```powershell
git add .
git commit -m "final"
git push
```

**Вопрос:** Чего не хватает перед публикацией?

A. Проверки status, staged diff и локального запуска
B. Удаления README
C. Преобразования main.py в строку

**Ответ:** A. Проверки status, staged diff и локального запуска

Массовое добавление без проверки может включить окружение, секреты и временные файлы.

**Исправленный вариант**

```powershell
python main.py
git status
git add main.py README.md .gitignore
git diff --staged
git commit -m "write setup instructions"
git push
```

## 07. Публикуем понятную историю проекта

История Git похожа на журнал этапов сборки. Сообщения `add task creation` и `write setup instructions` объясняют развитие проекта. Названия вроде `final2` не помогают восстановить смысл изменений.

### Сравнение решений

**Вопрос:** Какая история полезнее?

### Непонятные точки

```python
fix
update
final
final2
```

Нельзя восстановить смысл изменений.

### Этапы проекта

```text
create planner menu
add task creation
add search and statistics
write setup instructions
```

Каждый коммит описывает законченную возможность.

**Вывод:** предпочтителен правый вариант. История должна показывать развитие, а не настроение автора.

**финальная локальная проверка**

```powershell
python main.py
git status
git diff --staged
git log --oneline
git push
```

После публикации откройте репозиторий: README должен отображаться, `.venv` и `.env` должны отсутствовать, а `main.py` — находиться в ожидаемом месте.

`push` не исправляет ошибку запуска, не создаёт документацию и не удаляет секреты.

## 08. Проверьте основную модель

### Вопрос 1

Для кого пишется README?

A. для автора и другого разработчика
B. только для Python
C. только для GitHub

**Ответ:** A. для автора и другого разработчика

Документ должен быть полезен человеку без контекста проекта.

### Вопрос 2

Что делает .gitignore?

A. предотвращает добавление совпадающих новых файлов
B. удаляет историю
C. запускает тесты

**Ответ:** A. предотвращает добавление совпадающих новых файлов

Уже отслеживаемые файлы он автоматически не удаляет.

### Вопрос 3

Почему .env не коммитят?

A. там могут быть секреты
B. это всегда пустой файл
C. Python его не читает

**Ответ:** A. там могут быть секреты

Токены и пароли не должны попадать в историю.

### Вопрос 4

Что показывает git diff --staged?

A. будущий коммит
B. только удалённые ветки
C. версию Python

**Ответ:** A. будущий коммит

Команда показывает подготовленные изменения.

### Главное после занятия

- README описывает фактическое поведение и запуск.
- Ограничения проекта указываются честно.
- `.gitignore` предотвращает случайное добавление файлов.
- Уже опубликованный секрет нужно заменить.
- `git diff --staged` проверяет будущий коммит.
- `git push` публикует готовую локальную историю.

### Практика

Оформите README и .gitignore, проверьте staged diff и опубликуйте чистый репозиторий.
