# Ошибки, traceback и отладка

Научимся читать traceback снизу вверх, различать типы ошибок, отделять симптом от причины и исправлять проблему как проверяемый эксперимент, а не случайным редактированием.

**Темы:** traceback и тип ошибки · диагностика и регрессия

## 01. Маршрут занятия

Ошибка — это техническое сообщение о нарушенном правиле. Отладка начинается не с переписывания кода, а с наблюдения: что именно произошло, на каких данных и где остановилось выполнение.

1. **Прочитать сообщение:** начать с последней строки traceback и определить тип ошибки.
2. **Проверить гипотезу:** найти строку своего файла и увидеть реальные значения перед ней.
3. **Доказать исправление:** изменить одну причину и повторить ошибочный, обычный и граничный сценарии.

Дополнительно вы разберёте `SyntaxError`, `NameError`, `TypeError`, `ValueError`, `IndexError`, `KeyError`, минимальный пример и опасность `except: pass`.

> Полезная формулировка ошибки описывает не цвет текста, а **неверное предположение кода о данных**.

## 02. Traceback читают с последней строки

Внизу находится тип и краткое сообщение. Затем поднимайтесь к ближайшей строке собственного файла.

**код**

```python
def show_first(tasks):
    print(tasks[0])

show_first([])
```

**упрощённый traceback**

```text
Traceback (most recent call last):
  File "main.py", line 4, in <module>
    show_first([])
  File "main.py", line 2, in show_first
    print(tasks[0])
IndexError: list index out of range
```

### Порядок чтения traceback

Расположите действия от первого наблюдения до изменения кода.

**Фрагменты**

- `прочитать последнюю строку`
- `найти ближайшую строку своего файла`
- `проверить реальные значения`
- `сформулировать нарушенное правило`
- `изменить одну причину`
- `переписать всю функцию сразу` — теряется причинность

**Правильный порядок**

```python
прочитать последнюю строку
найти ближайшую строку своего файла
проверить реальные значения
сформулировать нарушенное правило
изменить одну причину
```

Сначала наблюдение и гипотеза, затем минимальное изменение.

### Вспомните модель

**Вопрос:** Почему причина не в print, хотя traceback указывает на строку print(tasks[0])?

**Ответ:** `print` только пытается показать переданное значение. Реальная причина — предположение, что у пустого списка существует элемент с индексом 0.

## 03. SyntaxError и NameError

`SyntaxError` означает, что Python не смог разобрать структуру файла. `NameError` возникает уже при выполнении, когда имя не найдено.

### Структура кода нарушена — SyntaxError

Проверьте двоеточия, скобки, кавычки, отступы и строку выше указателя.

```text
if priority > 3
    print("Высокий")
```

### Имя не определено — NameError

Проверьте порядок выполнения, регистр, опечатку и область видимости.

```python
print(task_title)
# task_title не присваивался
```

### Разбор ошибки

```python
print("Начало")
if priority > 3
    print("Высокий")
print("Конец")
```

**Вопрос:** Успеет ли напечататься Начало?

A. Да, ошибка находится ниже
B. Нет, Python не сможет разобрать файл
C. Только если priority равен 3

**Ответ:** B. Нет, Python не сможет разобрать файл

SyntaxError мешает построить структуру программы, поэтому выполнение файла не начинается.

**Исправленный вариант**

```python
print("Начало")
priority = 5
if priority > 3:
    print("Высокий")
print("Конец")
```

### Разбор ошибки

```python
title = "SQL"
print(Title)
```

**Вопрос:** Почему возникает NameError?

A. Python различает регистр, имя Title не определено
B. Строки нельзя хранить в переменных
C. print требует два аргумента

**Ответ:** A. Python различает регистр, имя Title не определено

title и Title — разные имена.

**Исправленный вариант**

```python
title = "SQL"
print(title)
```

> При `SyntaxError` указатель может стоять там, где Python окончательно заметил проблему, а незакрытая скобка находиться строкой выше.

## 04. TypeError и ValueError

Эти ошибки часто путают. `TypeError` сообщает о неподходящем типе или операции, `ValueError` — о неподходящем содержимом допустимого типа.

### Сравнение решений

**Вопрос:** Чем отличаются два случая?

### TypeError

```python
age = "18"
print(age + 1)
```

Операция + не определена между str и int.

### ValueError

```python
age = int("восемнадцать")
```

int принимает строку, но её содержимое нельзя преобразовать.

**Вывод:** предпочтителен правый вариант. Оба фрагмента ошибочны, но нарушают разные правила.

### Сопоставление

Соедините фрагмент с типом ошибки.

| Код | Ошибка |
| --- | --- |
| "18" + 1 | TypeError |
| int("абв") | ValueError |
| [1, 2][5] | IndexError |
| {"title": "SQL"}["priority"] | KeyError |

Тип ошибки сужает список возможных причин.

### Разбор ошибки

```python
age_text = "18"
next_age = age_text + 1
```

**Вопрос:** Какое наблюдение нужно сделать первым?

A. Проверить тип age_text
B. Переименовать next_age
C. Удалить число 1

**Ответ:** A. Проверить тип age_text

Фактический тип показывает, почему операция не определена.

**Исправленный вариант**

```python
age_text = "18"
age = int(age_text)
next_age = age + 1
```

### Верно или неверно

`int([])` и `int("abc")` обязательно вызовут один и тот же тип ошибки.

**Ответ:** неверно. Список приводит к TypeError, а строка неподходящего содержания — к ValueError.

## 05. IndexError, KeyError и неверные предположения

Эти ошибки часто показывают, что код ожидал данные, которых нет: позицию в последовательности или поле в словаре.

### Позиция отсутствует — IndexError

В списке существует только индекс 0.

```python
items = ["Python"]
print(items[1])
```

### Ключ отсутствует — KeyError

Обязательное поле не создано или имя ключа написано неверно.

```python
task = {"title": "SQL"}
print(task["priority"])
```

### Сравнение решений

**Вопрос:** Как исправлять отсутствующий priority?

### Скрыть обязательное поле

```python
priority = task.get("priority")
```

Ошибка исчезла, но priority может стать None.

### Сначала определить правило

```python
# если поле обязательно
task = {"title": "SQL", "priority": 1}

# если поле необязательно
priority = task.get("priority", 1)
```

Решение соответствует договору модели.

**Вывод:** предпочтителен правый вариант. Техническое подавление ошибки не заменяет предметное решение.

### Разбор ошибки

```python
task = {"title": "SQL"}
priority = task.get("priority")
print(priority + 1)
```

**Вопрос:** Почему замена [] на get не решила проблему?

A. get вернул None, и сложение с числом невозможно
B. Словарь нельзя передавать в get
C. Ключи должны быть числами

**Ответ:** A. get вернул None, и сложение с числом невозможно

KeyError исчез, но обязательное значение не появилось.

**Исправленный вариант**

```python
task = {"title": "SQL"}
priority = task.get("priority", 1)
print(priority + 1)
```

### Вспомните модель

**Вопрос:** Что важнее: убрать текст ошибки или восстановить верное правило данных?

**Ответ:** Нужно восстановить правило. Ошибка может исчезнуть после неудачной подстановки, но программа продолжит работать с неверным состоянием.

## 06. Отладка как управляемый эксперимент

Надёжная отладка повторяет научную проверку: воспроизвести, наблюдать, сформулировать гипотезу, изменить одну вещь и повторить.

| Код | Назначение |
| --- | --- |
| 1. воспроизвести | получить ошибку стабильно на известном входе |
| 2. наблюдать | прочитать traceback, значение и тип |
| 3. предположить | сформулировать конкретную причину |
| 4. одно изменение | не переписывать соседние части |
| 5. повторить | запустить тот же вход |
| 6. проверить регрессию | обычный и граничный сценарии |

**диагностическая печать**

```python
print("DEBUG task:", task)
print("DEBUG type:", type(task))
print("DEBUG keys:", list(task.keys()))
```

### Сравнение решений

**Вопрос:** Какой процесс сохраняет причинность?

### Случайная серия правок

```python
переименовать переменные
переписать условие
добавить try/except
изменить данные
```

Непонятно, какое изменение повлияло на результат.

### Одна гипотеза

```python
проверить, пуст ли список
добавить одну явную проверку
повторить тот же вход
```

Можно доказать связь причины и исправления.

**Вывод:** предпочтителен правый вариант. Маленькое изменение уменьшает риск нового дефекта.

> Диагностические `print` удаляют после расследования. Иначе они засоряют интерфейс и могут сломать ожидаемый вывод платформы.

## 07. Минимальный пример и регрессионная проверка

Уменьшите данные до самого короткого сценария, который всё ещё воспроизводит проблему. После исправления проверьте исходный, обычный и граничный входы.

**минимальный пример KeyError**

```python
tasks = [{"title": "Python"}]
print(tasks[0]["priority"])
```

```python
def first_char(text):
    if text == "":
        return ""
    return text[0]

print(first_char(""))
print(first_char("P"))
print(first_char("Python"))
print(first_char(" "))
```

**Пошаговый разбор**

1. Сценарий исходной ошибки: пустая строка. **Состояние:** результат = ""
2. Минимальный обычный вход из одного символа. **Состояние:** результат = "P"
3. Обычная строка сохраняет старое корректное поведение. **Состояние:** результат = "P"
4. Пробел не является пустой строкой и возвращается как символ. **Состояние:** результат = " "

### Разбор ошибки

```python
try:
    print(task["priority"])
except:
    pass
```

**Вопрос:** Почему этот код мешает отладке?

A. Он скрывает тип и причину любой ошибки
B. try нельзя использовать со словарями
C. pass автоматически удаляет ключ

**Ответ:** A. Он скрывает тип и причину любой ошибки

Голый except поглощает любую ошибку и уничтожает информацию.

**Исправленный вариант**

```python
if "priority" in task:
    print(task["priority"])
else:
    print("Нет обязательного приоритета")
```

### Соберите регрессионную проверку

Расположите сценарии после исправления.

**Фрагменты**

- `вход, на котором была ошибка`
- `обычный рабочий вход`
- `пустой или граничный вход`
- `соседний сценарий, который мог затронуть фикс`

**Правильный порядок**

```python
вход, на котором была ошибка
обычный рабочий вход
пустой или граничный вход
соседний сценарий, который мог затронуть фикс
```

Исправление должно устранить дефект и сохранить прежнее корректное поведение.

## 08. Проверь себя

### Вопрос 1

С какой части traceback начинать?

A. с последней строки
B. с первой строки файла
C. с названия папки

**Ответ:** A. с последней строки

Внизу находится тип и краткое сообщение.

### Вопрос 2

Что вызовет выражение "18" + 1?

A. TypeError
B. ValueError
C. IndexError

**Ответ:** A. TypeError

Операция не определена между str и int.

### Вопрос 3

Что вызовет int("abc")?

A. TypeError
B. ValueError
C. KeyError

**Ответ:** B. ValueError

Тип str допустим, но содержимое нельзя преобразовать.

### Вопрос 4

Зачем уменьшать данные до минимального примера?

A. сократить число возможных причин
B. скрыть traceback
C. не исправлять проект

**Ответ:** A. сократить число возможных причин

Короткий воспроизводимый случай быстрее запускать и анализировать.

### Вопрос 5

Что проверить после исправления?

A. только удачный вход
B. ошибочный, обычный и граничный сценарии
C. только синтаксис

**Ответ:** B. ошибочный, обычный и граничный сценарии

Регрессионная проверка защищает старое корректное поведение.

### Главное после занятия

- Traceback читают с последней строки.
- Тип ошибки сужает круг причин.
- `SyntaxError` мешает разобрать структуру.
- `TypeError` и `ValueError` описывают разные нарушения.
- `IndexError` и `KeyError` часто показывают неверное предположение о данных.
- Отладка начинается с воспроизведения и наблюдения.
- Меняйте одну причину за раз.
- После исправления проверяйте регрессию и удаляйте DEBUG-печать.
