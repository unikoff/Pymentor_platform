# Словари и модель задачи

Соберём одну задачу как набор именованных полей: разберём ключи и значения, обязательный доступ через [], безопасный get(), изменение словаря и список словарей как модель проекта.

**Темы:** ключ → значение · модель задачи

## 01. Маршрут занятия

Список отвечает на вопрос «какой элемент стоит на позиции?», а словарь — «какое значение хранится в поле с таким именем?». Это позволяет описать одну задачу понятнее, чем набором числовых индексов.

1. **Создать форму:** записать поля задачи как пары `ключ: значение`.
2. **Выбрать доступ:** использовать `task["title"]` для обязательного поля и `task.get(...)` для необязательного.
3. **Собрать проект:** хранить одну задачу в словаре, а несколько задач — в списке словарей.

Дополнительно вы добавите и удалите поля, переберёте пары через `items()`, разберёте `KeyError`, ссылки и копирование словаря.

> Главная идея: **имена ключей становятся договором между частями проекта**.

## 02. Словарь хранит именованные поля

Вместо неясного `task[2]` словарь позволяет написать `task["priority"]` и сразу увидеть смысл значения.

### Сравнение решений

**Вопрос:** Какая модель понятнее описывает одну задачу?

### Позиции без имён

```python
task = [1, "SQL", 3, False]
print(task[2])
```

Нужно помнить смысл каждой позиции.

### Именованные поля

```python
task = {
    "id": 1,
    "title": "SQL",
    "priority": 3,
    "is_done": False,
}
```

Каждое значение читается по предметному имени.

**Вывод:** предпочтителен правый вариант. Словарь делает структуру данных самодокументируемой.

**форма одной задачи**

```python
task = {
    "id": 1,
    "title": "Изучить словари",
    "priority": 3,
    "is_done": False,
}

print(task["title"])
print(task["priority"])
```

### Предскажите результат

```python
task = {"priority": 1, "priority": 5}
print(task)
```

*Подсказка: Ключи одного словаря уникальны, повторное значение заменяет предыдущее.*

**Ответ**

```text
{'priority': 5}
```

### Верно или неверно

Запись `{}` создаёт пустое множество.

**Ответ:** неверно. {} создаёт пустой словарь. Для пустого множества нужен set().

## 03. Обязательное поле и KeyError

Квадратные скобки означают: ключ обязан существовать по договору модели.

### Разбор ошибки

```python
task = {"title": "SQL"}
print(task["priority"])
```

**Вопрос:** Какое правило нарушено?

A. Словарь не может хранить строки
B. Обязательный ключ priority отсутствует
C. print нельзя использовать со словарём

**Ответ:** B. Обязательный ключ priority отсутствует

Доступ через [] требует существующий ключ и при его отсутствии вызывает KeyError.

**Исправленный вариант**

```python
task = {"title": "SQL", "priority": 1}
print(task["priority"])
```

`"priority" in task` проверяет наличие ключа, а не значения.

Для обязательного `title` ошибка полезна: она быстро показывает, что форма данных нарушена.

Если одна функция создаёт ключ `name`, а другая читает `title`, исправлять нужно место создания, а не маскировать проблему.

### Вспомните модель

**Вопрос:** Почему KeyError иногда полезнее значения по умолчанию?

*Подсказка: Подумайте об обязательном поле модели.*

**Ответ:** Если поле обязательно, ошибка сразу сообщает о нарушении формы. Значение по умолчанию могло бы скрыть опечатку или дефект места, где словарь был создан.

## 04. get() для действительно необязательного поля

`get()` возвращает значение либо выбранный запасной вариант, но не добавляет ключ в словарь.

```python
task = {"title": "Git"}
priority = task.get("priority", 1)
print(priority)
print(task)
```

**Пошаговый разбор**

1. Словарь содержит только обязательный title. **Состояние:** task = {title: Git}
2. Ключ priority отсутствует, поэтому get возвращает значение по умолчанию 1. **Состояние:** priority = 1
3. Печатается возвращённое значение. **Состояние:** вывод = 1
4. Словарь не изменился: get ничего не добавляет. **Состояние:** task = {title: Git}; вывод = 1 ⏎ {'title': 'Git'}

### Заполните пропуск

Прочитайте необязательный приоритет со значением 1

```python
priority = task.___("priority", 1)
```

**Варианты:** `get`, `pop`, `items`

**Ответ:** `get`

get("priority", 1) возвращает значение либо 1 и не меняет словарь.

### Разбор ошибки

```python
task = {"priority": 0}
priority = task.get("priority") or 1
print(priority)
```

**Вопрос:** Почему вывод равен 1, хотя ключ существует?

A. Ноль является falsy, поэтому or выбирает 1
B. get не умеет возвращать числа
C. В словаре нельзя хранить 0

**Ответ:** A. Ноль является falsy, поэтому or выбирает 1

Значение 0 ложно в логическом контексте. Значение по умолчанию лучше передать вторым аргументом get.

**Исправленный вариант**

```python
task = {"priority": 0}
priority = task.get("priority", 1)
print(priority)  # 0
```

> Не заменяйте через `get()` все обращения. Опечатка `task.get("titel")` тихо вернёт `None`.

## 05. Добавление, изменение, удаление и перебор

Присваивание по ключу добавляет новое поле или заменяет существующее. `pop()` удаляет ключ и возвращает старое значение.

| Код | Назначение |
| --- | --- |
| task["category"] = "python" | добавить новое поле |
| task["priority"] = 5 | заменить существующее значение |
| task.pop("category") | удалить ключ и вернуть старое значение |
| task.pop("category", None) | не падать, если ключ отсутствует |
| task.items() | получить пары ключ–значение для цикла |

**изменение и перебор**

```python
task = {"title": "SQL", "priority": 3}
task["is_done"] = False
task["priority"] = 4

for key, value in task.items():
    print(key, value)
```

### Сопоставление

Соедините способ перебора с данными на каждой итерации.

| Запись | Результат шага |
| --- | --- |
| for key in task | ключ |
| for value in task.values() | значение |
| for key, value in task.items() | пара ключ и значение |

В бизнес-логике чаще читают конкретные ключи, а универсальный перебор полезен для диагностики.

## 06. Одна модель во всём проекте

В StudyHub одна задача имеет фиксированную форму. Если части проекта используют разные ключи, ошибка появляется далеко от места создания данных.

**договор модели**

```python
task = {
    "id": 1,
    "title": "Изучить словари",
    "priority": 3,
    "is_done": False,
}
```

### Стабильные поля — обязательно

Их отсутствие означает нарушение модели.

```python
task["id"]
task["title"]
task["priority"]
task["is_done"]
```

### Расширяемое поле — необязательно

Для отсутствия заранее определено понятное поведение.

```python
task.get("deadline")
task.get("category", "other")
```

### Разбор ошибки

```python
created = {"name": "SQL"}
print(created["title"])
```

**Вопрос:** Где находится первичная проблема?

A. В print
B. В месте создания словаря с ключом name
C. В типе строки SQL

**Ответ:** B. В месте создания словаря с ключом name

Потребитель ожидает title. Нужно согласовать форму данных, а не случайно подставлять значение.

**Исправленный вариант**

```python
created = {"title": "SQL"}
print(created["title"])
```

> На следующем занятии создание словаря будет сосредоточено в функции `create_task(...)`.

## 07. Список словарей и копирование

Одна задача — один словарь. Список хранит множество таких записей и позволяет обрабатывать их циклом.

**модель проекта**

```python
tasks = [
    {"id": 1, "title": "Python", "is_done": True},
    {"id": 2, "title": "SQL", "is_done": False},
]

for task in tasks:
    if task.get("is_done", False):
        print(task["title"])
```

### Сравнение решений

**Вопрос:** Как создать обновлённую задачу, сохранив исходную?

### Второе имя

```python
updated = task
updated["is_done"] = True
```

Меняется тот же словарь.

### Копия

```python
updated = task.copy()
updated["is_done"] = True
```

Верхний словарь original остаётся прежним.

**Вывод:** предпочтителен правый вариант. Словарь изменяем, поэтому простое присваивание не создаёт независимую запись.

### Вспомните модель

**Вопрос:** Чем одна задача отличается от списка задач?

**Ответ:** Одна задача — словарь с именованными полями. Все задачи — список, каждый элемент которого является словарём одной записи.

> В практике функция не должна менять переданный словарь или список. Автопроверка может отдельно сравнить исходные данные после вызова.

## 08. Проверь себя

### Вопрос 1

Что произойдёт при отсутствующем task['title']?

A. вернётся None
B. возникнет KeyError
C. добавится пустая строка

**Ответ:** B. возникнет KeyError

Квадратные скобки требуют существующий ключ.

### Вопрос 2

Что делает task.get('priority', 1)?

A. возвращает значение или 1
B. всегда добавляет ключ
C. удаляет ключ

**Ответ:** A. возвращает значение или 1

get читает значение и не изменяет словарь.

### Вопрос 3

Что проверяет выражение 'title' in task?

A. наличие ключа
B. наличие значения
C. тип словаря

**Ответ:** A. наличие ключа

Оператор in для словаря проверяет ключи.

### Вопрос 4

Что делает task.items()?

A. возвращает пары ключ–значение
B. копирует словарь
C. оставляет только значения

**Ответ:** A. возвращает пары ключ–значение

items() используется для одновременного перебора ключа и значения.

### Вопрос 5

Почему updated = task не создаёт независимую задачу?

A. словарь неизменяем
B. оба имени указывают на один объект
C. Python запрещает копирование

**Ответ:** B. оба имени указывают на один объект

Для отдельного верхнего словаря используется copy().

### Главное после занятия

- Словарь хранит пары ключ–значение.
- Ключи одного словаря уникальны.
- `[]` выражает обязательность поля.
- `get()` подходит для необязательных полей и не добавляет ключ.
- Присваивание по ключу добавляет или заменяет поле.
- Одна задача — словарь, все задачи — список словарей.
- `copy()` создаёт отдельный верхний словарь.
