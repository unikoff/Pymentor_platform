# Проект: добавление и вывод задач

Добавим первые настоящие данные StudyHub: создадим карточку задачи, проверим её поля, сохраним в список и выведем в едином формате.

**Темы:** модель и состояние · создание и вывод

## 01. Путь новой задачи

Добавление задачи похоже на оформление новой карточки в каталоге. Сначала проверяют название и приоритет, затем создают карточку единой формы, присваивают ей номер и только после этого кладут в общую папку.

1. **Проверить данные:** пустое название и неверный приоритет не попадают в состояние.
2. **Создать модель:** функция `create_task()` возвращает словарь единой формы.
3. **Добавить и показать:** список хранит задачи, а `format_task()` отвечает за одну строку вывода.

Дополнительно вы разберёте правило id, изменение списка, возврат новой копии и поведение пустого приложения.

> Запомните маршрут карточки: **проверить → создать → добавить → показать**. Если проверка выполняется после `append()`, неверная карточка уже попала в общую папку.

## 02. Карточка задачи и папка со всеми задачами

Один словарь — это карточка одной задачи с подписанными полями. Список `tasks` похож на папку, в которой такие карточки лежат по порядку создания. Пока программа работает, папка хранится в памяти.

### Одна задача — dict

Поля одной связанной записи.

```python
task = {
    "id": 1,
    "title": "SQL",
    "priority": 3,
    "is_done": False,
}
```

### Все задачи — list

Упорядоченное состояние приложения.

```python
tasks = [task]
```

### Ограничение — memory

После завершения процесса данные исчезнут. JSON будет изучаться позже.

```python
tasks = []
```

### Сравнение решений

**Вопрос:** Как правильно получить название?

### Из одной задачи

```python
print(task["title"])
```

task — словарь.

### Из списка задач

```python
print(tasks[0]["title"])
```

Сначала выбирается элемент списка, затем ключ словаря.

**Вывод:** оба варианта допустимы. Обе записи корректны, но работают с разным уровнем структуры данных.

> Не смешивайте `task["title"]` и `tasks["title"]`: список не имеет строкового ключа `"title"`.

## 03. Одна форма карточки

Все карточки должны заполняться по одному шаблону. Если на одной написано `title`, а на другой `name`, функции поиска и вывода перестают понимать данные одинаково. Поэтому форму создаёт одна функция `create_task()`.

**единая фабрика задачи**

```python
def create_task(task_id, title, priority):
    return {
        "id": task_id,
        "title": title.strip(),
        "priority": priority,
        "is_done": False,
    }
```

```python
title = "  Изучить SQL  "
priority = 3
is_title_valid = title.strip() != ""
is_priority_valid = 1 <= priority <= 5
task = create_task(1, title, priority)
```

**Пошаговый разбор**

1. Исходное название содержит пробелы по краям. **Состояние:** title = " Изучить SQL "
2. После strip строка не пустая. **Состояние:** is_title_valid = True
3. Приоритет входит в диапазон 1–5. **Состояние:** is_priority_valid = True
4. Функция создаёт словарь с очищенным названием и is_done=False. **Состояние:** task['title'] = "Изучить SQL"; task['is_done'] = False

### Сопоставление

Соедините поле модели с его назначением.

| Левая часть | Правая часть |
| --- | --- |
| id | однозначный идентификатор |
| title | название задачи |
| priority | уровень от 1 до 5 |
| is_done | логический статус завершения |

Стабильные имена ключей позволяют функциям понимать одну модель одинаково.

### Разбор ошибки

```python
task = {"name": "SQL", "priority": 3}
print(task["title"])
```

**Вопрос:** Почему вывод завершится KeyError?

A. В словаре создан ключ name, а читается title
B. Словари нельзя печатать
C. priority должен быть строкой

**Ответ:** A. В словаре создан ключ name, а читается title

Функции используют разные имена одного поля.

**Исправленный вариант**

```python
task = {"title": "SQL", "priority": 3}
print(task["title"])
```

## 04. Как выдавать следующий номер задачи

Идентификатор похож на номер талона. Даже если карточку удалили, номер другой существующей карточки не должен повториться. Поэтому количество элементов не всегда подходит для вычисления следующего id.

### Сравнение решений

**Вопрос:** Какое правило устойчивее после удаления последней задачи?

### Количество плюс один

```python
def get_next_id(tasks):
    return len(tasks) + 1
```

Может повторить удалённый максимальный id.

### Максимум плюс один

```python
def get_next_id(tasks):
    if not tasks:
        return 1
    max_id = tasks[0]['id']
    for task in tasks:
        if task['id'] > max_id:
            max_id = task['id']
    return max_id + 1
```

Не зависит от длины и порядка списка.

**Вывод:** предпочтителен правый вариант. Количество элементов не гарантирует, что такой id ещё не использовался.

### Предскажите результат

```python
tasks = [{"id": 2}, {"id": 7}, {"id": 4}]
max_id = tasks[0]["id"]
for task in tasks:
    if task["id"] > max_id:
        max_id = task["id"]
print(max_id + 1)
```

*Подсказка: Цикл хранит максимальное увиденное значение.*

**Ответ**

```text
8
```

> Вариант `tasks[-1]["id"] + 1` допустим только при гарантии, что последний элемент всегда имеет максимальный id.

## 05. Изменить список или вернуть новую версию

Есть два честных способа работы. Локальный обработчик может изменить общую папку задач через `append()`. Чистая платформенная функция может сначала скопировать внешний список и вернуть новую версию. Важно заранее назвать выбранное поведение.

### Сравнение решений

**Вопрос:** Чем отличаются два контракта?

### Изменить состояние

```python
def add_task(tasks, task):
    tasks.append(task)
```

Исходный список становится длиннее.

### Вернуть новую версию

```python
def with_added_task(tasks, task):
    result = tasks.copy()
    result.append(task)
    return result
```

Исходный внешний список остаётся прежним.

**Вывод:** оба варианта допустимы. Оба контракта допустимы, если название и документация честно сообщают о поведении.

```python
original = [{"title": "Git"}]
result = original.copy()
result.append({"title": "SQL"})
print(original)
print(result)
```

**Пошаговый разбор**

1. Создан исходный список с одной задачей. **Состояние:** original = [{"title": "Git"}]
2. copy создаёт новый внешний список. **Состояние:** result = [{"title": "Git"}]
3. Новая задача добавляется только в result. **Состояние:** len(original) = 1; len(result) = 2
4. Исходный список не изменился. **Состояние:** original = [{"title": "Git"}]
5. Новый список содержит две записи. **Состояние:** result = [{"title": "Git"}, {"title": "SQL"}]

### Верно или неверно

`tasks.copy()` создаёт глубокие независимые копии всех словарей внутри списка.

**Ответ:** неверно. Копируется только внешний список; вложенные словари остаются теми же объектами.

## 06. Один формат вывода для всех задач

Представьте, что каждая карточка печатается по одному шаблону. Если значок статуса или порядок полей изменится, лучше исправить один шаблон `format_task()`, а не несколько разных участков проекта.

**форматирование одной задачи**

```python
def format_task(task):
    mark = "x" if task["is_done"] else " "
    return (
        f"[{mark}] {task['id']}. {task['title']} "
        f"(приоритет {task['priority']})"
    )
```

**вывод всего списка**

```text
def show_tasks(tasks):
    if not tasks:
        print("Задач пока нет")
        return

    for task in tasks:
        print(format_task(task))
```

### Предскажите результат

```python
tasks = [
    {"id": 1, "title": "SQL", "priority": 3, "is_done": False},
    {"id": 2, "title": "Git", "priority": 2, "is_done": True},
]
for task in tasks:
    mark = "x" if task["is_done"] else " "
    print(f"[{mark}] {task['id']}. {task['title']}")
```

*Подсказка: Знак зависит только от is_done.*

**Ответ**

```text
[ ] 1. SQL
[x] 2. Git
```

> Пустой список — нормальное состояние нового приложения. Для него нужен понятный текст, а не обращение к `tasks[0]`.

## 07. Подключаем добавление к меню

Обработчик меню похож на сотрудника у стойки: он спрашивает данные, вызывает готовые правила и показывает результат. Он не должен заново придумывать форму карточки, правило id или формат вывода.

### Соберите сценарий добавления

Расположите действия в безопасном порядке.

**Фрагменты**

- `title = input("Название: ")`
- `if not is_valid_title(title): return`
- `task_id = get_next_id(tasks)`
- `task = create_task(task_id, title, priority)`
- `add_task(tasks, task)`

**Правильный порядок**

```python
title = input("Название: ")
if not is_valid_title(title):
    return
task_id = get_next_id(tasks)
task = create_task(task_id, title, priority)
add_task(tasks, task)
```

Некорректные данные отсеиваются до создания и изменения списка.

### Разбор ошибки

```python
def handle_add(tasks):
    title = input("Название: ")
    task = create_task(1, title, 3)
    tasks.append(task)
    if title.strip() == "":
        print("Пустое название")
```

**Вопрос:** Какая проблема уже произошла до сообщения об ошибке?

A. Некорректная задача уже добавлена
B. Список автоматически очистился
C. create_task вернула None

**Ответ:** A. Некорректная задача уже добавлена

Проверка выполняется после изменения состояния.

**Исправленный вариант**

```python
def handle_add(tasks):
    title = input("Название: ")
    if title.strip() == "":
        print("Пустое название")
        return
    task = create_task(1, title, 3)
    tasks.append(task)
```

> Команда `2` может вызывать короткий обработчик `handle_list(tasks)`, который делегирует работу функции `show_tasks()`.

## 08. Проверьте основную модель

### Вопрос 1

Почему форма словаря должна быть стабильной?

A. функции ожидают одинаковые ключи
B. иначе list не работает
C. так требует Git

**Ответ:** A. функции ожидают одинаковые ключи

Создание, поиск и вывод должны одинаково понимать модель.

### Вопрос 2

Почему новая задача получает is_done=False?

A. это правило проекта
B. bool всегда False
C. иначе id не создастся

**Ответ:** A. это правило проекта

Новая задача ещё не завершена.

### Вопрос 3

Зачем в платформенной задаче использовать copy?

A. не менять исходный список
B. иначе нельзя добавить словарь
C. для преобразования типов

**Ответ:** A. не менять исходный список

Возвращается новая версия состояния.

### Вопрос 4

Зачем выделять format_task?

A. единое правило отображения
B. ускорить input
C. заменить словарь

**Ответ:** A. единое правило отображения

Формат меняется в одном месте.

### Главное после занятия

- Одна задача — словарь, все задачи — список словарей.
- Данные проверяются до добавления.
- Форма модели создаётся одной функцией.
- Правило id должно учитывать удаление.
- `append` изменяет список, а `copy` помогает вернуть новый.
- Пустой список обрабатывается явно.
