# Декомпозиция и чистые функции

Разделим широкую задачу на именованные шаги, зафиксируем вход и выход, отличим вычисление от побочного эффекта и научимся возвращать обновлённые копии данных.

**Темы:** декомпозиция и контракт · чистые функции

## 01. Маршрут занятия

Большая функция часто скрывает несколько разных правил. Декомпозиция делает их видимыми, но цель не в количестве функций, а в ясной ответственности каждой части.

1. **Разложить сценарий:** превратить «добавить задачу» в очистку, проверку, создание и добавление.
2. **Зафиксировать контракт:** назвать вход, результат и возможное изменение данных.
3. **Сделать поведение явным:** отделить чистые расчёты от `input`, `print` и изменения внешнего состояния.

Дополнительно вы разберёте глобальные зависимости, копирование словарей, композицию функций и границу между полезной и чрезмерной декомпозицией.

> Хороший основной сценарий читается как последовательность предметных действий, а не как набор технических деталей.

## 02. Широкую задачу разбивают на шаги

Фраза «добавить задачу» скрывает несколько независимых действий: получить данные, очистить название, проверить правила, создать словарь и изменить список.

### Сравнение решений

**Вопрос:** Какой вариант легче проверять и изменять?

### Всё в одной функции

```python
def add_task():
    title = input("Название: ").strip()
    priority = int(input("Приоритет: "))
    if title and 1 <= priority <= 5:
        tasks.append({"title": title, "priority": priority})
        print("Добавлено")
```

Ввод, проверка, создание данных, изменение списка и печать связаны вместе.

### Правила получили имена

```python
def normalize_title(title):
    return title.strip()

def is_valid_priority(priority):
    return 1 <= priority <= 5
```

Каждое правило можно вызвать и проверить отдельно.

**Вывод:** предпочтителен правый вариант. Декомпозиция уменьшает число причин, которые приходится анализировать одновременно.

### Разложите добавление задачи

Расположите действия от получения данных до изменения списка.

**Фрагменты**

- `получить title и priority`
- `очистить title`
- `проверить title и priority`
- `создать словарь task`
- `добавить task в tasks`
- `сообщить результат пользователю`

**Правильный порядок**

```python
получить title и priority
очистить title
проверить title и priority
создать словарь task
добавить task в tasks
сообщить результат пользователю
```

Проверка выполняется до изменения состояния приложения.

### Вспомните модель

**Вопрос:** Почему валидация должна идти до append?

**Ответ:** Некорректная запись не должна попадать в состояние приложения. Сначала правило доказывает допустимость данных, затем список изменяется.

## 03. Контракт делает функцию предсказуемой

Контракт отвечает на четыре вопроса: что функция получает, что возвращает, меняет ли переданные объекты и какие граничные случаи учитывает.

### Преобразование — normalize

Вход: строка. Выход: новая строка. Исходная строка не изменяется.

```python
def normalize_title(title):
    return title.strip()
```

### Проверка — validate

Вход: число. Выход: `bool`. Границы 1 и 5 включены.

```python
def is_valid_priority(priority):
    return 1 <= priority <= 5
```

### Создание модели — create

Входы явны, результатом является новый словарь.

```python
def create_task(title, priority):
    return {"title": title.strip(), "priority": priority, "is_done": False}
```

### Сопоставление

Соедините вопрос контракта с примером ответа.

| Вопрос | Ответ |
| --- | --- |
| что принимает? | title: str, priority: int |
| что возвращает? | словарь новой задачи |
| что меняет? | ничего вне функции |
| какая граница? | priority от 1 до 5 включительно |

Контракт позволяет понять функцию до чтения тела.

> Два подхода к добавлению допустимы: изменить переданный список или вернуть новый. Ошибка — не сообщить, какой контракт выбран.

## 04. Чистая функция зависит только от аргументов

Для одинаковых аргументов чистая функция возвращает одинаковый результат и не меняет внешний мир.

**чистое правило**

```python
def is_urgent(priority, is_done):
    return priority >= 4 and not is_done
```

### Верно или неверно

Функция `is_urgent(5, False)` может вернуть разные результаты в двух последовательных вызовах, даже если внешнее состояние не менялось.

**Ответ:** неверно. Результат определяется только одинаковыми аргументами.

### Сравнение решений

**Вопрос:** Где зависимость видна явно?

### Глобальное состояние

```python
tasks = []

def count_tasks():
    return len(tasks)
```

По параметрам не видно, откуда берутся данные.

### Аргумент функции

```python
def count_tasks(tasks):
    return len(tasks)
```

Источник данных указан в сигнатуре.

**Вывод:** предпочтителен правый вариант. Явная зависимость упрощает повторное использование и тестирование.

### Вспомните модель

**Вопрос:** Почему чистую функцию легко тестировать?

**Ответ:** Для проверки достаточно передать аргументы и сравнить возвращённое значение. Не нужно заранее готовить глобальное состояние, ввод или терминал.

## 05. Побочные эффекты должны быть видимыми

Побочный эффект — изменение внешнего состояния: печать, ввод, изменение переданного списка, запись файла или сетевой запрос. Эффекты нужны приложению, но не должны быть спрятаны внутри расчёта.

| Код | Назначение |
| --- | --- |
| return a + b | чистый результат |
| print(message) | вывод в терминал |
| input("Название: ") | чтение внешнего ввода |
| tasks.append(task) | изменение переданного объекта |
| global_tasks.append(task) | скрытое изменение глобального состояния |

### Разбор ошибки

```python
tasks = []

def add_default_task():
    tasks.append("Python")
```

**Вопрос:** Что скрывает сигнатура функции?

A. Функция зависит от глобального списка и меняет его
B. Строку Python нельзя добавить
C. Список должен быть кортежем

**Ответ:** A. Функция зависит от глобального списка и меняет его

По пустым скобкам вызова невозможно увидеть зависимость от tasks.

**Исправленный вариант**

```python
def with_default_task(tasks):
    result = tasks.copy()
    result.append("Python")
    return result
```

### Сравнение решений

**Вопрос:** Как отделить интерфейс от правила?

### Ввод внутри проверки

```python
def is_valid_title():
    title = input("Название: ")
    return title.strip() != ""
```

Функцию трудно вызвать с подготовленным тестовым значением.

### Значение передано аргументом

```python
def is_valid_title(title):
    return title.strip() != ""
```

Источник данных может быть input, тест или другой код.

**Вывод:** предпочтителен правый вариант. Интерфейс получает данные, а правило работает с уже переданным значением.

## 06. Изменяемые аргументы и обновлённая копия

Список и словарь могут измениться внутри функции. Если контракт обещает новую версию, сначала создайте копию верхнего уровня.

```python
original = {"title": "SQL", "is_done": False}
updated = original.copy()
updated["is_done"] = True
print(original)
print(updated)
```

**Пошаговый разбор**

1. Создан исходный словарь. **Состояние:** original = {title: SQL, is_done: False}
2. copy создаёт отдельный верхний словарь с теми же полями. **Состояние:** original = объект A; updated = объект B
3. Меняется только updated. **Состояние:** original = is_done: False; updated = is_done: True
4. Исходные данные сохранены. **Состояние:** вывод = {...False}
5. Новая версия содержит обновление. **Состояние:** вывод = {...False} ⏎ {...True}

### Сравнение решений

**Вопрос:** Как сохранить дополнительные ключи?

### Собрать заново

```python
return {"title": task["title"], "is_done": True}
```

Поля priority и category могут потеряться.

### Скопировать и обновить

```python
result = task.copy()
result["is_done"] = True
return result
```

Все существующие верхние поля сохраняются.

**Вывод:** предпочтителен правый вариант. Копирование уменьшает риск случайно удалить неизвестные вызывающей функции поля.

### Разбор ошибки

```python
def marked_copy(task):
    task["is_done"] = True
    return task
```

**Вопрос:** Почему название marked_copy вводит в заблуждение?

A. Функция меняет исходный словарь вместо создания копии
B. Ключ is_done нельзя менять
C. return нельзя использовать со словарём

**Ответ:** A. Функция меняет исходный словарь вместо создания копии

Имя обещает копию, но тело изменяет аргумент.

**Исправленный вариант**

```python
def marked_copy(task):
    result = task.copy()
    result["is_done"] = True
    return result
```

## 07. Композиция без чрезмерной дробности

Маленькие функции соединяются в сценарий. Управляющая функция может вызывать правила, но не должна дублировать их реализацию.

**StudyHub: подготовка задачи**

```python
def normalize_title(title):
    return title.strip()

def is_valid_title(title):
    return normalize_title(title) != ""

def create_task(title, priority):
    return {
        "title": normalize_title(title),
        "priority": priority,
        "is_done": False,
    }

def prepare_task(title, priority):
    if not is_valid_title(title):
        return None
    if not 1 <= priority <= 5:
        return None
    return create_task(title, priority)
```

### Соберите чистое обновление задачи

Выберите порядок, который не меняет исходный словарь.

**Фрагменты**

- `def marked_copy(task):`
- `result = task.copy()`
- `result["is_done"] = True`
- `return result`
- `task["is_done"] = True` — меняет аргумент

**Правильный порядок**

```python
def marked_copy(task):
    result = task.copy()
    result["is_done"] = True
    return result
```

Копия создаётся до изменения, затем возвращается как новый результат.

### Вспомните модель

**Вопрос:** Когда не нужно выделять новую функцию?

**Ответ:** Когда действие очевидно, не повторяется, не имеет самостоятельного предметного смысла и отдельное имя заставит читателя переходить по коду без пользы.

> Декомпозиция не означает «одна строка — одна функция». Критерий — улучшение понимания и проверяемости.

## 08. Проверь себя

### Вопрос 1

Что означает декомпозиция?

A. разбиение задачи на понятные шаги
B. удаление половины кода
C. замена функций классами

**Ответ:** A. разбиение задачи на понятные шаги

Каждый шаг получает собственную ответственность.

### Вопрос 2

Какой признак чистой функции?

A. зависит только от аргументов
B. всегда печатает
C. обязательно состоит из одной строки

**Ответ:** A. зависит только от аргументов

Для одинаковых аргументов результат одинаков, внешнее состояние не меняется.

### Вопрос 3

Что является побочным эффектом?

A. return a + b
B. изменение внешнего списка
C. сравнение priority >= 4

**Ответ:** B. изменение внешнего списка

Изменяется состояние за пределами возвращаемого значения.

### Вопрос 4

Зачем использовать task.copy()?

A. обновить отдельный верхний словарь
B. сделать словарь кортежем
C. удалить лишние поля

**Ответ:** A. обновить отдельный верхний словарь

Исходный словарь остаётся прежним.

### Вопрос 5

Нужно ли создавать функцию для каждой строки?

A. нет, только для осмысленного правила
B. да, всегда
C. только если строка длинная

**Ответ:** A. нет, только для осмысленного правила

Чрезмерная дробность ухудшает связность чтения.

### Главное после занятия

- Декомпозиция превращает широкое действие в именованные шаги.
- Контракт описывает вход, выход, изменение и границы.
- Чистая функция зависит только от аргументов.
- Побочные эффекты допустимы, но должны быть видимыми.
- Глобальная зависимость скрывает источник данных.
- Изменяемый аргумент может измениться снаружи.
- `copy()` помогает вернуть обновлённую версию.
- Полезная декомпозиция улучшает чтение, а не просто увеличивает число функций.
